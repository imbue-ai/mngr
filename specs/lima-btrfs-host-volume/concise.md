# Lima btrfs host-data volume

## Overview

- Today's Lima provider backs `host_dir` (canonically `/mngr`) with a 9p bind mount from `~/.mngr/providers/lima/<name>/volumes/<host_id>/` on the host machine; the host filesystem is generally not btrfs, so the data cannot be snapshotted as a single consistent unit.
- This change adds an alternative layout: an in-VM btrfs filesystem on a Lima-managed `additionalDisks` qcow2 attached to the VM (auto-mounted by Lima at `/mnt/lima-<disk_name>`), with `host_dir` symlinked directly to that path. The earlier draft of this spec called for an intermediate `/mnt/host-volume` bind-mount for naming symmetry with the other providers' `host_volume_mount_path` pattern; sandbox testing showed that the fstab-generated mount unit fired before Lima's additional-disk mount on reboot, stacking an empty-ext4 layer under the btrfs bind. Symlinking straight to Lima's path removes the whole class of mount-ordering problems for the cost of a slightly less uniform name across providers.
- Behavior is opt-in via a new `is_host_data_volume_exposed: bool = True` field on `LimaProviderConfig`. `True` (default) keeps today's host bind mount. `False` switches to the in-VM btrfs disk and omits the bind mount.
- The chosen value is persisted on the per-host `LimaHostConfig` record at create time so subsequent `start_host`/`stop_host`/`get_host` calls never change a host's layout, even if the provider config is later flipped. Records that predate this field deserialize with the default `True`, preserving today's behavior for all existing Lima hosts forever.
- The FCT lima template (`~/project/forever-claude-template/.mngr/settings.toml [create_templates.lima]`) flips the provider default to `False` via `setting__extend = ["providers.lima.is_host_data_volume_exposed=false"]`, so every minds-created Lima host gets the snapshottable layout. Out-of-template `mngr create @.lima` invocations keep today's behavior unless the user sets the flag themselves.
- This is the Lima-provider half of a multi-provider effort to consolidate all provider data onto a snapshottable volume; snapshot/backup wiring itself is out of scope for this PR.

## Expected Behavior

- New provider config field `is_host_data_volume_exposed: bool = True` on `LimaProviderConfig`.
- New persisted field `is_host_data_volume_exposed: bool = True` on `LimaHostConfig` inside the per-host `HostRecord`. The pydantic default of `True` is what existing on-disk records get when deserialized (today's behavior for pre-change hosts).
- When `is_host_data_volume_exposed=True` (default), `mngr create @.lima` produces a VM identical to today: a 9p bind mount of the host-side volume dir at `host_dir`, no additional disks, and `LimaProviderInstance.get_volume_for_host()` returns a `HostVolume` over the same host-side directory.
- When `is_host_data_volume_exposed=False`, `mngr create @.lima` produces a VM with:
  - A Lima `additionalDisks` entry named `mngr-<host_id_hex>-data`, `format: true`, `fsType: btrfs`, default logical size `100GiB` (qcow2 is sparse, so this is a logical cap, not upfront disk usage). Lima creates and formats the qcow2 on first boot under `~/.lima/_disks/`.
  - The btrfs disk auto-mounted inside the VM by Lima at `/mnt/lima-<disk_name>` using Lima's default mount options.
  - `host_dir` (e.g. `/mngr`) set up as a symlink to `/mnt/lima-<disk_name>`, installed by an additional shell block appended to Lima's existing `_build_provisioning_script` (the cloud-init-style `mode: system` script in `lima_yaml.py`), not via the Docker/Modal `build_check_and_install_packages_command` path.
  - No `mounts:` block in the Lima YAML — the host-side bind mount is omitted entirely.
- When `is_host_data_volume_exposed=False`, `LimaProviderInstance.get_volume_for_host()` returns `None`. Existing callers in `libs/mngr/imbue/mngr/api/events.py`, `libs/mngr_claude/imbue/mngr_claude/plugin.py:on_before_host_destroy`, `libs/mngr_tmr/imbue/mngr_tmr/pulling.py`, and `libs/mngr_file/imbue/mngr_file/cli/target.py` already handle `None` gracefully and degrade by either skipping or falling back to online-host SSH access.
  - User-visible consequence: `mngr event` / `mngr transcript` against a *stopped* btrfs-mode Lima host stops working until the host is started. While the host is running, the online-host fallback in `events.py` keeps working.
  - User-visible consequence: when a btrfs-mode Lima host is destroyed without the agent's own `on_destroy()` running, `mngr_claude`'s `on_before_host_destroy` hook will skip session preservation rather than read from the host volume.
- If `mkfs.btrfs` or the post-format mount fails during VM boot, `create_host` fails hard (raises `LimaHostCreationError`) and the host record is saved with `failure_reason` set. No silent fallback to bind-mount mode.
- `destroy_host` removes the named Lima disk via `limactl disk delete --force mngr-<host_id_hex>-data` when the persisted record says `is_host_data_volume_exposed=False`, so the qcow2 under `~/.lima/_disks/` is cleaned up alongside the VM.
- `stop_host` / `start_host` always replay the layout captured in the persisted record. A host created with `is_host_data_volume_exposed=False` keeps the btrfs disk attached on every restart; flipping the provider config later does not migrate it. Same in the other direction.
- `discover_hosts`, `list_volumes`, `get_host_resources`, snapshot rejection, tag management, and the SSH wiring are all unchanged.
- FCT lima template wired to opt minds in: minds Lima workspaces created by `apps/minds/imbue/minds/desktop_client/agent_creator.py:_build_mngr_create_command` (LIMA mode) automatically get the btrfs layout.

## Changes

- Add `is_host_data_volume_exposed: bool = True` to `LimaProviderConfig` (`libs/mngr_lima/imbue/mngr_lima/config.py`) with help text describing the two layouts, the trade-off around offline volume access, and the FCT-template recommendation.
- Add `is_host_data_volume_exposed: bool = True` to `LimaHostConfig` (`libs/mngr_lima/imbue/mngr_lima/host_store.py`) so the create-time choice is persisted on the per-host record. The pydantic default is what makes pre-change records keep today's behavior on deserialize.
- Add a constant for the default btrfs disk size and a helper `lima_host_data_disk_mount_path(disk_name)` returning `/mnt/lima-<disk_name>` (the path Lima auto-mounts named additional disks at) in `libs/mngr_lima/imbue/mngr_lima/constants.py`. Add a second helper for building the Lima disk name from a `HostId` (`mngr-<host_id_hex>-data`).
- Extend `generate_default_lima_yaml` (`libs/mngr_lima/imbue/mngr_lima/lima_yaml.py`) to accept the new flag plus a disk-name string. When the flag is `False`:
  - Omit the `mounts:` block entirely.
  - Emit an `additionalDisks` block referencing the named disk with `format: true` and `fsType: btrfs`.
  - Append a shell block to `_build_provisioning_script` that (a) waits for Lima's auto-mount of the additional disk under `/mnt/lima-<disk_name>`, (b) `chmod 0777`s the btrfs root so the Lima default non-root user can write `host_dir` without sudo (fresh `mkfs.btrfs` leaves the root dir 0755 root-owned -- discovered via the lima-playground sandbox before merge), and (c) replaces `host_dir` with a symlink to `/mnt/lima-<disk_name>` directly (idempotent: remove any existing non-symlink dir first, then `ln -sfn`). No intermediate bind-mount, no `/etc/fstab` manipulation -- the bind approach the first draft used stacked an empty-ext4 layer when the fstab unit fired before Lima's mount on reboot.
- In `LimaProviderInstance.create_host` (`libs/mngr_lima/imbue/mngr_lima/instance.py`):
  - Resolve the effective layout from `self.config.is_host_data_volume_exposed`.
  - Pass the flag and the derived disk name into `generate_default_lima_yaml` (and into `merge_lima_yaml` for the `--file` path so user YAML cannot drop the new `additionalDisks` / provisioning bits when btrfs mode is on).
  - When the flag is `False`, skip `_ensure_host_volume_dir` (no host-side directory is needed) and do not pass a `volume_host_path` for the YAML's mounts list.
  - Persist the effective value into the new `LimaHostConfig.is_host_data_volume_exposed` field on the saved `HostRecord`.
  - In btrfs mode, call a new `limactl_disk_create(disk_name, size)` helper *before* `limactl_start_new`. Lima's `additionalDisks` with `format: true` only auto-formats an already-existing disk record under `~/.lima/_disks/<name>/`; without this pre-create step `limactl start` fails with "could not load disk ... no such file or directory" (discovered via the lima-playground sandbox before merge). The error-recovery path also calls `limactl_disk_delete` so a failed create_host that already created the disk doesn't leak it.
- In `LimaProviderInstance.destroy_host`, when the record's `is_host_data_volume_exposed` is `False`, additionally call a new `limactl_disk_delete(disk_name, force=True)` helper in `libs/mngr_lima/imbue/mngr_lima/limactl.py` to remove the Lima-managed qcow2 under `~/.lima/_disks/`. Tolerate the disk already being absent.
- In `LimaProviderInstance.delete_host`, skip the host-side `volume_dir` `shutil.rmtree` when the record's `is_host_data_volume_exposed` is `False` (there is no host-side volume dir to remove for btrfs-mode hosts).
- In `LimaProviderInstance.get_volume_for_host`, return `None` when the persisted record's `is_host_data_volume_exposed` is `False`. Today's behavior (return a `HostVolume` over the bind-mount dir) is preserved for `True`.
- In `LimaProviderInstance.list_volumes`, only emit `VolumeInfo` entries for hosts whose persisted record is in `True` mode (btrfs-mode hosts have no host-side volume to list).
- Add the FCT-template wiring: `~/project/forever-claude-template/.mngr/settings.toml [create_templates.lima]` gets `setting__extend = ["providers.lima.is_host_data_volume_exposed=false"]`. This is the only change outside the mngr repo that this spec touches; commit on FCT's mirror branch.
- Add the per-PR changelog entry at `libs/mngr_lima/changelog/mngr-lima-btrfs-host-dir.md` (one project touched).
- Tests:
  - Unit tests in `libs/mngr_lima/imbue/mngr_lima/lima_yaml_test.py` covering: YAML for `True` matches today (mounts present, no additionalDisks, no provisioning symlink block); YAML for `False` omits `mounts:`, includes the `additionalDisks` entry with the derived disk name and `format: true`/`fsType: btrfs`, and the provisioning script contains the symlink-and-mount block.
  - Unit tests in `libs/mngr_lima/imbue/mngr_lima/host_store_test.py` covering `LimaHostConfig` round-trip with the new field, and that a JSON record missing the field deserializes with `is_host_data_volume_exposed=True`.
  - One `@pytest.mark.lima` end-to-end test in `libs/mngr_lima/imbue/mngr_lima/instance_test.py` that creates a host with `is_host_data_volume_exposed=False`, writes a file to `host_dir`, calls `stop_host` then `start_host`, and asserts the file is still there (proves the btrfs disk survives a restart).
- Documentation: a short section in `libs/mngr_lima/README.md` describing the two modes, the trade-off (snapshot-consistency vs offline volume access), and the FCT-template recommendation. No new top-level doc file.
- Out of scope: any `create_snapshot` / `list_snapshots` API implementation (Lima provider's `supports_snapshots` stays `False`); migration of existing Lima hosts to the new layout; bumping `MINIMUM_LIMA_VERSION` (the (1,0,0) floor already supports `additionalDisks` with `format: true` and `fsType: btrfs`); any change to other providers; any change to `host_dir` resolution semantics; any change to the `mounts:` block on `True`-mode hosts.

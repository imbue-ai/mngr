# Remove the primary chat agent from minds

## Overview

- The "primary agent" of a minds workspace stops being a Claude that the user chats with; it becomes a non-interactive `system-services` agent whose sole job is to run the bootstrap service manager (`uv run bootstrap`) in tmux window 0.
- The user-facing chat agent (the thing they actually talk to in the dockview ChatPanel) is a regular `chat` agent named `assistant`, structurally identical to any agent created later via the workspace UI's "+" → "New Chat" button.
- The `assistant` agent is created exactly once per workspace, idempotently, by bootstrap itself on its first reconcile pass — with the same `/welcome` initial message the desktop client used to inject.
- The user's chosen workspace name applies only to the mngr *host*, never to any agent inside; this turns the imbue_cloud lease-adoption flow from "pre-baked agent → rename to user's name" into "pre-baked agent kept as `system-services`, host name set on the lease".
- This is a hard break: existing in-the-wild workspaces (with a Claude in window 0 and `is_primary=true` on that Claude) are not migrated in place; users destroy and recreate.

## Expected Behavior

- Creating a workspace from the desktop client produces one host (named whatever the user typed in the form) running one system-services agent (running bootstrap), which in turn brings up the existing services (`svc-system_interface`, `svc-web`, `svc-cloudflared`, `svc-app-watcher`, `svc-runtime-backup`) plus a freshly-created `assistant` chat agent on the same host.
- On every container restart, bootstrap re-runs its reconcile loop but skips re-creating the assistant when `runtime/.assistant-created` is present; a user who deliberately destroys the assistant does not see it auto-recreated unless they also remove the sentinel.
- The desktop client's "go to workspace" navigation lands the user at the system-services agent's URL (same as today); the workspace_server rendered inside that agent shows a ChatPanel whose default selected agent is the one named `assistant` on the same host, with a graceful fall-back to the most-recent chat agent if `assistant` is gone.
- The workspace sidebar lists only chat-style agents — the system-services agent is hidden from the user-visible list because there is nothing useful for the user to do with an empty bootstrap shell — but it remains discoverable via `mngr list` and through every other mngr-facing surface.
- The user-supplied workspace name is the mngr `HostName`, is mirrored as a `host.labels.workspace=<name>` tag, and is what's stored on the imbue_cloud connector's lease record; the form rejects invalid names by surfacing the connector's / mngr's own validation error rather than silently slugifying.
- `MINDS_WORKSPACE_NAME` inside the container continues to carry the user-chosen workspace name (now identical to `HostName`); existing skills (`crystallize-task`, `launch-task`, `heal-skill`, `update-skill`) that filter sub-agents by `--label workspace=$MINDS_WORKSPACE_NAME` keep working unchanged.
- Destroying a workspace from the desktop client behaves as it does today — the existing host-fanout in `destroying._build_destroy_command` already destroys every agent on the system-services agent's host, so the assistant and any "+ New Chat" agents go away with it.
- Cloudflare-tunnel setup, the account ↔ workspace association, the `mngr forward` agent-include filter, and the api_v1 hash-based authentication continue to be keyed by the system-services agent's `AgentId`; `is_primary=true` is repurposed to mean "the primary *system* agent" (always present, never chat) rather than "the primary chat agent".
- Telegram bot routing is intentionally left as-is — it is broken in the current world and explicitly out of scope here, so the spec does not attempt to repoint it from the now-non-Claude window-0 process.
- Running `mngr create --template system_services --template <mode>` from outside minds produces the same shape of agent that minds itself produces, including the `is_primary=true` / `user_created=true` labels (now declared inside the template rather than injected inline).

## Changes

- Add the connector-side `host_name` field to the `/hosts/lease` request body and persist it on the lease row; add a one-shot connector migration that backfills random `host_name` values onto every pre-existing lease so the column is fully populated before the new code path goes live.
- Implement `ImbueCloudProvider.rename_host` against the same connector endpoint so any subsequent UI-driven rename actions also propagate; today's `NotImplementedError` goes away.
- Replace `ImbueCloudHost.create_agent_state`'s name/command/labels patching of the pre-baked agent with a path that adopts the pre-baked agent verbatim (the bake names it `system-services` and that's the name we keep); the host's `host_name` is set via the new lease parameter, not by renaming an agent.
- Add a new `[agent_types.system_services]` block in FCT's `.mngr/settings.toml` with `parent_type = "command"` and `command = "uv run bootstrap"` so mngr's liveness / idle / state-detection treats window 0 as a generic command process rather than expecting Claude.
- Add a new `[create_templates.system_services]` block in FCT's `.mngr/settings.toml` that inherits the still-relevant fields from today's `[create_templates.main]` — `host_env`, every `extra_window` entry *except* the `bootstrap='uv run bootstrap'` one (because bootstrap is now the window-0 command), `pass_env` including `MINDS_WORKSPACE_NAME`, the reviewer env block, `extra_provision_command`, etc. — and adds `idle_mode = "disabled"` plus a `label = ["is_primary=true", "user_created=true"]` declaration so external `mngr create --template system_services` callers get the same labelling minds applies. Delete `[create_templates.main]`.
- Update `imbue/minds/cli/pool.py` to bake pool hosts with `name=system-services`, `agent_type=system_services`, and `--template system_services --template vultr`; drop the inline `--label is_primary=true` / `--label user_created=true` / `--label workspace=<bake-name>` flags now that the template carries the labels (the bake no longer cares about per-bake workspace labels — the user-chosen workspace name lands at lease time).
- Teach bootstrap to do an idempotent "create assistant if missing" step on its first reconcile pass: gate on the presence of `runtime/.assistant-created`, look up the system-services agent's `workspace=` and `project=` label values (e.g. via `mngr` against its own `MNGR_AGENT_ID`), and run `mngr create assistant --transfer none --template chat --no-connect --message "/welcome" --label workspace=<inherited> --label project=<inherited>`; on success, write the sentinel file. The step runs after `_init_runtime_worktree()` and *before* the first services reconcile so the assistant appears alongside the svc-* windows.
- Update the minds desktop client's `_build_mngr_create_command` (and the matching code paths in `agent_creator.py` / `app.py`) to:
  - pass `--template system_services --template <mode>` instead of `--template main --template <mode>`;
  - use the user's chosen workspace name directly as both the host name (i.e. `<agent>@<workspace_name>.<provider>`, no more `<name>-host` suffix) and the mngr `HostName` argument;
  - stop passing `--message "/welcome"` (now bootstrap's job);
  - stop passing `--label is_primary=true` / `--label user_created=true` inline (now carried by the template);
  - keep passing `--label workspace=<workspace_name>` so the system-services agent advertises the workspace tag for backwards compatibility with existing CEL filters.
- Pass the user-supplied workspace name onto the new `host_name` lease parameter for the IMBUE_CLOUD code path in `_build_mngr_create_command`; surface the connector's 409-on-duplicate-name response as a form-level error message via `render_create_form`.
- Pass the user-supplied workspace name *verbatim* (no client-side slugification) to mngr / the connector; let invalid names produce an error that the user retries.
- Drop `ImbueCloudHost.create_agent_state` and `provision_agent` overrides that patched the pre-baked agent's name / labels / command — the adoption now keeps the bake's identity verbatim. Keep `set_env_vars`'s merge behavior since host env still needs to be preserved across writes.
- Update the workspace_server (`apps/system_interface/imbue/minds_workspace_server/`) so its `agent_manager` keeps using `MNGR_AGENT_ID` for app-watcher / observe-cwd / work_dir bookkeeping but exposes the system-services agent through the API the same as any other agent; remove the implicit "primary == own agent" assumption baked into comments and naming (`get_applications`, `_resolve_agent_work_dir(self._own_agent_id)`, etc.) so it's correct that "+" → "New Chat" inherits labels from the system-services agent.
- Update the workspace_server frontend (`apps/system_interface/frontend/`) ChatPanel + sidebar so that:
  - the system-services agent (identified by `is_primary === "true"` in its labels, or equivalently by the new agent_type) is filtered out of the sidebar and ChatPanel agent picker;
  - the ChatPanel's default selected agent on first load is "the agent on this host named `assistant`", falling back to the most-recent chat agent if absent.
- Update the desktop client `forward_cli.py` filter (`has(agent.labels.workspace) && has(agent.labels.is_primary)`) and `backend_resolver.list_known_workspace_ids` so they continue to find exactly one workspace per host — semantics unchanged because `is_primary` is still on the system-services agent; verify the filter still does the right thing now that there are multiple non-primary agents per host.
- Delete the now-unused `[agent_types.main]` block in FCT's `.mngr/settings.toml` if nothing else depends on it (verify by searching the FCT tree); keep `[agent_types.claude]` and `[agent_types.worker]` since the chat / worker / worktree templates still reference them.
- Delete the `[create_templates.main]` block and any test fixtures that depended on it; update `apps/minds/imbue/minds/desktop_client/conftest.py` and the e2e tests so they construct discovered-agent fixtures with the new `agent_type=system_services` shape rather than the legacy `is_primary=true` Claude.
- Document the hard-break upgrade path in the changelog entry: existing minds workspaces must be destroyed and recreated after this change ships; no automated migration is provided.
- Out of scope for this spec: telegram-bot routing, in-place migration of pre-existing workspaces, any redesign of the workspace_server's "+" → "New Chat" flow or work_dir conventions, any change to where chat agents store their data.

Added `specs/claude-settings-overrides.md`: the settled design for how mngr injects Claude hooks and `settings_overrides`. mngr writes everything it owns into the per-agent config-dir `settings.json` (leak-safe) and passes no `--settings` of its own, so a user's raw `--settings` passes through and Claude layers it natively (deep-merging dicts and concatenating same-event hooks). This supersedes the interim `--settings`-merge approach. Includes a clearly-documented limitation: in `use_env_config_dir` mode, mngr can only inject hooks via `--settings`, so `settings_overrides` is unapplied and a user `--settings` collides -- an accepted, scoped limitation (that mode is not yet in production). This design is now implemented (see the `mngr_claude` and `mngr_claude_subagent_proxy` changelog entries).

Added `specs/deferred-extend-against-provision-base.md`: design for making the `settings_overrides`-onto-home merge follow mngr's config-consistent semantics (bare = assign + narrowing guard; `__extend` = merge), by generalizing the `create_templates` deferred-`__extend` mechanism so `settings_overrides` `__extend` resolves at provision against the runtime base, with provision-time narrowing and a "every deferred marker must be consumed" enforcement. Replaces the interim deep-merge-by-default. Spec only -- not yet implemented.

Added `specs/config-deep-merge-dict-fields.md` (CONCLUDED, no change needed): considered making the `__extend` operator merge dicts recursively, but concluded mngr's existing one-level `__extend` is sufficient (place `__extend` at the level you want merged) and should stay non-recursive. Kept as a record of the reasoning.

Added `specs/layered-config-merge-lib-readme.md`: a captured design + draft README for a future standalone library (`overlay`, working name) that would own the generic layered config-merge algebra (`__extend`/`__assign`/`Static*` operators, `merge`/`finalize`, recursive narrowing) as a dependency-free dict-to-dict library, with all consumer policy expressed via pre-processing rather than dependency injection. Captures the extraction plan and the mngr-specific couplings (model serialization, container deep-merge, transparent-wrapper paths). Future work, not implemented.

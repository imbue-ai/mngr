`scripts/lima_image/bake_provision.sh` now refuses to run unless `build-lima-image.sh` marks it as running inside the bake VM. The script deletes every user account and wipes `/home`, which is what the disposable bake VM wants and a catastrophe on a real machine; nothing previously stopped someone from running it directly as root on their laptop.

`scripts/build-lima-image.sh` now tells a Linux bake host what it actually needs. Lima drives `vz` on macOS but **qemu** on Linux, where it boots the guest via UEFI: that needs the arch's system emulator plus the EDK2 firmware (`ovmf` for x86_64, `qemu-efi-aarch64` for arm64) and the virtio option ROMs (`ipxe-qemu`), none of which `qemu-utils` pulls in. Without them `limactl start` fails with `could not find firmware for "x86_64"` and then `failed to find romfile "efi-virtio.rom"`. The preflight now checks for `qemu-system-<arch>` and prints the exact package list per arch, and `release.md` documents the full set. Found by running the bake on an actual Linux/KVM host, which is the path the x86_64 image is built on and which nothing had ever exercised.

When `limactl start` fails, the script now prints the tail of the instance's `ha.stderr.log` before deleting the instance. `limactl`'s own output says only `exiting, status={...}`; the reason lives in that log, and the retry's cleanup deleted it -- so a failing bake destroyed the only evidence of why it failed.

`scripts/build-lima-image.sh` now checks for `limactl` and `qemu-img` before it starts rather than after. `qemu-img` is only needed for the final flatten and is not preinstalled on either macOS or a stock Linux host (where it lives in `qemu-utils`), so a missing one used to surface as a failure at the end of the toolchain bake -- the long pole -- rather than in the first second.

It also builds its temporary Lima configs under an explicit `/tmp` template instead of `mktemp -t`. BSD and GNU `mktemp` disagree about `-t`: BSD treats the argument as a bare prefix and appends its own random suffix, so `-t foo-XXXXXX.yaml` left the `X`s literal and produced a file not even ending in `.yaml`, while GNU substituted them. The explicit form is the one both agree on, and it keeps the path short enough for Lima's ssh socket (`UNIX_PATH_MAX`). The boot probe's temp dir and instance are now torn down by the exit trap, so an interrupted build no longer leaks them.

`scripts/build-lima-image.sh` now bakes the pre-baked Lima image with a 20GiB disk by default (was 40GiB), matching the `--disk=20` the default-workspace-template lima create template passes. Lima can grow a smaller image to the instance size but fatals on shrink ("disk shrinking is not supported"), so a 40GiB baked image broke every default fast-path create that consumed it. Found by driving the fast-lima create end-to-end against a locally baked and published image.

`scripts/lima_image/bake_provision.sh` now deletes the baking host's user account before capturing the image, and fails the bake if any user with a `/home` directory survives. Lima names the guest user after the host's account and gives it that account's uid; macOS assigns its first account uid 501, so an image that kept the baker's user made cloud-init's `useradd <host-user> --uid 501` fail with "UID 501 is not unique" on every machine whose account was named differently. The guest then had no user to accept Lima's SSH key and every fast-path create hung on `Waiting for the essential requirement 1 of 3: "ssh"` until it timed out. The image booted only on the machine that baked it, which is also why it passed local testing there.

`scripts/build-lima-image.sh` now boots the finished image as a host user the image has never seen (`limaprobe`, uid 501 -- the uid macOS gives its first account) and fails the build if it does not come up. The in-VM check can only assert the image looks clean from the inside; booting it as a stranger is what actually proves it is not tied to the machine that baked it.

`apps/minds/docs/release.md` now treats publishing the pre-baked Lima image as part of the release rather than an optional coda, and adds a gate (§8c) that fetches the manifest for the released tag and fails the release if it is absent, names another version, or is missing an arch. Skipping the publish is otherwise silent: clients ask for a manifest that does not exist, get `VERSION_UNAVAILABLE`, and quietly fall back to building the workspace in-VM, so creates go from ~45s back to ~5 minutes with nothing turning red.

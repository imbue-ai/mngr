`scripts/build-lima-image.sh` now bakes the pre-baked Lima image with a 20GiB disk by default (was 40GiB), matching the `--disk=20` the default-workspace-template lima create template passes. Lima can grow a smaller image to the instance size but fatals on shrink ("disk shrinking is not supported"), so a 40GiB baked image broke every default fast-path create that consumed it. Found by driving the fast-lima create end-to-end against a locally baked and published image.

`scripts/lima_image/bake_provision.sh` now deletes the baking host's user account before capturing the image, and fails the bake if any user with a `/home` directory survives. Lima names the guest user after the host's account and gives it that account's uid; macOS assigns its first account uid 501, so an image that kept the baker's user made cloud-init's `useradd <host-user> --uid 501` fail with "UID 501 is not unique" on every machine whose account was named differently. The guest then had no user to accept Lima's SSH key and every fast-path create hung on `Waiting for the essential requirement 1 of 3: "ssh"` until it timed out. The image booted only on the machine that baked it, which is also why it passed local testing there.

`scripts/build-lima-image.sh` now boots the finished image as a host user the image has never seen (`limaprobe`, uid 501 -- the uid macOS gives its first account) and fails the build if it does not come up. The in-VM check can only assert the image looks clean from the inside; booting it as a stranger is what actually proves it is not tied to the machine that baked it.

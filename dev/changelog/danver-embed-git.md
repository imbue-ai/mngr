Added `specs/minds-managed-git/concise.md`: a design for making the git binary bundled in the minds desktop app fully managed (version-pinned, SHA256-verified, self-contained) on every shipped platform by adopting dugite-native downloads driven by a JSON manifest, replacing the unpinned copy of the build runner's Xcode CLT git on macOS and the incomplete `cp $(which git)` stub on Linux. The design ships and tests darwin-arm64 and linux-x64, carries manifest entries for darwin-x64/linux-arm64/windows-x64, specifies the required runtime environment contract (`GIT_EXEC_PATH`, `GIT_TEMPLATE_DIR`, `GIT_CONFIG_SYSTEM`, and `GIT_SSL_CAINFO` on Linux -- empirically shown to be mandatory, not optional), adds a weekly CI freshness nag keyed to dugite-native releases, records credential-helper cost tiers (deferred for now), and covers GPLv2 notice files and the test plan.

Added `.github/workflows/minds-git-freshness.yml` implementing that nag: a weekly (and manually dispatchable) job that maintains a single tracking issue (matched by stable title) for the bundled git pin. It respects the repo's 14-day dependency cooldown -- ignoring releases too new to have been vetted -- and nags only when a cooldown-cleared dugite-native release carries a newer upstream *git version* (the axis git security fixes ship on), deliberately not chasing same-git-version dugite rebuilds. A maintainer may still bump before the cooldown for an urgent CVE.

`offload-modal-acceptance.toml` now lists `apps/minds/scripts/git-manifest.json` in checkpoint `build_inputs`: the shared CI image pre-fetches the manifest-pinned dugite-native asset (offload sandboxes have no network), so a git bump must rebuild the checkpoint image for the bundled-git acceptance test.

The top-level `uncertainties.md` ledger was introduced with an entry about stale git-bundling claims in `apps/minds/docs/desktop-app.md`, and that entry was resolved and removed in the same PR once the docs were rewritten.

`scripts/snapshot_minds_e2e_state.py` now wraps the minds e2e image build's `pnpm install --frozen-lockfile` in a bounded retry (3 attempts, linear backoff). That install runs electron's postinstall, which streams the electron binary from GitHub's release CDN; the transfer occasionally aborts mid-stream (`ReadError: The server aborted pending request`) and `@electron/get` does not retry a streamed-body abort, so a single transient network blip previously failed the whole `build-minds-snapshot` job. Re-running `pnpm install` only re-runs the postinstalls that did not complete, so the retry is cheap. This mirrors the curl retry loop the mngr Dockerfile uses for the dugite-native asset.

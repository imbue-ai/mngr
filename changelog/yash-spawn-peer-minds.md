Minds-managed agents can now spawn and destroy peer minds, and follow each operation through to completion, by calling the desktop client's `/api/create-agent`, `/api/create-agent/{id}/status`, `/api/create-agent/{id}/logs`, `/api/destroy-agent/{id}`, `/api/destroying/{id}/status`, `/api/destroying/{id}/log`, and `/api/destroying/{id}/dismiss` endpoints through the shared latchkey gateway. `minds run` registers a `minds` latchkey service, persists a bearer token under `<minds-data-dir>/auth/api_token`, and adds a generic `minds` scope to the permission catalog with seven named permissions (`minds-create`, `minds-status`, `minds-logs`, `minds-destroy`, `minds-destroying-status`, `minds-destroying-log`, `minds-destroying-dismiss`). The scope-level path gate covers `/api/create-agent`, `/api/destroy-agent`, and `/api/destroying` together. The first peer-management attempt goes through the standard permission-request dialog ("Peer minds"); the user can grant `any` for one-click full access or any subset of the named permissions, and subsequent calls run silently. A startup migration backfills the new `minds` scope and named-permission schemas into pre-existing per-host `latchkey_permissions.json` files, and re-runs whenever the scope's path pattern itself drifts so installs that already had only the create-side schemas pick up the destroy schemas on the next restart. The permission dialog now treats a latchkey-reported `unknown` credential status as "credential present but unverifiable" and proceeds, so users no longer see a spurious "Manual credential setup required" panel for services (like `minds`) whose raw curl credentials are pre-installed at startup.

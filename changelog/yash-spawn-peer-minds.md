Minds-managed agents can now spawn peer minds and follow their creation through to completion by calling the desktop client's `/api/create-agent`, `/api/create-agent/{id}/status`, and `/api/create-agent/{id}/logs` endpoints through the shared latchkey gateway. `minds run` registers a `minds` latchkey service, persists a bearer token under `<minds-data-dir>/auth/api_token`, and adds a generic `minds` scope to the permission catalog with three named permissions (`minds-create`, `minds-status`, `minds-logs`); the scope is designed to absorb future peer-mind operations (destroy, list) as additional named permissions. The first peer-management attempt goes through the standard permission-request dialog ("Peer minds"); the user can grant `any` for one-click full access or any subset of the named permissions, and subsequent calls run silently. A startup migration backfills the new `minds` scope and named-permission schemas into pre-existing per-host `latchkey_permissions.json` files. The permission dialog now treats a latchkey-reported `unknown` credential status as "credential present but unverifiable" and proceeds, so users no longer see a spurious "Manual credential setup required" panel for services (like `minds`) whose raw curl credentials are pre-installed at startup.

Minds-managed agents can now spawn peer minds and follow their creation through to completion by calling the desktop client's `/api/create-agent`, `/api/create-agent/{id}/status`, and `/api/create-agent/{id}/logs` endpoints through the latchkey gateway's `minds-api-proxy` extension. The proxy injects the per-startup central `MINDS_API_KEY` as a bearer token (agents never see it), and the three endpoints now accept that key as an alternative to the browser session cookie. A generic `minds` scope ("Peer minds") with three named permissions (`minds-create`, `minds-status`, `minds-logs`) is added to the permission catalog and materialized -- but not pre-granted -- in every per-agent permissions baseline; the scope is designed to absorb future peer-mind operations (destroy, list) as additional named permissions. The first peer-management attempt goes through the standard permission-request dialog; the user can grant `any` for one-click full access or any subset of the named permissions, and subsequent calls run silently. A startup migration backfills the new `minds` scope and named-permission schemas into pre-existing per-host `latchkey_permissions.json` files. The permission dialog now treats a latchkey-reported `unknown` credential status as "proceed" (only `missing` / `invalid` trigger credential setup), so scopes without a registered latchkey service -- like `minds` -- no longer show a spurious "Manual credential setup required" panel.

Minds-managed agents can now enumerate their peer minds by calling the desktop client's new `GET /api/list-agents` endpoint through the shared latchkey gateway. The response is a `{"agents": [...]}` JSON object combining workspace-primary peers known to the resolver with any in-flight destroying records, so a caller can both discover which `agent_id` to target for destroy and see which peers are mid-teardown. A new `minds-list` named permission gates the endpoint under the existing `minds` scope (alongside the seven create/destroy permissions), the scope-level path pattern now also covers `/api/list-agents`, and the startup schema migration injects the new `minds-list` schema and widened scope pattern into pre-existing per-host `latchkey_permissions.json` files.

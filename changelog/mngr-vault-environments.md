Move minds to multi-environment deploys (`dev`, `staging`, `production`) backed by HCP Vault. New `minds env {deploy,list,destroy} <name>` CLI provisions (and idempotently upgrades) dynamic per-developer dev environments (Modal env + Neon DB + SuperTokens app + per-env Modal Secrets + Modal app deploys), writing a self-contained `~/.<root>/envs/<name>.toml` per dev env. `minds run` gains `--config-file <path>` and honors `MINDS_CLIENT_CONFIG_PATH` as an env-var fallback so Electron (via `just devminds-start`) can target a dynamic dev env without baking the path into the build. `scripts/push_modal_secrets.py` reads tier secrets from Vault and pushes them to Modal -- the old `.minds/<env>/*.sh` flow is gone. Every baked-in `joshalbrecht`-owned connector URL has been removed; callers now configure the URL explicitly via the per-tier `client.toml` or env var. Also: minds shutdown is cleaner now (terminates the `mngr forward` subprocess before draining the concurrency group, so reader threads no longer time out on every clean exit); the browser auto-open lands directly on the login URL with the one-time code instead of the bare origin; `list_agents`' ABORT-mode failures are now properly attributed to the failing provider so minds' auto-disable-on-auth-error handler actually fires; and `scripts/push_vault_from_file.py` pipes values as JSON on stdin to avoid the vault CLI's `@`-as-file sigil. New docs at `apps/minds/docs/environments.md` and `apps/minds/docs/vault-setup.md` walk through the new operator workflow.

Minds-managed agents can now spawn peer minds by `POST`ing to the desktop client's `/api/create-agent` endpoint through the shared latchkey gateway. `minds run` registers a `minds` latchkey service, persists a bearer token under `<minds-data-dir>/auth/api_token`, and adds a `mind-creation` scope to the permission catalog. The first spawn attempt goes through the standard permission-request dialog ("Spawn a peer mind"); approving once grants the rule for the rest of the host, and subsequent spawns run silently. A startup migration backfills the new `mind-creation` schema into pre-existing per-host `latchkey_permissions.json` files. The permission dialog now treats a latchkey-reported `unknown` credential status as "credential present but unverifiable" and proceeds, so users no longer see a spurious "Manual credential setup required" panel for services (like `minds`) whose raw curl credentials are pre-installed at startup.

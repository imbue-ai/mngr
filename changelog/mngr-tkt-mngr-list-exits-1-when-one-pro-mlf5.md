`mngr list` no longer exits 1 when a single provider is unauthed or unavailable. Per-provider construction/discovery failures (Modal auth missing, Docker daemon down, Vultr API key absent, unknown backend in config, etc.) are now demoted to warnings -- the listing of the working providers proceeds and the command exits 0, mirroring how `mngr destroy --force`'s GC pass treats per-provider failures via `_discover_hosts_for_gc` in `api/gc.py`. Host- and agent-level errors still honor `--on-error`. The failure surfaces as a `ProviderErrorInfo` on `result.errors` carrying a new `is_provider_unavailable=True` flag so callers can distinguish a recoverable unavailable provider from a genuinely fatal error (the CLI uses that flag to keep exit code at 0).

Serialize `gh pr list` invocations in the GitHub data source behind a process-wide lock. Kanpan used to fire two `gh pr list` calls (open + all) in parallel per repo; both perform the same `SearchType` GraphQL introspection beforehand, which gh caches in a single shared file using non-atomic writes. On a cold cache the concurrent writes corrupted that file, after which every `gh pr list` returned `invalid character '{' after array element` until the 24h TTL expired. Running the two calls one at a time eliminates the race. The conflicts / unresolved metadata fetches (`gh pr view`, `gh api graphql`) make uncached direct requests and continue to run in parallel.

Improved the quality of the `mngr_modal` instance unit tests (no user-visible behavior change). Migrated the volume/discovery tests (host-record listing, agent persistence, `discover_hosts`, `_list_running_host_ids`, secret building, shutdown-script generation, config defaults) off hand-rolled MagicMock providers and onto the real `FakeModalInterface`-backed `testing_provider`/`testing_modal` fixtures, so they now assert on real round-tripped state instead of mock call routing. Removed the dead MagicMock factory helpers and tightened the `unittest.mock` import ratchet.

Strengthened the `mngr_modal` Modal acceptance/release tests so they assert the observable behavior they claim to verify (no user-visible behavior change). The "Done."-only create tests now verify the actual result after create: the echo test execs a unique marker command via `mngr exec` and asserts its output (proving the created host runs commands); the custom-Dockerfile test reads back an image-baked marker file; the build-args test reads the host's recorded memory via `mngr list --format json` (`host.resource.memory_gb`, since Modal does not expose `--memory` as an in-sandbox cgroup limit); the git-transfer tests assert the transferred untracked file and the checked-out branch; and the default-image release test checks for the default image's bundled tools (`ttyd`/`uv`). The network-restriction tests now capture curl's explicit exit code (asserting a network-block code 6/7/28) instead of masking every failure with `|| echo blocked`. The invalid-snapshot test now requires both the bad snapshot id and a resolution-failure phrase instead of a too-broad generic banner. Per-test Modal host names are now suffixed with a random string to avoid collisions in shared-env offload mode, the `on_agent_created` test calls pass a real `BaseAgent` instead of a `None` placeholder, the restart-from-snapshot test was renamed and rewritten to prove the most-recent snapshot is restored, and the flaky failing-Dockerfile assertion is marked `@pytest.mark.flaky`.

Re-enabled the previously-skipped `test_idle_shutdown_creates_both_initial_and_idle_snapshots` idle-shutdown acceptance test (the only coverage of the idle-shutdown snapshot flow) and fixed the three real reasons it could not run in offload:
- It resolved the keyframe commit from `GITHUB_SHA` first, then fell back to `.mngr/image_commit_hash`. In CI `GITHUB_SHA` is the GitHub merge commit, which is absent from the offload sandbox's git history, so `git checkout` failed. It now always uses `.mngr/image_commit_hash` (the commit the sandbox actually has).
- It built the keyframe tarball into a throwaway temp directory the image build never read, while the build's `context-dir=.mngr/dev/build/` was left empty. It now builds the tarball into `.mngr/dev/build/` (replicating the repo's `pre_command_scripts.create` hook, which does not fire under the isolated test project root).
- The `mngr create` subprocess used repo-root-relative build-arg paths but inherited the isolated temp HOME that the autouse fixture chdir's into. It now runs with `cwd=repo_root`.

As a prerequisite for that test (and for any `mngr create --file=<Dockerfile>` whose later `RUN`/`ENV` references an earlier `ARG`), fixed a per-layer-caching bug in `_build_image_from_dockerfile_contents`: because each instruction is built as a separate `dockerfile_commands` layer and Docker `ARG` scope does not persist across those independent builds, a later `RUN ... ${NODE_VERSION}` expanded to an empty string (e.g. producing a 404 when downloading Node). `ARG` declarations are now carried forward and prepended to every subsequent cached layer via the new pure helper `_group_cached_layer_commands` (unit-tested directly). This changes layer cache keys, so the first build after this change rebuilds the affected layers once.

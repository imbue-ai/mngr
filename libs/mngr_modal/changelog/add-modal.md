Backs the "Modal (1-day ephemeral)" compute option in DIRECT mode (the local machine authenticates to Modal with its own token, `modal token new`). The default sandbox size is now **2 CPU / 4 GB** and the default sandbox + idle timeouts are **24h** (Modal's "1-day ephemeral" ceiling), matching the lima/docker convention instead of the old 1-core / 900s defaults that left a sandbox tiny and reaped mid-session.

`ModalMode.PROXIED` remains a `NotImplementedError` stub: Modal is imbue-internal and its sandboxes are ephemeral, so it is not offered for public/keyless use -- anyone running it authenticates to Modal directly via DIRECT.

`start_host` no longer auto-restores the bare "initial" snapshot. That snapshot is captured right after provisioning, before the agent runs, so restoring it produces a sandbox with no agent -- an agent-less, billing orphan (which is what happened when minds auto-restarted an expired Modal workspace on open). Auto-restart (no explicit `snapshot_id`) now selects the most recent **resumable** snapshot (a stop/pause snapshot carrying the host's state) and raises `NoResumableSnapshotModalMngrError` when the only snapshot is the initial one, so the workspace should be recreated rather than restarted. An explicit `snapshot_id` can still restore the initial snapshot (and now logs a warning that the resulting sandbox is stateless).

`start_host` now clears `stop_reason` on the persisted host record when it resumes a host from a snapshot. Previously a host that was resumed (from PAUSED/STOPPED) and then later hard-killed was mis-derived as PAUSED/STOPPED instead of CRASHED, so it was never garbage-collected and wrongly showed a Resume control. This mirrors the lima provider's behavior on restart.

`stop_host` no longer silently swallows a failed `sandbox.terminate()`. A failed terminate means the Modal sandbox is still running (a billing orphan), so it now raises a `ModalMngrError`; `destroy_host` records it as a cleanup failure so `mngr destroy` reports failure (and exits non-zero) instead of falsely succeeding while the sandbox keeps running.

`ModalProviderConfig` now validates that `default_sandbox_timeout + shutdown_buffer_seconds` does not exceed Modal's hard cap of 86,400s (24h) -- the timeout actually sent to Modal is the sum of the two -- and raises a clear `ConfigStructureError` if it would. The `is_persistent` field docstring now states its real consequence: when False the Modal app is ephemeral and the sandbox dies when the creating `mngr-create` subprocess exits (it is not a mere test marker).

Added an acceptance test (`test_destroy_with_stale_volume.py`) covering the core fix where `mngr destroy` could not resolve an agent on a RUNNING Modal sandbox whose per-agent state-volume record (`/hosts/<host_id>/<agent_id>.json`) was missing/stale. It deletes that record on a running sandbox and asserts the resolver still finds the agent by reading the host live.

- Added `mngr_foreman`, an always-on web remote-control plugin for mngr agents. `mngr foreman` runs a single Flask server over every agent in mngr's view: a mobile-friendly chat UI for claude agents (live transcript with markdown, syntax highlighting, KaTeX, mermaid, inline images and file uploads; send messages; interrupt) and a web terminal (xterm.js over a pty bridge) for any agent type. No code is deployed to target boxes and there is no auth by design — bind to a private network. Includes a warm SSH connection pool that keeps resolution and connections hot for fast sends/reads. Third-party frontend libraries (marked, xterm, highlight.js, KaTeX, mermaid, Atkinson Hyperlegible fonts) are pinned by exact URL + sha256 and fetched into a local cache on first run instead of being vendored in the repo (see `ASSETS.md`), so the package stays small and offline boxes degrade gracefully. The warm-pool keepalive fans out across hosts with a bounded per-command timeout (one unresponsive host no longer stalls the fleet), and `run_on_host` serializes on a per-*host* lock so two agents sharing one cached connection can't drive it concurrently. The transcript stream polls adaptively — ~0.3s right after a send and while the agent is working, decaying to ~1s and ~4s while idle — cutting the visible round-trip without extra load (stat-before-read keeps fast polls cheap).
- Added `mngr foreman -d`/`--background`: run the server detached in the background. It prints `foreman started (PID <pid>) on http://<host>:<port>/  — logs: <logfile>  stop: kill <pid>` and returns immediately (exit 0) while the server keeps serving. Rather than forking the already-multi-threaded process (a fork+threads deadlock hazard) and fighting the parent-death watcher that SIGTERMs a foreground server when its launcher exits, `-d` re-execs `mngr foreman` as a fresh, fully detached child: its own session (`start_new_session`/`setsid`, no controlling tty), stdin from `/dev/null`, and stdout+stderr appended to `--foreman-log-file` (default `~/.mngr/foreman.log`; named `--foreman-log-file` because `--log-file` is already mngr's structured-log flag). A single `MNGR_FOREMAN_DAEMONIZED` env marker tells that child to run in the foreground (not re-daemonize) and to skip the parent-death watcher, so it outlives its launcher. A `--pid-file` (default `~/.mngr/foreman.pid`) guards against a second copy — `-d` refuses to start when a live PID is recorded there (`foreman already running (PID <pid>)`, exit 1) and treats a stale (dead) PID as not running. The pinned frontend assets are ensured in the foreground parent *before* forking, so a slow or failing first-run download surfaces on the terminal instead of being buried in the daemon log; the child's own `ensure_assets` is then a cache hit.
- Added `pi-coding` chat support: the live transcript, working/waiting status, and composer now render for `pi-coding` agents, not just claude (the agent list and web terminal already worked for any type). Rather than write a second bespoke parser, foreman consumes pi's already-normalized *common transcript* (`events/pi-coding/common_transcript/events.jsonl`, emitted by pi's mngr lifecycle extension) through a thin normalizer (`pi_transcript.py`: validate + dedup on `event_id`, rename pi's `finish_reason` to the `stop_reason` the claude events use, re-cap tool output only when foreman is configured stricter than pi's own 2000-char cap). The transcript source, parser, and whether the composer's blocked-state comes from a `tmux capture-pane` are now selected by a per-agent-type `TranscriptStrategy` registry (`harness.py`, keyed on `agent.type`) — the single seam that makes foreman multi-harness. claude keeps its raw-session-JSONL parser and `❯`-numbered-choice dialog detection unchanged; pi (which runs every tool unattended and never blocks the composer) uses the common transcript and skips the pane capture, so its working/waiting dot rides the same `active` RUNNING/WAITING marker claude uses. An unsupported agent type still shows the "no transcript" notice. Known gaps for pi (accepted): tool calls render from a ~200-char `input_preview` only (no `input_full`, so no diff view) and inline images are dropped, both because pi's common transcript carries neither.

- Added `mngr_foreman`, an always-on web remote-control plugin for mngr agents. `mngr foreman` runs a single Flask server over the *actively-running coding agents* in mngr's view â€” claude, codex, opencode, and pi-coding agents in a RUNNING or WAITING state; stopped/done/tombstoned and non-coding system agents are hidden. It gives a mobile-friendly chat UI for claude agents (live transcript with markdown, syntax highlighting, KaTeX, mermaid, inline images and file uploads; send messages; interrupt) and a web terminal (xterm.js over a pty bridge). No code is deployed to target boxes and there is no auth by design â€” bind to a private network. A plain discovery loop re-lists agents in-process every ~3s (no `mngr observe` subprocess) and refreshes that live set. The connection pool keeps an SSH connection **always warm** for every agent in the set â€” pinging each every ~10s, reconnecting immediately on a drop, and dropping the connection the moment an agent leaves the set â€” so the *first* send and terminal-open are as fast as every later one (no cold start). Foreman never auto-starts or resurrects a stopped agent. Third-party frontend libraries (marked, xterm, highlight.js, KaTeX, mermaid, Atkinson Hyperlegible fonts) are pinned by exact URL + sha256 and fetched into a local cache on first run instead of being vendored in the repo (see `ASSETS.md`), so the package stays small and offline boxes degrade gracefully. The warm-pool keepalive fans out across hosts with a bounded per-command timeout (one unresponsive host no longer stalls the fleet), keeps *both* channels hot â€” the paramiko connection the send/transcript paths use and the system-ssh ControlMaster socket the web terminal reuses (pre-opened so the first terminal open is warm too) â€” and `run_on_host` serializes on a per-*host* lock so two agents sharing one cached connection can't drive it concurrently. The transcript stream polls at a fixed, consistently fast rate over the warm connection (stat-before-read keeps each poll cheap when the file hasn't grown), and the browser keeps its list and transcript streams open regardless of tab visibility.
- Added `mngr foreman -d`/`--background`: run the server detached in the background. It prints `foreman started (PID <pid>) on http://<host>:<port>/  â€” logs: <logfile>  stop: kill <pid>` and returns immediately (exit 0) while the server keeps serving. Rather than forking the already-multi-threaded process (a fork+threads deadlock hazard) and fighting the parent-death watcher that SIGTERMs a foreground server when its launcher exits, `-d` re-execs `mngr foreman` as a fresh, fully detached child: its own session (`start_new_session`/`setsid`, no controlling tty), stdin from `/dev/null`, and stdout+stderr appended to `--foreman-log-file` (default `~/.mngr/foreman.log`; named `--foreman-log-file` because `--log-file` is already mngr's structured-log flag). A single `MNGR_FOREMAN_DAEMONIZED` env marker tells that child to run in the foreground (not re-daemonize) and to skip the parent-death watcher, so it outlives its launcher. A `--pid-file` (default `~/.mngr/foreman.pid`) guards against a second copy â€” `-d` refuses to start when a live PID is recorded there (`foreman already running (PID <pid>)`, exit 1) and treats a stale (dead) PID as not running. The pinned frontend assets are ensured in the foreground parent *before* forking, so a slow or failing first-run download surfaces on the terminal instead of being buried in the daemon log; the child's own `ensure_assets` is then a cache hit.
- Added `mngr foreman install` / `mngr foreman uninstall`: register foreman as a systemd service so it is controlled via `systemctl` from then on (replacing hand-rolled unit files / `nohup` / `-d` in setup scripts). `mngr foreman` is now a command group â€” bare `mngr foreman [--host â€¦] [--port â€¦] [-d]` still runs the server exactly as before (`invoke_without_command`), and `install`/`uninstall` are subcommands. `install [--host 0.0.0.0] [--port 8700]` writes `/etc/systemd/system/foreman.service` (Type=simple, Restart=always/RestartSec=3, After=network.target docker.service, WantedBy=multi-user.target) whose `ExecStart` is the *absolute* path to the running mngr binary (resolved via `MNGR_FOREMAN_MNGR_BINARY` â†’ `shutil.which` â†’ the venv bin dir, never a bare `mngr`, so the unit doesn't depend on PATH), `User=` the current user, `WorkingDirectory=` the mngr checkout (the `â€¦/.venv` parent) or `$HOME`, and an `Environment=PATH=` covering the binary dir, `~/.local/bin`, and the standard system dirs; it then runs `systemctl daemon-reload` and `systemctl enable --now foreman`. The privileged steps (write via `tee`, `systemctl`) are re-invoked through `sudo` when not already root, so a sudo-capable user can just run `mngr foreman install`; it is idempotent (re-running rewrites the unit and restarts) and prints the URL plus the `systemctl status|start|stop|restart foreman` and `journalctl -u foreman -f` control commands. `uninstall` stops+disables the service and removes the unit (a no-op if it was never installed).

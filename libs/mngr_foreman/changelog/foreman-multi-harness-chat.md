- foreman now renders live chat (transcript + composer + working/needs-input status) for `codex`, `opencode`, and `pi-coding` agents, not just `claude` (the agent list and web terminal already worked for any type). These three consume mngr's already-normalized, agent-agnostic *common transcript* (`events/<type>/common_transcript/events.jsonl`) through a single shared normalizer (`common_transcript_parser.py`: validate + dedup on `event_id`, whitelist the `user_message`/`assistant_message`/`tool_result` record types, require a timestamp, rename the common schema's `finish_reason` to the `stop_reason` the claude events use, and re-cap tool output to foreman's own bound) — one parser for all three, since their records are the identical envelope. Chat dispatch now flows through a per-agent-type `TranscriptStrategy` registry (`harness.py`, keyed on `agent.type`): each row picks the transcript subpath, the parser, and whether needs-input is found by a `tmux capture-pane` `❯`-dialog scrape. claude is unchanged — it keeps its bespoke raw-session-JSONL parser and pane-dialog detection. codex, opencode, and pi-coding skip the pane scrape: pi-coding never blocks the composer, and codex/opencode surface a tool-approval block through mngr's pane-less `waiting_reason == PERMISSIONS` field (read straight off `AgentDetails`), so their working/waiting/needs-input dot rides the generic `active` RUNNING/WAITING signal with no host round-trip. The agent-list card now exposes `supports_chat` in place of `is_claude`. Known gaps for the common-transcript harnesses (accepted): tool calls render from a ~200-char `input_preview` only (no `input_full`, so no diff view) and inline images are dropped, both because the common transcript carries neither.

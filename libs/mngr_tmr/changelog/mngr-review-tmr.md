Reworked how TMR runs report their findings, so that repeated per-test fixes get consolidated instead of accumulating across runs.

Escalations are now a first-class, orthogonal part of a test agent's outcome. Previously an agent could only signal "this needs wider attention" by degrading a change's status to `BLOCKED`, which forced a choice between reporting a successful fix and reporting a systemic problem. The `BLOCKED` status is gone; outcomes now carry an `escalations` list alongside (and independent of) their changes, so a test that passes cleanly can still raise one.

Escalations come in two kinds. `BLOCKER` is the old case: the agent could not proceed without a shared change. `SHARED_PATTERN` is new and is aimed at the common failure mode where an agent's local fix works, but the same fix already exists on sibling tests -- meaning one suite-wide change should have replaced N local ones. The mapper prompt now instructs agents to grep their siblings before committing and escalate when they find they are the Nth agent applying the same patch.

Every pytest invocation TMR makes now passes an explicit `--timeout` (120s), shared by the mapper and the reducer's verification runs so the two cannot disagree. Agents are told not to add or raise `@pytest.mark.timeout` markers on tests; a test genuinely needing longer is a finding to escalate. This removes the churn where each run's agents added timeout markers that the next run's agents adjusted again.

Test agents no longer write changelog entries. One agent per test meant a PR carrying dozens of files describing a single change set; the reducer now writes one unified entry per affected project, drawing on the mapper summaries.

The reducer now opens the run's pull request itself, rather than the workflow doing it from a branch name. This lets the description carry the run's actual findings: a mechanical title that reads at a glance (`TMR tmr-mngr 2026-07-21: 12 fixed, 3 escalated`), a breakdown of every mapper's status, and a table of all escalations. A run that found nothing to integrate still opens a PR, on an empty commit, so its escalations are not silently dropped. The title and both tables are generated by a new `pr_summary` module rather than tabulated by the agent, so the counts are accurate.

The HTML report's "Blocked" section is renamed "Unresolved" (results where the agent tried and could not land a change), and the escalations block now aggregates escalations from every test agent as well as the integrator, labelled by kind and source.

Fixed a latent bug in report generation: parsed outcomes were cached by agent name alone, so the same agent name under two different output directories would return the first directory's outcome.

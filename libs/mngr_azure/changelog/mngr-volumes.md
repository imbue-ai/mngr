`mngr azure prepare` now also creates a private Azure Storage account + Blob container ("state bucket") that holds mngr's control-plane state, and `mngr azure cleanup` deletes it. Since cleanup runs after the refuse-while-VMs-exist check, any state left in the account is orphaned (hosts no longer running as VMs), so it refuses to delete a non-empty account rather than silently dropping offline records; pass the new `--force` flag to delete the account and its remaining state. This mirrors the AWS S3 state bucket.

The full host record and per-agent records are written to the bucket (by the mngr host machine, using the operator's own Azure credentials) instead of being mirrored into VM tags. This fixes two problems the VM-tag mirror had: per-agent `labels` larger than 256 chars are no longer silently dropped, and a deallocated VM's full host record (not just a lossy tag subset) is now readable while it is stopped. The bucket is required infrastructure -- there is no VM-tag fallback -- so when it has not been provisioned mngr raises an actionable error pointing at `mngr azure prepare` (on the create/label write path as well as on offline reads).

The storage-account name defaults to a deterministic `mngrst<hash>` derived from the subscription + resource group; override it with `state_storage_account_name` in `[providers.azure]`. The container is named `mngr-state`.

Blob data-plane access uses the same `DefaultAzureCredential` the provider already uses (no storage keys), so the operator needs the `Storage Blob Data Contributor` role on the state storage account to read/write host state, in addition to the storage-account create/delete permission that `prepare`/`cleanup` use. The bucket is prepare's primary job: a missing storage permission or any account/container create failure now fails the command (rather than warning and continuing with a network-only prepare).

Offline `host_dir` (spec part c, on by default): a deallocated VM's `host_dir` is now readable without SSH. When `is_offline_host_dir_enabled` is on (the default) and the state bucket exists, an on-box systemd oneshot+timer daemon syncs `<host_dir>` to the bucket's `hosts/<host_id>/host_dir/` prefix every 60s (and once more on `mngr stop`, before the VM deallocates) using `azcopy sync` authenticating as the VM's managed identity via MSI (`--auth-mode login`; no keys on the box). `mngr event` / `mngr transcript` then work against a stopped agent. Set `is_offline_host_dir_enabled = false` in `[providers.azure]` to disable the sync (offline host metadata still works via the bucket).

When `is_offline_host_dir_enabled` is on, `mngr azure prepare` provisions the bucket-write identity that lets a VM push its `host_dir`: a user-assigned managed identity plus a `Storage Blob Data Contributor` role assignment scoped to **just the state storage account** (least privilege -- never the resource group or subscription). A permission failure now fails prepare (the identity is part of its setup when the feature is on). `mngr azure cleanup` deletes the identity (idempotent; raises on a delete failure). VM create attaches the identity when the feature is on and it exists (an operator-supplied identity still takes precedence).

Provisioning the identity needs `Microsoft.ManagedIdentity/userAssignedIdentities/write` + `Microsoft.Authorization/roleAssignments/write` (Owner or User Access Administrator). When offline `host_dir` is requested for a host whose VM has no attached managed identity, mngr logs a non-fatal diagnostic pointing at re-running `mngr azure prepare` instead of returning an empty volume.

`mngr azure prepare` now also creates a private Azure Storage account + Blob container ("state bucket") that holds mngr's control-plane state, and `mngr azure cleanup` deletes it (refusing while any managed host still has state in it). This mirrors the AWS S3 state bucket.

When the state bucket exists, the full host record and per-agent records are written to it (by the mngr host machine, using the operator's own Azure credentials) instead of being mirrored into VM tags. This fixes two problems the legacy VM-tag mirror had: per-agent `labels` larger than 256 chars are no longer silently dropped, and a deallocated VM's full host record (not just a lossy tag subset) is now readable while it is stopped. When the bucket has not been created (older `prepare`, or a credential without storage permissions), mngr falls back to the legacy per-agent VM tag mirror so existing deployments keep working.

The storage-account name defaults to a deterministic `mngrst<hash>` derived from the subscription + resource group; override it with `state_storage_account_name` in `[providers.azure]`. The container is named `mngr-state`.

Blob data-plane access uses the same `DefaultAzureCredential` the provider already uses (no storage keys), so the operator needs the `Storage Blob Data Contributor` role on the state storage account to read/write host state, in addition to the storage-account create/delete permission that `prepare`/`cleanup` use. A missing storage permission during `prepare` degrades to a warning (the network prepare still succeeds; offline state falls back to VM tags).

Offline `host_dir` (spec part c, on by default): a deallocated VM's `host_dir` is now readable without SSH. When `is_host_dir_synced_to_bucket` is on (the default) and the state bucket exists, an on-box systemd oneshot+timer daemon syncs `<host_dir>` to the bucket's `hosts/<host_id>/host_dir/` prefix every 60s (and once more on `mngr stop`, before the VM deallocates) using `azcopy sync` authenticating as the VM's managed identity via MSI (`--auth-mode login`; no keys on the box). `mngr event` / `mngr transcript` then work against a stopped agent. Set `is_host_dir_synced_to_bucket = false` in `[providers.azure]` to disable the sync (offline host metadata still works via the bucket).

`mngr azure prepare` gains `--host-dir-identity {auto,require,skip}` (default `auto`), which provisions the bucket-write identity that lets a VM push its `host_dir`: a user-assigned managed identity plus a `Storage Blob Data Contributor` role assignment scoped to **just the state storage account** (least privilege -- never the resource group or subscription). `auto` warns and continues on a permission failure (the network + bucket prepare still succeed); `require` fails the command; `skip` does not attempt it. `mngr azure cleanup` deletes the identity (idempotent). VM create attaches the identity when the feature is on and it exists (an operator-supplied identity still takes precedence).

Provisioning the identity needs `Microsoft.ManagedIdentity/userAssignedIdentities/write` + `Microsoft.Authorization/roleAssignments/write` (Owner or User Access Administrator). When offline `host_dir` is requested for a host whose VM has no attached managed identity, mngr logs a non-fatal diagnostic pointing at `mngr azure prepare --host-dir-identity require` instead of returning an empty volume.

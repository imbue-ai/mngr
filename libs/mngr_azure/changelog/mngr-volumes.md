`mngr azure prepare` now also creates a private Azure Storage account + Blob container ("state bucket") that holds mngr's control-plane state, and `mngr azure cleanup` deletes it (refusing while any managed host still has state in it). This mirrors the AWS S3 state bucket.

When the state bucket exists, the full host record and per-agent records are written to it (by the mngr host machine, using the operator's own Azure credentials) instead of being mirrored into VM tags. This fixes two problems the legacy VM-tag mirror had: per-agent `labels` larger than 256 chars are no longer silently dropped, and a deallocated VM's full host record (not just a lossy tag subset) is now readable while it is stopped. When the bucket has not been created (older `prepare`, or a credential without storage permissions), mngr falls back to the legacy per-agent VM tag mirror so existing deployments keep working.

The storage-account name defaults to a deterministic `mngrst<hash>` derived from the subscription + resource group; override it with `state_storage_account_name` in `[providers.azure]`. The container is named `mngr-state`.

Blob data-plane access uses the same `DefaultAzureCredential` the provider already uses (no storage keys), so the operator needs the `Storage Blob Data Contributor` role on the state storage account to read/write host state, in addition to the storage-account create/delete permission that `prepare`/`cleanup` use. A missing storage permission during `prepare` degrades to a warning (the network prepare still succeeds; offline state falls back to VM tags).

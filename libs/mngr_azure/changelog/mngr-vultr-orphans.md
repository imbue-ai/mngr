mngr_azure: the session-end leak detector (`pytest_sessionfinish`) now *fails* the test session instead of skipping silently when `MNGR_AZURE_RELEASE_TESTS=1` is set but Azure credentials cannot be resolved, no subscription is resolvable, or the management clients cannot be built. It still no-ops on ordinary unit-only runs (opt-in unset). A release run that cannot authenticate cannot have scanned for leaked VMs, so skipping would silently green a run that did no cleanup; failing surfaces the misconfiguration.

Add `imbue.mngr_azure.cleanup`, an out-of-band, age-based reaper for leaked test VMs (and orphaned NICs / public IPs). It scans the mngr resource group for VMs tagged `mngr-pytest-launched=true` (which production VMs never carry) whose `mngr-created-at` tag is older than a max age and deletes them, reclaiming unattached pytest-launched network resources in the same pass. The same `find_old_test_vms` / `force_delete_vms` / `reclaim_orphan_test_network` functions now back the session-end leak detector, so the two cleanup paths can never drift. A new CI job (`cleanup-azure-instances`, via `scripts/cleanup_old_azure_test_instances.py`) runs this reaper on every push to main and pull request, so orphans from sessions killed before `pytest_sessionfinish` could run are still cleaned up.

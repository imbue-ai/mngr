mngr_azure: the session-end leak detector (`pytest_sessionfinish`) now *fails* the test session instead of skipping silently when `MNGR_AZURE_RELEASE_TESTS=1` is set but Azure credentials cannot be resolved, no subscription is resolvable, or the management clients cannot be built. It still no-ops on ordinary unit-only runs (opt-in unset). A release run that cannot authenticate cannot have scanned for leaked VMs, so skipping would silently green a run that did no cleanup; failing surfaces the misconfiguration.

Add an out-of-band, age-based reaper for leaked test VMs tagged `mngr-pytest-launched=true` (which production VMs never carry), plus orphaned NICs / public IPs from capacity-failed creates. A new CI job (`cleanup-azure-instances`, via `scripts/cleanup_old_azure_test_instances.py`) runs it on every push to main and pull request, so orphans from sessions killed before `pytest_sessionfinish` could run are still cleaned up.

The VM-reaping logic is the shared `imbue.mngr_vps.leak_cleanup`; `imbue.mngr_azure.cleanup` now supplies only Azure's `CreatedAtExtractor` (require the `mngr-pytest-launched` tag, read the ISO `mngr-created-at` tag). The orphaned-NIC / public-IP reclaim is delegated to the client's own production `AzureVpsClient.reclaim_orphaned_network_resources` (the same sweep `mngr gc` runs) instead of a bespoke copy. The session-end hook + standalone script are thin wrappers that build an `AzureVpsClient`, reclaim network, then call the shared scan/destroy; the hook fails the session on a scan error instead of silently passing.

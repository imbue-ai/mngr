New `imbue-mngr-donate` plugin: `mngr donate` spends spare Claude capacity (from the `imbue-mngr-usage` snapshot, a dependency) on a donation skill instead of letting idle quota expire. When the 5h window has budget and the week is under pace, it launches a headless Claude agent that runs the skill to completion, then auto-cleans up; `--start`/`--stop` install/remove a launchd LaunchAgent (macOS) so idle quota drains on a schedule (launchd, not cron, so it runs in the login session where Claude's keychain token lives). The donation skill (code + prompts) is owned by its upstream git repo -- `mngr donate` checks it out at `--skill-ref` (a branch to track, or a pinned commit for a reviewed version) into a host-dir cache and points the agent at it, so the skill updates without an mngr release.

Fixed: the donation agent failed with "Not logged in" even when a valid `CLAUDE_CODE_OAUTH_TOKEN` was available (from the keychain stash or the caller's environment). The token was placed in the `mngr create` subprocess env but never forwarded into the agent's sanitized environment -- `mngr create` only passes through vars named via `--pass-env`. `build_create_argv` now passes `--pass-env CLAUDE_CODE_OAUTH_TOKEN`, so the long-lived OAuth token reaches the headless claude agent and it authenticates.

Fixed: the per-run log only captured the agent's assistant text (reasoning + final summary), not the raw stream-json (tool calls, tool results) -- `mngr create --foreground` streams via `StreamJsonReader`, which extracts only assistant text deltas. The README's "full event stream" claim was inaccurate. donate now also writes a sibling `<agent>-<ts>.stream.jsonl` by live-tailing the agent's own `stdout.jsonl` before the agent auto-destroys (which would otherwise delete it); secret values (OAuth tokens, `Authorization: Bearer ...`, `sk-ant-...`) are redacted before writing. The assistant-text log is unchanged.

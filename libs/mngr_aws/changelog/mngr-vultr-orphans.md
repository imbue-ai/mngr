mngr_aws: the session-end leak detector (`pytest_sessionfinish`) now *fails* the test session instead of skipping silently when `MNGR_AWS_RELEASE_TESTS=1` is set but AWS credentials cannot be resolved (or the EC2 client cannot be built). It still no-ops on ordinary unit-only runs (opt-in unset). A release run that cannot authenticate cannot have scanned for leaked EC2 instances, so skipping would silently green a run that did no cleanup; failing surfaces the misconfiguration.

Add `imbue.mngr_aws.cleanup`, an out-of-band, age-based reaper for leaked test EC2 instances. It scans for instances tagged `mngr-pytest-launched=true` (which production instances never carry) whose `LaunchTime` is older than a max age and terminates them. The same `find_old_test_instances` / `terminate_test_instances` functions now back the session-end leak detector, so the two cleanup paths can never drift. A new CI job (`cleanup-aws-instances`, via `scripts/cleanup_old_aws_test_instances.py`) runs this reaper on every push to main and pull request, so orphans from sessions killed before `pytest_sessionfinish` could run are still cleaned up.

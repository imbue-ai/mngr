mngr_gcp: the session-end leak detector (`pytest_sessionfinish`) now *fails* the test session instead of skipping silently when `MNGR_GCP_RELEASE_TESTS=1` is set but GCP Application Default Credentials cannot be resolved, no default project is configured, or the GCE client cannot be built. It still no-ops on ordinary unit-only runs (opt-in unset). A release run that cannot authenticate cannot have scanned for leaked GCE instances, so skipping would silently green a run that did no cleanup; failing surfaces the misconfiguration.

Add `imbue.mngr_gcp.cleanup`, an out-of-band, age-based reaper for leaked test GCE instances. It scans for instances labeled `mngr-pytest-launched=true` (which production instances never carry) whose `creation_timestamp` is older than a max age and deletes them. An instance whose `creation_timestamp` is missing or unparseable is left alone rather than crashing the scan (which runs inside `pytest_sessionfinish`), matching the AWS / Azure / Vultr reapers, which never act on an instance whose age they cannot establish. The same `find_old_test_instances` / `force_delete_instances` functions now back the session-end leak detector, so the two cleanup paths can never drift. A new CI job (`cleanup-gcp-instances`, via `scripts/cleanup_old_gcp_test_instances.py`) runs this reaper on every push to main and pull request, so orphans from sessions killed before `pytest_sessionfinish` could run are still cleaned up.

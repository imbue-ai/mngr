Fixed mngr's Claude hooks leaking into "normal" (non-mngr) Claude config. mngr previously wrote its readiness/credential/permission hooks into the project's `.claude/settings.local.json`, which plain `claude` runs in that directory also read -- so the hooks (e.g. an activity-event `mkdir`) fired outside mngr where `$MNGR_AGENT_STATE_DIR`/`$MNGR_HOST_DIR` are unset, producing errors like `mkdir: cannot create directory '/events': Permission denied`.

mngr now writes all of its own hooks to a private per-agent file, `$MNGR_AGENT_STATE_DIR/plugin/claude/mngr_managed_settings.json`, and launches `claude --settings <that file>`. The file is owned outright by mngr and rewritten fresh on every provision, so there is no cross-version accumulation and nothing lands in a file plain `claude` reads.

A user-supplied `--settings` (whether in an agent type's `cli_args` or passed through on the `mngr create` command line) is now merged into that managed file rather than competing with it. `claude` honors only the last `--settings` on its command line, so mngr folds the user's settings -- inline JSON or a referenced file -- into the managed file (hooks from both sides are kept) and drops the user's own `--settings` flag. This keeps a user's own hooks (e.g. a `SessionStart` env-export hook) working without silently clobbering mngr's readiness hooks, or vice versa.

Note: this stops *new* leaks; it does not remove hooks already written into existing `settings.local.json` files by a prior mngr -- clean those up manually if present.

`mngr create` no longer requires the project's `.claude/settings.local.json` to be gitignored across the board. mngr writes its own hooks to the private managed file, so that requirement now applies only when the `claude_subagent_proxy` plugin (PROXY mode) actually needs to rewrite user-defined Stop hooks in `settings.local.json` -- enforced by that plugin, at the point it writes.

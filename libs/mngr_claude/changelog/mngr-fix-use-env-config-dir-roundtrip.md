Fixed a regression where the deprecated `use_env_config_dir` claude agent-type option stopped working on the in-host `mngr create` path, which made `minds` mind creation hang forever on "waiting for initial chat agent". A config that set only `use_env_config_dir = true` (the deprecated inverse of `isolate_local_config_dir`, used by the forever-claude-template to share the services agent's `$CLAUDE_CONFIG_DIR`) was rejected with a spurious "Contradictory Claude config" error: the config-load pipeline serializes and reparses the config, which materialized the default `isolate_local_config_dir=True` into the set fields, so the alias reconciliation misread it as a hand-written contradiction.

The deprecated alias is now resolved to the canonical `isolate_local_config_dir` (its inverse) at config-load time, before the config is parsed -- so the resolved value is genuinely set from the start and survives every later config-layer and `parent_type` overlay merge and the agent-construction reparse. The alias is kept (not dropped) so the provisioning-time deprecation warning still fires, and a genuine hand-written contradiction (both keys set to the same, non-inverse value) is still rejected.

Note: fully fixing the `parent_type` inheritance path (which `minds` uses -- `[agent_types.main]` with `parent_type = "claude"`) additionally depends on the overlay-merge `model_fields_set` preservation fix; this change is stacked on top of it.

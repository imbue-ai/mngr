`mngr message` now declines, upfront and with a clear error, TUI-local slash commands that open an interactive panel a headless send cannot drive. Commands verified to always panel (`/login`, `/cost`, `/usage`, `/status`, `/mcp`, `/plugin`, `/add-dir`, `/config`, `/help`, `/theme`, `/permissions`, `/hooks`, `/resume`, `/memory`) are declined regardless of arguments; `/model` and `/effort` are declined only when bare, since their argument forms (`/model opus`, `/effort high`) apply directly and still pass. The error names the command and points to `mngr connect <agent>` to run it interactively. Unknown commands and user-defined commands are unaffected.

As defense in depth for panels opened by other means (e.g. a human over `mngr connect`), a send to an agent already showing a panel is also declined fast: the panel is detected by its Escape-to-dismiss footer (`Esc to cancel` / `Esc to clear`), which is present in these panels but absent from the ready and mid-response states, so ready and busy agents are unaffected.

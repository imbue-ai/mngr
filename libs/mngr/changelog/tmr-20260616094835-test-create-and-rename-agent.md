Made the `mngr rename` e2e tests (`test_create_and_rename_agent` and `test_rename_dry_run_does_not_rename`) robust to environments where cloud provider plugins are installed but unconfigured. The verification step now scopes `mngr list` to the local provider (`--provider local`), where the renamed agent actually runs, instead of doing a full cross-provider discovery. Previously the discovery would fail with a non-zero exit when a registered-but-unconfigured provider (e.g. AWS, which surfaces missing credentials as a hard error) could not be reached, even though that provider is irrelevant to a rename test.

Removed the now-incorrect `@pytest.mark.rsync` marker from both tests: rsync was only exercised incidentally by the remote-provider discovery path, not by the rename or local-agent operations, so the resource guard correctly flagged the marker as unused once the listing was scoped to the local provider. This mirrors the existing `mngr list --provider local` tests, which are not rsync-marked.

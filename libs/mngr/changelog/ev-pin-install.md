`mngr plugin add` (and the interactive plugin-install wizard) now pin dependency resolution to the versions mngr was tested with, using a lockfile-derived constraints file shipped inside the wheel. Previously a plugin install did a fresh, unpinned PyPI resolution, so a newly published transitive dependency could be pulled into a version mngr had never been tested against and break the CLI on the next run.

A single constraints file (`libs/mngr/constraints.txt`, covering the whole third-party dependency tree from the workspace lock) is generated from the lockfile, committed to the repo, and force-included into the published wheel at `imbue/mngr/constraints.txt`, so it is present in every real install. It is passed to `uv tool install --constraints`. If the bundled file is somehow missing, that indicates the mngr installation itself is broken (a packaging bug), not that any one plugin is at fault, so plugin add, the install wizard, and plugin remove all abort with a clear error rather than silently resolving the remaining tree unpinned.

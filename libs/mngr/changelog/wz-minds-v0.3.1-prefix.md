Re-pin `cryptography>=42.0,<47` in `libs/mngr/pyproject.toml` to fix the Apple-M5 lima-VZ SIGILL regression.

`cryptography` 47.0.0+ ships an aarch64 Rust OpenSSL wheel that uses a CPU instruction lima-VZ on Apple Silicon M5 does not expose to the guest -- importing it (via the paramiko import chain during `mngr` tool install / `mngr plugin add`) SIGILLs with exit 132, crashing the in-VM agent into a recovery state. 46.x imports cleanly; M4 Max + native Linux are unaffected. The cap was added in `f3d498bef`, then reverted in `68f3f6b7a` with no stated reason, which silently regressed the `launch_to_msg` agent on the M5 runner. Re-confirmed by a controlled experiment on the self-hosted runner: same binary, FCT `vendor/mngr` pinned `<47` -> full green; unpinned (`48.0.0`) -> W1 recovery/SIGILL. The comment now warns against dropping the cap without re-verifying launch-to-msg on the M5 runner.

Fixed e2e tutorial tests hanging on sandboxed CI hosts: the shared `e2e` fixture now exports `AWS_EC2_METADATA_DISABLED=true` and `NO_GCE_CHECK=true` for the mngr subprocess. Every discovery-based command builds all installed provider plugins (including AWS and GCP) before scanning, and with no cloud credentials configured their credential chains fall through to probing the cloud metadata servers (AWS IMDS, GCP `is_on_gce`). On a sandbox where those link-local IPs are routable but black-holed, the probes block well past their nominal timeouts, so a command that should fail fast (e.g. `mngr event no-such-agent --follow` resolving an unknown agent) instead hung until killed. Disabling the probes makes credential resolution fail fast, exactly as on a real machine with no metadata server.

Strengthened `test_troubleshoot_follow_events_unknown_agent` to assert the failure is the clean "Could not find agent" target-resolution diagnostic (matching its non-follow sibling), not merely that the agent name appears somewhere in stderr.

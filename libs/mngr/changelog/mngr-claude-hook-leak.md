The `__extend` config operator now merges dicts **recursively**. A nested `key__extend` inside an `__extend` value merges deeper (extending `base[key]`) instead of replacing it, while a nested bare `key` still replaces at that level (preserving its siblings). This is backward-compatible: any value that nests no inner `__extend` marker merges exactly as the previous one-level operator did, so no existing config changes behavior.

The deferred-`__extend` carveout (markers preserved verbatim through config-load for later resolution against a concrete runtime base) now covers a small registry of paths: the existing `create_templates.<name>` options (resolved at `mngr create` time) plus any path under `agent_types.<name>.settings_overrides` (resolved at Claude provisioning time against the home `settings.json`). This lets a claude agent type's `settings_overrides` merge onto your home Claude settings with config-consistent semantics -- bare keys assign (with the narrowing guard), `key__extend` merges -- rather than a blanket deep-merge.

A field marked as a settings *patch* (e.g. a claude agent type's `settings_overrides`) now **accumulates** across config scopes (user < project < local) and across `parent_type` inheritance, rather than a higher/child scope replacing the entire lower/parent value. The per-scope patches are combined associatively per-key (preserving and combining `__extend` markers, with a higher bare key winning over a lower marker), so non-overlapping keys from every scope survive and same-key `__extend`s accumulate. The narrowing guard against the provision base is now threaded recursively through the merge, so a bare assign that drops a non-empty aggregate is caught **at any depth** -- including a bare key nested inside an `__extend` value.

New config-merge operator `key__assign`: an explicit "assign without the narrowing warning". It replaces the value from the layer below exactly like a bare `key`, but records no narrowing violation -- the per-key opt-out to the assign-narrowing guard (coexisting with the global `allow_settings_key_assignment_narrowing` flag). Within a single layer, resolution is now two-phase and order-independent: assigns (bare and `__assign`) apply first, then `__extend` stacks on top, so `key__assign` + `key__extend` means "reset-without-warning, then add". A bare `key` and `key__assign` for the same field in the same layer is a contradictory double-assign and raises a clear `ConfigParseError`.

New `Static*` aggregate markers (`StaticTuple`, `StaticList`, `StaticDict`): wrap an aggregate value to mark it **atomic**, so replacing it from a higher layer is a value-set rather than aggregate narrowing and is exempt from the narrowing guard. mngr's existing `StringDerivedTuple` / `ScalarTuple` are now `StaticTuple` subclasses, so the long-standing "a string-form `cli_args` is a whole value, not narrowing" behavior is a reusable concept rather than a special case.

The two settings-patch primitives (`combine_patches` for cross-layer combine and the provision-time narrowing fold) are unified into `merge(lower, higher) -> (patch, narrowings)` plus `finalize(patch) -> dict`. `merge` is the associative four-rule combine *and* the recursive narrowing tracker (honoring `__assign` / `Static*` suppression; never raising for narrowing -- the caller decides what to do with the returned paths); `finalize` resolves any leftover `__extend` against an empty base into a marker-free dict.

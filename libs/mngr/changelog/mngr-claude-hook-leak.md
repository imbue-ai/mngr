The `__extend` config operator now merges dicts **recursively**: a nested `key__extend` inside an `__extend` value merges deeper (extending `base[key]`) instead of replacing it, while a nested bare `key` still replaces at that level (preserving its siblings). Backward-compatible -- any value that nests no inner `__extend` marker merges exactly as the previous one-level operator did.

New `key__assign` config operator: an explicit "assign without the narrowing warning". It replaces the value from the layer below exactly like a bare `key`, but records no narrowing violation -- the per-key opt-out to the assign-narrowing guard (coexisting with the global `allow_settings_key_assignment_narrowing` flag). Within a single layer, resolution is now two-phase and order-independent: assigns (bare and `__assign`) apply first, then `__extend` stacks on top, so `key__assign` + `key__extend` means "reset-without-warning, then add". A bare `key` and `key__assign` for the same field in the same layer is a contradictory double-assign and raises a clear `ConfigParseError`.

New `Static*` aggregate markers (`StaticTuple`, `StaticList`, `StaticDict`): wrap an aggregate value to mark it **atomic**, so replacing it from a higher layer is a value-set rather than aggregate narrowing and is exempt from the narrowing guard. mngr's existing `StringDerivedTuple` / `ScalarTuple` are now `StaticTuple` subclasses, so the long-standing "a string-form `cli_args` is a whole value, not narrowing" behavior is a reusable concept rather than a special case.

A settings *patch* field (e.g. a claude agent type's `settings_overrides`) now **accumulates** across config scopes (user < project < local) and `parent_type` inheritance, rather than a higher/child scope replacing the entire lower/parent value. The per-scope patches are combined associatively per-key (preserving and combining `__extend` markers, with a higher bare key winning over a lower marker), so non-overlapping keys from every scope survive and same-key `__extend`s accumulate. The narrowing guard against the provision base is threaded recursively through the merge, so a bare assign that drops a non-empty aggregate is caught **at any depth** -- including a bare key nested inside an `__extend` value.

The deferred-`__extend` carveout (markers preserved verbatim through config-load for later resolution against a concrete runtime base) now covers a small registry of paths: the existing `create_templates.<name>` options (resolved at `mngr create` time) plus any path under `agent_types.<name>.settings_overrides` (resolved at Claude provisioning time against the home `settings.json`). A deferred `key__assign` under `settings_overrides` is likewise preserved rather than collapsed to a bare assign, carrying its no-warn intent through to the provision-time fold. Scoped to `settings_overrides`; `create_templates` is excluded because its consumer (`apply_create_template`) reads only `__extend`.

Cross-scope `settings_overrides` narrowing is now surfaced at config-load. Because a `settings_overrides` field accumulates across config scopes, the field-level narrowing walker exempts it; previously a higher-precedence scope (project, local, `MNGR__*` env var) whose `settings_overrides` *bare* key dropped a non-empty aggregate set by a lower scope did so silently. The merge now computes those cross-scope narrowings and routes them into the loader's existing flag-gated narrowing aggregation. So a local-scope `settings_overrides.permissions.allow = ["B"]` that drops a project-scope `allow = ["A"]` now raises the standard settings-narrowing error (naming both scopes and how to opt in), escapable as before via `allow_settings_key_assignment_narrowing = true` or by using `allow__extend` / `allow__assign`. A cross-scope `settings_overrides` that only *adds* keys still loads unchanged. (Exposed via the new `MngrConfig.merge_with_narrowings`.)

Fixed a latent bug in the config narrowing detector (`detect_settings_narrowing`): it identified `MngrConfig`'s container dict fields (`agent_types`, `providers`, `plugins`, `commands`, `create_templates`) by **bare field name**, so a same-named field nested inside a sub-model -- e.g. a plugin config's own `commands` dict -- was mis-treated as a top-level container and per-key-recursed, which silently skipped dropped keys instead of flagging the drop. The detector now matches on the **fully-qualified path**, so a nested container-named field is narrowing-checked as a leaf aggregate and cross-scope drops in such fields surface as the standard flag-gated narrowing error rather than vanishing silently.

Fixed a latent crash in `MngrConfig.merge_with`: it called `.merge_with` on `self.retry` / `self.logging` without checking they were set, raising `AttributeError` when the base config's `retry` / `logging` was `None` (as a `model_construct`'d layer can be) and the override set them. It now takes the override outright when the base sub-model is unset. Masked in production because the merge's base operand is always the loader's defaulted accumulator, but a real latent footgun.

Internal (no user-facing behavior change): the generic, model-free part of this merge algebra (the `__extend` / `__assign` operators, the `Static*` markers, and the recursive narrowing detection) is extracted into a new standalone, dependency-free `overlay` library; mngr now imports it from `imbue.overlay`, and `ConfigParseError` subclasses `overlay`'s `OverlayError` so existing `except ConfigParseError` handling still catches raw algebra errors. The whole config-merge surface -- `AgentTypeConfig.merge_with`, `MngrConfig.merge_with`, and `parent_type` inheritance (`_apply_custom_overrides_to_parent_config`) -- now computes through that library's typed-node pipeline (`config/overlay_merge.py`, `merge_models_via_overlay`) rather than field-by-field pydantic copies, guarded by property tests asserting equivalence to the frozen old logic over a diverse corpus. The now-dead `_merge_container_dict` / `_assign_scalar` helpers and the vestigial per-model `merge_with` overrides (`RetryConfig`, `LoggingConfig`, `ProviderInstanceConfig`, `PluginConfig`, `CommandDefaults`, `CreateTemplate`) are removed.

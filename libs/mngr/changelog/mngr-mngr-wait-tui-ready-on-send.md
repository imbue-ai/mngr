Fixed a race where restarting an interactive TUI agent and immediately sending it a message (the web UI's "interrupt and send" / interrupt-then-resend) could fail with "Failed to send message to agent (0 successful agents)". The restart returns as soon as the tmux session is spawned, so the follow-up send used to type into a still-booting TUI; the pasted text never rendered and the send raised `SendMessageError`.

`InteractiveTuiAgent.send_message` now waits for the TUI to be ready to receive input before typing, via a new `wait_until_ready_for_input(timeout)` hook on the agent interface (a no-op by default, so agents without a readiness signal are unaffected). The wait uses the agent's configured ready timeout and raises `SendMessageError` on timeout, matching the error class callers already handle. For an already-running agent the readiness signal is already present, so normal sends -- including to a busy, mid-turn agent -- proceed with no perceptible added latency.

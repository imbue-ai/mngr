Provisioning now resolves plugin enablement from the agent's own work_dir config, not just the directory `mngr create` was invoked from. Plugin enablement is enforced by blocking disabled plugins on the plugin manager, whose blocked-set was previously built once from the invocation cwd; this let a plugin disabled by the agent's project config still fire its provisioning hooks when `mngr create` ran from a context that didn't see that disable (a temp dir, an `MNGR_PROJECT_CONFIG_DIR` override, `--from` another repo, or an in-place create like `mngr robinhood`). The clearest symptom: the `claude_subagent_proxy` plugin's hooks getting written into a worktree whose own `.mngr/settings.toml` disables the proxy, which then broke every subsequent subagent spawn (`Unknown agent type 'mngr-proxy-child'`). `create()` now reads the work_dir's project/local config (via the host interface, so it works for local and remote work_dirs) and blocks whatever the work_dir disables before firing provisioning hooks. Because blocking only adds, this suppresses a plugin the work_dir disables but does not re-enable one already disabled at startup.

Also: the subagent proxy is now disabled by default across the test suite, so its real provisioning side effects (spawning mngr subagents, writing Claude hooks into the agent's work_dir) no longer fire for agents tests provision. In-process tests get this via the shared `plugin_testing.plugin_manager` fixture (the proxy is blocked there); subprocess tests get it via `get_subprocess_test_env`, which writes `[plugins.claude_subagent_proxy] enabled = false` into the subprocess's user profile so the real `mngr` blocks the proxy at startup. The proxy's own tests are unaffected: the in-process ones invoke its hookimpls directly, and the real-subagent subprocess test builds its env directly (not via `get_subprocess_test_env`), so it keeps the proxy enabled without any explicit opt-back-in.

New reusable helper `disabled_plugins_from_raw_layers` in `config/pre_readers.py` (factored out of `read_disabled_plugins`) computes the disabled-plugin set from raw config layers regardless of which directory they came from.

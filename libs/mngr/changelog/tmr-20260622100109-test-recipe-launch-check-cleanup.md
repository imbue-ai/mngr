Fixed the COMMON TASKS recipe e2e tests (`test_common_tasks.py`).

The shared e2e fixture now gates provider discovery on the test's marks: `local` is always enabled, `docker` only when `@pytest.mark.docker` is set, and `modal` only when `@pytest.mark.modal` is set. Previously every provider plugin installed in the dev workspace (aws, azure, gcp, vultr, modal, ...) defaulted to enabled, so an enabled-but-unreachable backend made read paths like `mngr list` exit non-zero -- the aws backend raised "AWS credentials not configured", and the docker backend raised a daemon-socket error in sandboxes without Docker. Gating on marks matches both how offload routes tests to sandboxes and a real PyPI `mngr` install, which ships none of the remote plugins. It is also consistent with the resource guard, which already requires a test to be marked for any remote backend it invokes.

Removed the superfluous `@pytest.mark.rsync` marks from both recipe tests: their local command agents are created via `git worktree` (no rsync), so the mark tripped the resource guard's "marked but never invoked" check. Bumped `test_recipe_multi_agent_parallel_workflow`'s timeout from 120s to 300s to match the single-agent test, since creating and tearing down three agents legitimately needs the extra time.

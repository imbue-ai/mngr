Fix the `test_plugin_disable_affects_create` e2e release test so it disables the `claude` plugin at the user scope (`mngr plugin disable claude --scope user`) instead of the default project scope.

The default project scope wrote a brand-new `.mngr/settings.toml` containing only the disabled flag. Because the e2e fixture deliberately leaves that file unseeded, it lacked the `is_allowed_in_pytest` opt-in, so the subsequent `mngr create` was rejected by the pytest config guard before it could reach the real "plugin disabled" check -- the test failed for the wrong reason. The user scope's config file (the profile `settings.toml`) is already seeded with the opt-in, and `set_plugin_enabled` preserves existing keys, so the disable now exercises the genuine gating behavior. This is a test-only change; no user-facing behavior of `mngr plugin disable` changed.

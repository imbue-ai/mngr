- `mngr git push`, `mngr git pull`, and `mngr rsync` now `os.exec` into the underlying `git` / `rsync` binary after mngr's setup work (URL/env construction, destination `safe.directory` and `receive.denyCurrentBranch` config, default refspec resolution, destination stash check, destination `mkdir`). This makes the wrappers feel native: `git` / `rsync` own stdout/stderr directly, signals (Ctrl-C, SIGPIPE) reach the right process, and the underlying tool's exit code propagates as mngr's. The `_complete` terminating events on `--output-format json|jsonl` (`{"success": true}` for git, `{"event": "rsync_complete", ...}` for rsync) and the trailing "Rsync complete: N files, M bytes transferred" human line are gone -- `rsync --stats` already prints transfer counts, and callers should rely on exit code instead of a synthetic terminator.
- `mngr rsync --uncommitted-changes=merge` is now rejected: the mode pops the destination stash *after* rsync exits, which has no equivalent under the exec model. Use `--uncommitted-changes=stash` and `git stash pop` manually. `stash`, `clobber`, and `fail` are unchanged.
- The existing `imbue.mngr.api.git.git_push` / `git_pull` and `imbue.mngr.api.rsync.rsync` / `rsync_to_remote` / `rsync_from_remote` API functions grow an optional `exec_final_command: bool = False` parameter. Default behavior (subprocess via `ConcurrencyGroup`) is unchanged, so in-process callers like `mngr_pair` and `mngr_mapreduce` are unaffected; only the `mngr` CLI sets it to `True`.

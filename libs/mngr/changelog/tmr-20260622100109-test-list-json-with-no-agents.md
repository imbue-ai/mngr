Fix the `mngr list --format json` tutorial e2e test (`test_list_json_with_no_agents`) so it passes reliably.

The e2e fixture now pins `enabled_backends` to the backends the suite actually exercises (`local`, `ssh`, `modal`, plus `docker` when a daemon socket is reachable). The monorepo dev/test venv installs every provider plugin (aws, azure, gcp, ovh, vultr, ...), but a default `mngr` install ships none of them. Left unrestricted, those credential-requiring cloud backends were probed during every `mngr list`; an unconfigured one (e.g. AWS with no credentials) raised `ProviderUnavailableError`, which under the default `--on-error abort` aborted the whole listing with exit 1. The Docker backend behaves the same way when its daemon is unreachable, so it is only enabled when a daemon socket is actually present (a pure filesystem probe -- calling the Docker SDK from the fixture would trip the `docker_sdk` resource guard). This makes the test environment match a realistic install and keeps `mngr list` clean.

`test_list_json_with_no_agents` also gained an explicit `@pytest.mark.timeout(120)` and a 90s per-command timeout (mngr's CLI startup plus full provider discovery routinely exceeds the default 10s per-test / 30s per-command limits when run on its own), and now records its subprocess Modal usage via `_record_subprocess_modal_usage()` so the `@pytest.mark.modal` resource guard reflects the authenticated Modal lookup that `mngr list` performs in-process inside the `mngr` subprocess (mirroring `test_list_local_filter`).

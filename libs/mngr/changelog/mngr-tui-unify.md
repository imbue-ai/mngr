Fixed a leak in the TUI message-submission path that could make a later `mngr message` time out even though the message was submitted.

Every submission to an interactive agent registers a `tmux wait-for` client on a channel that the agent's submit hook signals, and that channel (`mngr-submit-<session>`) is the same for every submission in a session. A submission that reached its deadline used to leave its waiter registered on that channel, because the waiter ran inside a subshell and the cleanup killed the subshell rather than the `tmux` client inside it. This matters because of how `tmux wait-for` handles a signal: one fired while nobody is waiting is retained and satisfies the *next* waiter, but one fired while a client is waiting is consumed by it and retained for nobody. A leftover waiter therefore silently disarms that retention -- the very thing that covers a hook firing before the next submission has registered its own waiter -- and that submission waits out its full timeout and reports `Timeout waiting for message submission signal`. Measured on macOS 26.4.1 against a real tmux server: one orphaned `tmux wait-for` client per timed-out submission before, zero after.

The waiter is now a direct background job, so the cleanup can reach the `tmux` client itself. That in turn removes the reason the submission command needed an external deadline: it used to bound the waiter with `timeout(1)`, or with a perl `Time::HiRes` alarm on hosts that ship no `timeout` (stock macOS), purely because nothing else could stop the waiter. The deadline is now a plain loop in the script, so a submission needs only bash builtins plus `tmux`, `date` and `sleep` -- no `timeout`, no perl, no temp file.

The two submission-command builders are now one. Waiting on the hook alone is the same command with an acceptance-marker probe that prints an empty token and so can never confirm, leaving the hook as the only thing that can. Agents that watch an acceptance marker (Claude) and agents that do not (codex, antigravity) are unaffected in behavior, except that a hook-confirmed submission for the latter now returns up to one poll interval (0.25s) later than before, against a submission budget of 15-90 seconds.

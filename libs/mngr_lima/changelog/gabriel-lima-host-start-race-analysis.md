Fixed: a concurrent `mngr start` (or any command that auto-starts a host) no longer fails spuriously with a raw connection error while a Lima VM is mid-boot. Lima reports an instance as `Running` from the instant its hostagent spawns -- 15-25s before the guest sshd actually answers -- so a second mngr process could classify the half-booted host "online", skip all start machinery, and immediately fail to connect. Lima classification now also checks the process table for an in-flight `limactl start` for that instance (the start process spawns before the status flips to `Running` and exits only once ssh is ready, so it precisely covers the whole readiness window). A host in that state is surfaced as `STARTING`: `get_host` returns a not-yet-online host and `mngr list`/discovery report `STARTING` instead of `RUNNING`, so callers route through `start_host`, whose own `limactl start` no-ops on the already-booting VM and whose wait-for-sshd absorbs the rest of the boot.

The detection matches mngr's own two `limactl start` invocation shapes exactly (existing-instance and new-instance `--name=`), plus a best-effort bare `limactl start <name>`; instance-name matching is exact-argument, so `mngr-foo` never matches `mngr-foo2`. The load-bearing Lima behavior (the start process outlives the window and exits at ssh-ready) is pinned by a release-tier behavior-guard test that boots a real VM and samples the restart; run the lima release tests whenever the shipped/minimum lima version is bumped.

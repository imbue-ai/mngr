# Pin all imbue_cloud SSH host keys (remove TOFU)

## Overview

- Today every SSH host key in the imbue_cloud stack is already generated by us deterministically at bake time -- container keys (`setup_container_ssh`), VPS/VM-root keys (`mngr_vps/cloud_init.py` and the lima YAML) -- *except* the OVH bare-metal box itself. The TOFU/blind-trust we do is a plumbing gap, not a fundamental unknown.
- Carry the already-known keys to the lease client and connector: persist the outer-root + container host public keys on `pool_hosts`, and the box host public key on `bare_metal_servers`, then pin from the DB instead of scanning.
- Close the one real gap: generate our own ed25519 host key for each OVH box and inject it during install via an inline base64 `customizations.postInstallationScript` on the existing `POST /dedicated/server/{s}/reinstall` (no endpoint migration, no custom image; the secret stays in the authenticated request body).
- Eliminate every TOFU / blind-trust site (lease-client container scan, connector `AutoAddPolicy`, admin/box `StrictHostKeyChecking=no`, lima `accept-new`, slow-path VPS-root scans) and enforce strict pinning with no runtime scan fallback.
- Migrate existing rows once via a sanctioned one-shot `mngr imbue_cloud admin` keyscan-backfill subcommand; during the deploy->backfill window, fail closed (refuse to lease a null-key row) rather than silently scanning.

## Expected behavior

- `mngr create ...@<host>.imbue_cloud_*` leases a pool host and connects to its container by pinning the host key returned in the lease response -- no `ssh-keyscan`, no first-connect trust.
- The lease response (`/hosts/lease`) returns both the outer-root and container host public keys; both are "initial" keys the user may rotate later.
- Fast/adopt path: the user pins the DB-provided initial container key. A mismatch is a hard failure.
- Slow/rebuild path: the user rebuilds the container, generating their own host key locally; the lease client pins that locally-known key. The DB's initial container key is unused on this path and is never written back.
- The slow-path outer (VM/VPS-root) connection used during rebuild pins the outer-root key from the lease response instead of scanning it.
- The connector pins the outer-root and container keys (read from the `pool_hosts` row it already selects) when injecting the user's key at lease time, and pins the box key (from `bare_metal_servers`) on the release/cleanup teardown SSH.
- A freshly OVH-installed box comes up already serving a host key we generated; the very first connection (box prep) pins our known key -- no TOFU even on first contact. The box offers only ed25519.
- Admin/box SSH (prep, bake carve over limactl, teardown) and the lima slice client verify the box against its stored/known key instead of disabling host-key checking.
- Re-running the OVH box provision rotates the box key: it generates+injects a fresh keypair and overwrites the stored `bare_metal_servers` key (the sanctioned recovery path).
- Only public host keys are ever surfaced (`mngr create --format json`) or persisted; generated private host keys are discarded after injection.
- Existing pre-migration pool hosts/boxes are leasable again only after the one-time backfill populates their key columns; until then, leasing a null-key row fails closed with a clear error.

## Changes

### Connector DB schema + service (`apps/remote_service_connector`)

- New migration: add nullable host-key columns to `pool_hosts` (outer-root host public key, container host public key) and to `bare_metal_servers` (box host public key). Public keys only.
- `pool_hosts` lease SELECT additionally reads the two key columns; `LeaseHostResponse` returns both. Lease fails closed (no silent scan) when a selected row's key columns are null.
- `_management_ssh_client` / `_append_authorized_key`: replace `paramiko.AutoAddPolicy()` with strict verification against the keys from the row being leased (outer-root + container) instead of trust-on-first-use.
- Release / cleanup sweep (box teardown SSH, incl. `scripts/cleanup_released_hosts.py` if it connects): verify the box against the stored `bare_metal_servers` key rather than auto-adding.
- Update connector README + the migration runbook note (operator runs the migration, deploys, then runs the backfill once).

### Lease client + provider (`libs/mngr_imbue_cloud`)

- `LeaseResult` / `LeasedHostInfo` (`data_types.py`): add the outer-root and container host public-key fields parsed from the connector response.
- Delete `_scan_and_record_container_host_key` and its `_scan_ssh_host_key` usages on the lease path (`providers/instance.py`).
- Fast path: record the DB-provided container key into the per-host `known_hosts` (pin), then connect.
- Slow path: pin the outer-root key from the lease response for the rebuild's outer connection (`_outer_for_leased_vps`, `_rebuild_leased_container`); pin the rebuilt container's host key from the rebuild provider's own keypair (locally known) -- no scan, no DB write-back.
- Bake/registration (`cli/admin.py`, `slices/bare_metal_db.py`, `bake/pool_bake.py`): persist the outer-root + container host public keys into the new `pool_hosts` columns at bake time, sourced from the bake's `mngr create --format json` output.
- OVH box provisioning (`slices/ordering.py`, `cli/admin.py`, `mngr_ovh` client): generate an ed25519 box host keypair; pass an inline base64 `customizations.postInstallationScript` (alongside the existing `sshKey`) that writes the private key to `/etc/ssh/`, removes OVH's other host key types, fixes perms, and restarts sshd; persist the box public key into `bare_metal_servers`; discard the private half.
- Box-reaching SSH: `cli/admin.py::_run_ssh_command` stops using `StrictHostKeyChecking=no` + `UserKnownHostsFile=/dev/null` and instead pins the stored box key; `slices/lima_slice_client.py` replaces `StrictHostKeyChecking=accept-new` with strict verification against the stored box key.
- New one-shot `mngr imbue_cloud admin` backfill subcommand: keyscans each existing `pool_hosts` row (outer + container ports) and each `bare_metal_servers` box, writing the new columns. This is the single sanctioned (migration-only) TOFU; documented as run-once after deploy.

### Bake surfacing (`libs/mngr_vps`, slice provider)

- Ensure `mngr create --format json` surfaces the outer + container host *public* keys (the slice/vps providers already hold them via `_get_vps_host_keypair` / `_get_container_host_keypair` and `RealizedPlacement.container_ssh_host_public_key`); extend the create output/parse as needed so the bake can persist them. Public keys only.
- No change to mngr_vps's own connection behavior (it already pins, no scan) beyond surfacing the keys.

### Tests + docs

- Unit/integration: pure helpers (post-install script rendering, key-column insert value builders, lease-response parsing), DB plumbing (columns present, lease returns keys, null-key lease fails closed), and the known_hosts/pin logic (fast pins DB key, slow pins rebuilt key, mismatch fails closed). No live network in unit/integration.
- Acceptance/release left as follow-up where infra is required; manually verify most paths during implementation, being especially careful about actually running the live OVH reinstall code.
- Update `libs/mngr_imbue_cloud/README.md`, connector README, and changelog entries for each touched project (`libs/mngr_imbue_cloud`, `libs/mngr_vps`, `apps/remote_service_connector`, and `dev/` if root files change).

### Out of scope

- Vendoring the changes into `forever-claude-template` and the connector deploy itself (handled separately).
- User-driven rotation of the initial keys post-lease (the design returns both keys precisely so this is possible later, but the rotation flow is not built now).

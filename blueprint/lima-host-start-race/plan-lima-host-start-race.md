# Fix the lima host-start readiness race via boot-in-flight detection

## Overview

* Incident (2026-07-13, minds staging): a lima host stopped out-of-band was booted by an auto-starting `mngr exec`; ~10s into the ~20-30s boot, `mngr start --quiet` classified the host online (lima reports `Running` from the instant the hostagent spawns), skipped all start machinery, and failed spuriously with a raw connection-refused — five seconds before the boot finished.
* Root cause: lima's `_get_host_by_id` (`libs/mngr_lima/imbue/mngr_lima/instance.py:946`) classifies online purely from the `limactl list` status string, and `ensure_host_started` (`libs/mngr/imbue/mngr/api/find.py:326`) reaches `start_host` — the only place `wait_for_sshd` runs — only when the provider classified the host offline. Only a concurrent second mngr process can observe the window; serial usage never sees it because `start_host` does not return until sshd answers.
* Fix: lima classification additionally checks the process table for an in-flight `limactl start` for that instance. The `limactl start` process demonstrably outlives the whole window (it spawns before status flips to `Running` and exits only when ssh is ready), so "status `Running` + start process alive" precisely identifies a booting host. Threat model is mngr's own invocations; out-of-band lima usage is best-effort only.
* A booting host is surfaced as `HostState.STARTING` consistently (routing, listings, and the offline host object's state). The enum value exists and imbue_cloud already emits it in production listings, so consumers tolerate it.
* The load-bearing lima behavior (start-process lifetime covers the window; start exit implies sshd ready) is pinned by a release-tier behavior-guard test following the `test_lima_btrfs_release.py` pattern, run whenever the lima version is bumped.

## Expected behavior

* `mngr start` against a mid-boot lima host succeeds: classification sees the in-flight `limactl start`, returns a not-online host, `ensure_host_started` routes through `start_host`, whose `limactl start` no-ops ("already running", exit 0) and whose `wait_for_sshd` absorbs the remaining boot (bounded by the existing `ssh_connect_timeout`, default 120s; no new knob).
* Every other start-intending flow (`mngr exec`, `connect`, `create --reuse`/existing-target/`--from`, `rename --start`, `message`, `rsync`/`git`) gets the same behavior for free — they all route through `ensure_host_started`.
* Detection is race-free for the incident shape: `limactl list` is read first, then the process table. "`Running` and no start process" implies the boot already finished (sshd ready), so classifying online is safe. A comment must pin this ordering.
* Detection matches mngr's own two invocation shapes exactly (`limactl --log-level=info start <name>` from `limactl_start_existing`, `limactl --log-level=info start --name=<name> <yaml>` from `limactl_start_new`), with best-effort matching of a bare out-of-band `limactl start <name>`. Instance-name matching is exact-argument, not substring (`mngr-foo` must not match `mngr-foo2`). Users independently driving lima while mngr runs are otherwise on their own.
* A booting host reports `HostState.STARTING` everywhere: `mngr list`/discovery shows STARTING instead of RUNNING during the window, and the offline-shaped host object returned by `get_host` also reports STARTING (not the CRASHED that a stop-reason-less offline host derives today).
* A fully-up `Running` host with no start in flight classifies online exactly as today; the only added cost is one local process-table scan (no SSH probe on any hot path).
* A `Running` host whose guest sshd is dead with no boot in flight is unchanged: classified online, callers get today's connection error. Likewise the killed-`limactl` edge (hostagent survives, start process gone) is not covered — both are accepted residuals under the narrowed threat model.
* Non-start flows during the boot window get offline-flavored behavior, accepted as strictly better than today's raw connection errors: `mngr stop AGENT` raises `HostOfflineError`, `mngr destroy` takes the offline whole-host path, gc work-dirs skips the host.
* `mngr gc` (any provider) no longer aborts its work-dirs sweep when an online-classified host is unreachable: the catch in `_gc_single_host_work_dir` (`libs/mngr/imbue/mngr/api/gc.py:237-240`) widens from `HostOfflineError`/`HostAuthenticationError` to the base `HostConnectionError`, matching the machines phase's deliberate skip (`gc.py:405-408`).
* No minds code changes. One verification item: observe that minds' local-workspace liveness/recovery treats a STARTING lima host benignly during a normal boot (imbue_cloud's production STARTING emissions are precedent; this confirms the local flow specifically).
* The release-tier guard test fails if a lima upgrade changes either load-bearing behavior: (a) during a stopped-to-ready boot, every sample where status is `Running` and sshd is unreachable also shows the start process alive (the invariant the fix rests on), and at least one such sample exists (the window is real and covered); (b) when `limactl start` exits 0, sshd answers immediately. Cadence rule: run the lima release tests whenever `minimum_lima_version` (or the shipped lima) is bumped.
* Residual accepted: two `limactl start`s colliding in the sub-second window before the hostagent registers can still produce a spurious nonzero "seems running" failure. Follow-ups if issues persist: verify-after-error hardening in lima's `start_host` (on `LimaCommandError`, re-read status; if `Running`, proceed to `wait_for_sshd`), and/or the `ensure_host_started` seam readiness gate from the superseded analysis. Also deferred: `mngr destroy`'s escalation of connection errors into whole-host teardown (`destroy.py:435-445`, needs a product decision).

## Changes

* `libs/mngr_lima`: add boot-in-flight detection — a psutil-based scan (psutil is already a `libs/mngr` dependency) for a live `limactl start` process targeting the instance, invoked after the `limactl list` read; `_get_host_by_id` returns a STARTING offline-shaped host when detected; `discover_hosts` maps the same condition to `HostState.STARTING`; the offline representation's `get_state()` reports STARTING.
* `libs/mngr_lima`: unit tests for the detection and classification via an injected process-listing seam (dependency injection, no monkeypatch), covering: exact-name matching (no prefix collisions), both mngr argv shapes plus bare `limactl start`, list-then-scan ordering, and the STARTING state surfacing in `get_host`/`discover_hosts`.
* `libs/mngr_lima`: the release-tier behavior-guard test (`@pytest.mark.release + @pytest.mark.lima`, `test_lima_btrfs_release.py` pattern): stop a real VM, spawn `limactl start`, sample (process alive, `limactl list` status, sshd handshake) every ~250-500ms until exit, assert the safety invariant, window coverage, and ready-on-exit. Document the two distinct failure meanings and the "run on lima bump" cadence rule.
* `libs/mngr`: widen the `_gc_single_host_work_dir` catch to the base `HostConnectionError` (skip-and-continue, mirroring the machines phase), with a unit test.
* `libs/mngr`: correct the stale comment at `cli/start.py:235-237` to describe the actual routing (online short-circuit; offline routes through `start_host`; booting lima hosts now classify as not-online).
* Delete `specs/host-start-readiness-race.md` — its incident/mechanism analysis and accepted follow-ups are absorbed above; the full provider audit and entry-point survey remain in git history. `specs/uncertainties.md` is left untouched. Update the existing `dev/changelog/gabriel-lima-host-start-race-analysis.md` entry on this branch to describe the blueprint.

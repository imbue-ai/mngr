# Robust message delivery: durable-evidence confirmation for `mngr message`

## Overview

- A production incident showed `mngr message` reporting success for a message that was never submitted: a stale tmux `wait-for` latch (tmux latches signals fired with no waiter, contrary to the assumption in `tui_utils.py`) confirmed the send instantly, and the script's EXIT trap then killed the backgrounded Enter keystroke before it fired. The message sat in the input box while every layer reported success.
- Replace signal-based confirmation with durable-evidence polling: `send_message` exits 0 only when on-disk evidence proves the agent accepted the message. Claude gets content-verified confirmation (the message text found in enqueue/user-message records past a pre-Enter baseline, in the native session jsonl OR the mngr raw-transcript copy); codex and antigravity confirm via their existing `active` marker advancing past baseline (same timing as today's hook, no hook changes).
- The three send-Enter strategies in `tui_utils.py` are replaced by one unified submit-and-confirm engine with two policies (strict: hard-fail on no evidence; relaxed: exit 0 + warning, used for slash commands). The confirmation window runs as ONE sequential remote script — no background jobs, no EXIT trap — with bounded, pane-gated Enter retries and never a re-paste, so mngr's own retries can never duplicate a message.
- Backward compatible in both directions: confirmation relies only on evidence that already exists on agents created before this change (enqueue log / transcripts / `active` marker), and newly created agents keep firing the now-unlistened `mngr-submit` hooks so old-version senders still confirm (hook removal explicitly noted for a future release). The bare `command` agent keeps its current fire-and-forget behavior; opencode/pi-coding/headless paths are untouched.
- Testing centers on consolidated per-agent-type release-test journeys (idle → busy/queued → rapid-sequential → slash → long-message in one or two agent sessions, which also helps with rate limits) plus targeted concurrency and old-hook-config compat tests, layered over unit and tmux-integration tests of the engine.

## Expected behavior

- `mngr message <agent>` (and the in-process API used by minds) returns success only after durable evidence confirms acceptance:
  - claude: a normalized tail-probe of the message content appears in a `queue-operation`/`enqueue` record or user-message record appended after the pre-Enter baseline offset, in either the native session jsonl or `logs/claude_transcript/events.jsonl` — whichever shows it first. Baselines are byte offsets, so repeated identical messages ("yes" twice in a row) confirm correctly.
  - codex / antigravity: the `active` marker appears or advances past its pre-Enter baseline (identical timing semantics to today's hook — set by `set_active_marker.sh` / `statusline.sh`, which old agents already have).
  - messages whose normalized content probe is empty (emoji, punctuation-only) fall back to any-new-evidence confirmation: any enqueue/user-message record appended after baseline.
- While evidence is absent, Enter is re-sent at roughly 3s/10s/30s into the window — but only when a pane capture still shows the pasted text in the input box (a swallowed Enter). Enter is the only thing ever retried; the message text is never re-pasted, so duplicates cannot originate from mngr.
- If the 90s confirmation window (~0.5s poll interval) expires without evidence, `send_message` raises with rich diagnostics: pane capture, each probe's baseline and final value, and the Enter-retry history. `mngr message` exits non-zero.
- Messages starting with `/` (uniformly across all TUI agents) use the relaxed policy: same paste, Enter, pane-gated retries, and a brief evidence poll (any evidence counts, including session-id change from `/clear`), but always exit 0 — logging a warning and appending a structured agent event when nothing was observed.
- A message to a busy claude agent confirms near-instantly (Claude Code records the enqueue the moment it queues the input). A message to a busy codex/antigravity agent confirms when the prompt is dequeued (unchanged from today) and may legitimately use the whole window.
- `mngr create --message ...` and `mngr start` (resume message) propagate confirmation failure: the command exits non-zero with a clear "the agent is up, but the message was not delivered — resend with `mngr message`" error.
- Pre-existing text found in the input box keeps today's append behavior, now with a warning log and a structured agent event so the condition is auditable (`mngr event` / minds).
- The bare `command` agent behaves exactly as before: literal keys + Enter, no confirmation, no new failure modes. Headless agents still refuse messages; opencode/pi-coding keep their own send paths.
- Existing agents (created before this change) are messaged successfully by the new sender with no reprovisioning; their frozen hooks firing `tmux wait-for -S` into an unlistened channel are harmless. Old-version senders messaging newly created agents also keep working, because new agents still fire the hooks.
- No new CLI flags; timing is defaults-only. The tmux `wait-for` channel no longer participates in confirmation anywhere, eliminating the stale-latch false-success class entirely: no false successes, and false failures bounded by retry + dual evidence sources (a dead transcript watcher alone cannot cause one for claude).

## Changes

- `libs/mngr` — `tui_utils.py`: delete `send_enter_best_effort`, `send_enter_and_poll_for_cleared_indicator`, `send_enter_via_tmux_wait_for_hook`, and both composite-script builders (background waiter, sentinel file, EXIT trap all go away). Replace with a unified submit-and-confirm engine: capture baselines → send Enter → run one sequential remote confirmation script (evidence polling + pane-gated Enter retries per schedule) → parse a structured result; strict vs relaxed policy; rich-diagnostics error construction. Fix the incorrect "signals with no waiter are lost" commentary wherever it appears.
- `libs/mngr` — `tui_agent.py` / internal agent contract: replace `_send_enter_and_validate` with an internal (underscore-private) per-agent evidence-probe contract that supplies shell probe snippets and a content-match spec; rename `enter_submission_timeout_seconds` to match the new confirmation-window semantics. Detect pre-existing input-box text before pasting (warning + structured event, then proceed). Out-of-tree plugins that don't implement probes fall back to base `SendKeysAgent` behavior.
- `libs/mngr` — structured agent events for the soft states (relaxed send unconfirmed; pre-existing-text append), written to the agent's events dir.
- `libs/mngr_claude`: content-evidence probes reading both the native session jsonl (resolved via the existing session-id / config-dir helpers used by session adoption) and the raw copy; enqueue + user-message record matching with the existing normalization approach; keep all hooks firing (including the SessionStart clear/compact signal) with an explicit note scheduling their removal in a future release.
- `libs/mngr_codex`, `libs/mngr_antigravity`: `active`-marker baseline/advance probes; keep the hook/statusline signal firing with the same future-removal note.
- Callers: `api/message.py` semantics unchanged (still exactly one `send_message` call — retries live inside); `cli/create.py` / `api/create.py` / `cli/start.py` gain the "agent is up, message undelivered" error framing.
- Tests:
  - unit: engine ordering (Enter always precedes any confirmation check), strict/relaxed policies, probe parsing, low-content fallback, diagnostics content — via the existing `_ProbeAgent`/`_RecorderHost` stub patterns.
  - tmux-marked integration: real-tmux delivery through the engine; `command`-agent unchanged-behavior test (runs in CI, needs no real agent binary).
  - release: one consolidated message-delivery journey per TUI agent type (idle → busy/queued → rapid sequential → slash → long message) via the `AgentReleaseProfile` harness, plus targeted tests for concurrent sends to two agents on one tmux server and for an agent provisioned with the pre-change hook config (old-agent compat, claude at minimum).
- Docs and changelog: update `docs/commands/secondary/message.md` delivery-guarantee description; changelog entries for every touched project (`libs/mngr`, `libs/mngr_claude`, `libs/mngr_codex`, `libs/mngr_antigravity`).
- Out of scope (explicitly): remote-host message locking (stays local-only, fixed later), opencode/pi-coding confirmation tightening, minds vendored-mngr bump (normal minds release process), any new CLI flags.

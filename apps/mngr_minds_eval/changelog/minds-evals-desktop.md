- New `minds-evals` CLI (`apps/mngr_minds_eval`): a harness for running persona-based evals against Minds. `launch --config <json>` creates one self-completing Modal workspace per persona case: each sandbox drives its own multi-turn conversation (literal prompts, or `DECIDE_FROM_PERSONA` role-played via the Anthropic API), restic-snapshots `/mngr` to S3 per turn, uploads its transcript, and enforces a per-case wall-clock budget (default 1h; `test_state: timed_out`, distinct from `ongoing`). Results are read back from S3 (`list-batches` / `inspect`), and `evaluate <batch>` scores finished cases (avg words per agent turn + three LLM-graded 1-10 scores), writing per-case and batch results back to S3.

- Every batch gets its OWN Modal env (`minds-staging-<batch-slug>`, via the Modal provider's `user_id`), recorded in the batch's S3 config together with the exact mngr SHA it ran on. A box therefore only ever discovers its own batch's workspaces (small, fast discovery -- no OOM from enumerating one giant shared env), and batches are isolated from each other.

- The box is a full Minds computer in Docker, pinned to an exact mngr SHA, with two run modes from one image: headless (`minds run` serving the Minds HTTP API -- used by `launch` to create workspaces, then torn down) and desktop (the real Minds Electron app on Xvfb, streamed to the browser via x11vnc + noVNC -- one published port, no host-side tunnels).

- `visit-batch <batch>` rebuilds the exact computer a batch ran on: it reads the recorded `(mngr_branch, mngr_sha, modal env)` from S3, boots a desktop box that IS that machine, and prints a noVNC URL -- you enter a real desktop running the Minds app whose discovery sees exactly that batch's workspaces, and open them as native windows. The Minds app reaches its workspaces itself (mngr forward on the container's own loopback); nothing is tunnelled through the host.

- The `evaluator` mngr profile (and its Modal SSH keypair, persisted at `~/.minds-eval/modal-profile/`) is shared across all boxes, so a visit box can open workspaces a launch box created.

- Eval names are now globally unique and ARE the batch identity: the S3 prefix and the Modal env (`minds-staging-<name>`) key on the name alone (no launch timestamp). `launch` preflights both -- it fails out if the name already exists as an S3 batch or as a Modal env -- and `load_config` requires the name to be a valid Modal user id (lowercase alnum + dashes, max 40 chars) so no sanitization can alias two names. The name is given on the `launch` command line (`minds-evals launch <name> --config <template>`), making the config a reusable template -- same shape as inspect/evaluate/visit-batch, which already take the name positionally. Added `eval-config-small.json` (3 cases) for quick end-to-end runs.

- One box mode: every box is the desktop computer (headless mode removed). `launch` creates the batch's workspaces inside the batch's own desktop box -- the in-box CLI discovers the Minds app's API port from within the container, so nothing is pinned or published except the single noVNC port -- and leaves the box running with its desktop URL printed, so you can watch the workspaces appear and run. `visit-batch` reuses that same box by name while it is up. Dropped: MINDS_BARE_PORT, the headless entrypoint branch, launch's auto-teardown, and the per-mode container naming.

- The box now runs as a MODAL SANDBOX -- local Docker is gone entirely. The image is built from `docker/Dockerfile` on Modal's builders (cached per mngr SHA); the desktop streams through Modal's encrypted tunnel as one `https://...modal.host` URL usable from any machine (deliberately no auth: the URL's randomness is the only lock); the shared mngr-profile SSH keypair lives on a persistent modal.Volume mounted into every box; the Modal token + AWS creds are passed as sandbox env vars. Boxes are found by tag (batch name + mngr ref) for reuse, get 16GB/6cpu, self-terminate after 8h, and a new `minds-evals stop <name>` kills one early. The `box` dev utility's default user-id is now `minh`. Your machine's job is reduced to API calls: no Docker daemon, no local builds, no port mappings, no VM sizing.

- `box` gained `--dwt-link` / `--dwt-branch` / `--workspace-name`: boot the desktop box AND create one workspace in it from the given template repo/branch (api_key provider; requires ANTHROPIC_API_KEY, which is piped into the workspace). The create runs inside the box via the same in-sandbox API discovery the launch flow uses.

- `box --dwt-link` gained `--vendor-box-mngr`: overwrite the workspace's committed `vendor/mngr` subtree (which is release-pinned in the template repo) with the BOX's mngr before creating -- so `--mngr-branch` governs both the backend that provisions the workspace AND the mngr the workspace carries internally. Same mechanism launch has always used for eval cases.

- Results storage is Cloudflare R2 (S3-compatible) instead of AWS S3. Setup collapses from four AWS steps (admin keys, `aws s3 mb`, IAM user, scoped policy) to a single `./setup-r2.sh [name]`, which runs `mngr imbue_cloud bucket create` (bucket + scoped readwrite key) and writes `~/.minds-eval/r2.env` from its output -- no AWS account, no IAM, no copy-paste. Credentials live at `~/.minds-eval/r2.env` (renamed from `aws.env`) and add `MINDS_EVAL_S3_ENDPOINT` (the account's R2 endpoint); region defaults to `auto`. boto3 and restic reach R2 via that endpoint under the standard `AWS_*` credential names.

- Eval Modal sandboxes now use a dedicated `modal_eval` template overlay (3h sandbox timeout) instead of pinning the shared `modal` template's timeout. The eval box sets `MINDS_MODAL_EXTRA_TEMPLATE=modal_eval`; minds' create stacks that overlay on `modal` (like `docker_runsc` on `docker`). The role-play worker now reads `ANTHROPIC_API_KEY` from the workspace env (forwarded by the template) rather than having the key written into each case's `test_case_metadata.json`.

# Changelog - minds

A concise, human-friendly summary of changes for the `minds` app. Entries are categorized using the [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) categories: Added, Changed, Deprecated, Removed, Fixed, Security.

For the full, unedited changelog entries, see [UNABRIDGED_CHANGELOG.md](UNABRIDGED_CHANGELOG.md).

## [Unreleased]

### Added

- Added: User-controlled error reporting via a "Help improve Minds" consent screen (shown on first launch and after upgrading) and a Settings page reached from the sidebar. Two toggles ("Report unexpected errors" and "Include logs") take effect live, gating what Sentry sends across the backend, the browser web UI, and the Electron main process. The `MINDS_SENTRY_ENABLED` and `MINDS_SENTRY_S3_UPLOADS` env vars are retired.
- Added: "Get help" button (question mark) in the top bar opens a modal for reporting a bug to Imbue. Reports include optional app-diagnostics / workspace-details / remote-access checkboxes (remembered across reports), attach logs when the include-logs setting is on, and show a copyable Sentry reference ID on success. Workspace-recovery and full-app error screens carry report buttons too; a backend-down report still files from the app shell with host info and (opt-in) logs.
- Added: In-workspace agent bug reporting via a new authenticated `POST /api/v1/agents/<id>/report` endpoint.
- Added: Welcome / sign-in screen shown whenever the desktop app is functionally empty (signed out of every account, no workspaces) instead of silently restoring a stale window from a prior session.
- Added: Frontend (browser) and Electron main-process JavaScript error reporting to Sentry, sharing one set of JavaScript Sentry projects (production / staging / dev), kept separate from the backend's Python projects.
- Added: Optional Name field at the top of the create form's advanced configuration, so you can name the mind explicitly instead of always getting an auto-generated `mind-N`.
- Added: Bumped bundled Latchkey to 2.19.1.
- Added: Discovery-health watchdog — minds now watches its own workspace-discovery pipeline and tries to recover it automatically. The watchdog detects a stall (via snapshot freshness) and self-heals with a cheap producer nudge that leaves your open workspace untouched, then, if that doesn't help, a heavier restart. If the pipeline can't be recovered (healing attempts exhausted, the discovery consumer died, or a fresh launch never managed a first update), minds takes over the whole app with a clear "disconnected from your workspaces and can't automatically reconnect" screen and a "Restart Minds" button instead of leaving you stuck on an indefinite "Loading workspace" page. The old "Forwarding subprocess died — restart minds" notification is removed in favor of this screen.
- Added: Backend Sentry error reporting — `minds run` initializes Sentry during startup (Flask integration carries request context: transaction name, URL, method, query string, headers), reports Python backend errors with attached log files and traceback-with-locals uploaded to S3. Selects DSN and S3 bucket from the activated minds env (production / staging / dev with no S3 uploads). Gated behind `MINDS_SENTRY_ENABLED` (default off); S3 attachment uploads behind `MINDS_SENTRY_S3_UPLOADS` (default off, even in production / staging). Performance tracing is disabled (`traces_sample_rate=0.0`). Reports the desktop app version as the Sentry release and the git SHA as the `git_sha` tag.
- Added: Hardened Sentry error reporting against failures inside Sentry event processing (the `before_send` hook, Sentry callbacks, and the HTTP transport) — failures now always land in the local app log and report via a minimal Sentry event. Made `log_error_inside_sentry` non-reentrant so a failure inside `before_send` can no longer recurse until the stack is exhausted.
- Added: `minds pool backfill-host-keys` — env-aware wrapper around `mngr imbue_cloud admin pool backfill-host-keys`. Run it once per tier after deploying the host-key-pinning connector to keyscan + record SSH host keys for pool rows and bare-metal boxes baked before the host-key columns existed, restoring leasing / prep for those hosts.
- Added: Compiled Tailwind v4 build step replaces the runtime Tailwind Play CDN — a single minified, tree-shaken `app.min.css` built ahead of time from `static/app.css` (`just minds-css`, replaces `just minds-tailwind`). Packaged builds compile it before packaging via `scripts/build.js` and `pnpm start` compiles it via its `prestart` hook. Fully offline and smaller. `just minds-start` runs the compiler in `--watch` mode alongside Electron, so class changes rebuild live.
- Added: Design-token system with light / dark mode — text, border, surface / fill, status / feedback, and accent / interactive token families, plus a 16x16 icon set (replacing lucide), six-role semantic type ramp, constrained spacing scale, four-step corner-radius scale, and a two-step elevation scale. Switching the whole app between modes is a single `.dark` class on `<html>`. A new dev styleguide at `/_dev/styleguide` (visible in every tier) catalogues the tokens and patterns, with a floating light/dark toggle, sticky TOC, and scrollspy.
- Added: Light/dark-aware titlebar that self-themes from the workspace color in pure CSS — derives a black/white contrast from `--titlebar-bg` via relative color (`lch(from ...)`) on a `.titlebar-surface` scope, so the title and buttons read correctly on any workspace color with no JavaScript luminance and no server-side foreground calculation.
- Added: Notification badge component (`templates/Badge.jinja`) — bare 8px `important` dot, or a count pill that grows for wider numbers and caps at "99+". The titlebar's pending-request count now sits in an inline pill beside the messages icon instead of a corner dot overlapping the icon.
- Added: `apps/minds/docs/vendor-mngr-sync.md` as the single source of truth for how FCT's `vendor/mngr` is synced (the two mechanisms: `git archive` for reproducible committed release snapshots, `rsync` for working-tree dev iteration and pool bakes).
- Added: Released minds v0.3.2 (2026-06-22) and v0.3.3 (2026-06-24). Both pin `FALLBACK_BRANCH` at the matching `minds-v<version>` forever-claude-template tag, so a shipped binary clones the FCT snapshot it was verified against. v0.3.3 also carries the Tailwind v4 ToDesktop build fix.
- Added: `minds pool create --backend slice` bakes pool hosts onto bare-metal "slices" (lima VMs carved on a pre-registered, prepped bare-metal box) instead of OVH VPSes. Default `--backend ovh_vps` is unchanged. The slice path resolves the tier's `pool-ssh` private key from Vault and injects it as `POOL_SSH_PRIVATE_KEY` (mirroring the OVH path's secret injection); accepts `--dry-run` and `--max-concurrency`; requires `--server-id` (the explicitly-chosen bare-metal box); and rejects OVH-only flags with a clear error.
- Added: `minds paid {add,remove,list}` command group (plus a `just add-paid-email <email>` one-liner) for managing the connector's paid-user allowlist from an activated env. Like `minds pool`, it resolves the connector URL and the paid-list admin key from `client.toml` and Vault automatically.
- Added: `MngrCaller` keeps a single pre-warmed, single-use `mngr` process running ahead of time and hands each call's argv over an anonymous, connected `socketpair`, so in-app `mngr` invocations (e.g. the `mngr message` after a permission decision) skip the multi-second interpreter-and-plugin import cost. `MngrMessageSender` routes through it by default.
- Added: imbue_cloud workspace creation now passes the form's repository through to the lease as `-b repo_url=<repository>` (the imbue_cloud provider canonicalizes it via the new `repo_identity.canonicalize_repo_source`), so the fast path can only adopt a pre-baked host that genuinely matches the requested repo. Previously the repo was dropped and only an operator-chosen branch label was matched, so a request for one repo could silently adopt a host running another.
- Added: `--skip-deferred-install-wait` passthrough on `minds pool create` (forwarded to the OVH admin pool bake) for faster dev pool bakes.
- Added: 12-color palette workspace picker (replacing the SHA-derived per-workspace accent) with custom-hex input, exposed in the Create form and Workspace settings; the chosen color is persisted as an mngr `color=<hex>` label and propagates live via SSE. Titlebar text / nav icons / account button use a WCAG-luminance contrast picker server-side, so legibility holds across the full hex range.
- Added: Floating sidebar menu (rounded dark panel, shadow, colored dot per workspace) opens from the titlebar hamburger button and replaces the docked sidebar. Each row hovers in a settings gear and an "Open in new window" arrow; two rows at the bottom go to Create and Manage account(s); the menu dismisses on outside-click / Escape, sharing the inbox's modal overlay so the workspace stays in place.
- Added: Landing-page workspace rows gain the same "open in new window" arrow next to the settings gear.
- Added: File-sharing permission dialog accepts `~` / `~/...` notation for the current user's home directory when editing the path to share; `~user` for another user is rejected with a clear error.
- Added: Tiered system-interface restart for the workspace recovery flow — the "Workspace unresponsive" page auto-recovers without clicks where it safely can and otherwise offers a "Restart workspace" button, with troubleshooting (error details, copy-pasteable diagnostics, copy SSH command) collapsed below and a dedicated "Workspace misconfigured" tier. A failed restart polls in the background and returns the user to the workspace once it recovers. Adds a sidebar "Restart workspace…" context menu entry and a per-workspace restart button on the home page.
- Added: Workspace web content can now open a permission-request modal from inside the workspace by posting a `minds:open-request-modal` message (with a request id) to `window.parent`. In the desktop app the content view gains a minimal, allowlist-only relay preload (no `window.minds` bridge); in browser mode the shell navigates the content iframe to the request page instead.
- Added: Per-tier `[scaledown_window]` deploy.toml block (connector / litellm_proxy seconds) threaded into each `modal deploy` so containers stay hot for a configurable idle window — dev defaults to 600s (10 min); staging/production and ci/test tiers omit it (Modal default).
- Added: Per-tier `[paid]` deploy.toml block (`domains` / `emails`) seeded by `minds env deploy` into the connector's `paid_domains` / `paid_emails` tables — all four tiers default `domains = ["imbue.com"]`. Seeding uses `INSERT ... ON CONFLICT DO NOTHING` so a redeploy never re-activates a soft-removed entry.

The Create Workspace form is now name-first with a `Configure...` disclosure for compute / AI / backup providers and a nested `Show advanced settings` for repository / branch / `GH_TOKEN`. After clicking Create the workspace is built in the background while the user answers three optional onboarding questions (about getting to know them, what to start with, and how to handle permissions), whose responses feed a local user scan, the workspace's first chat message, and the workspace's permissions preferences.

The user-facing creation log states which path was taken.

The underlying clone path switched from `git clone --single-branch --branch <ref>` to `git init && git fetch origin <ref> && git checkout -B <name> FETCH_HEAD`, which is uniform across all three input shapes (non-shallow, mirror-pushable). Selecting AWS runs the agent in a runsc-hardened Docker container on EC2; the workspace listing now labels every row with its compute provider, and AWS hosts are long-lived (no idle shutdown, no max-lifetime cap).

### Changed

- Changed: Switched all minds Vault reads/writes to the **"split" secret layout**. Each service entry is now a Vault directory whose children are single-field leaf secrets (`secrets/minds/<tier>/<service>/<KEY>` holds `{"value": ...}`), instead of one flat KV entry with many fields. Deploy, host-pool, paid-admin, generation-id, and deployment-test code paths read the new locations only; there is no fallback to the old flat layout. Docs updated accordingly.
- Changed: Replaced the minds desktop client's FastAPI/asyncio web stack with a synchronous Flask app served by a graceful cheroot WSGI server. Internal framework swap with no user-visible behavior change: every route, status code, header, redirect, and SSE stream behaves as before. Removed `fastapi` / `uvicorn` / `a2wsgi` / `python-multipart` / `websockets` deps; added `flask` / `cheroot` / direct `werkzeug`. SSE endpoints are now plain synchronous generators (a client disconnect is noticed on the next write rather than proactively — WSGI exposes no disconnect signal). Raised the Electron shell's initial chrome-state fetch timeout from 4s to 10s to absorb the cold first-call latency.
- Changed: Multiple developer environments can now safely share a single bare-metal slice box. `minds pool create --backend slice` stamps the activated env into each slice's lima names (`mngr-slice-<env>-<host-hex>`). `minds env destroy` tears down the env's unleased pool slices on their bare-metal boxes before deleting the per-env database. Slice baking derives free-slot capacity from the box's real occupancy across all envs; concurrent bakes from different envs never collide on capacity or ports.
- Changed: Workspace recovery now understands remote (Imbue Cloud) minds, not just local docker/lima ones. When the workspace's backend can't be reached (the provider's connector is down, the local Docker daemon is stopped or paused, or your login has expired) minds shows a single "Can't connect to ..." page in place of the two nearly identical ones it used to pick between based on the kind of error. The page surfaces the provider's own error verbatim, offers a Retry, and returns you to the workspace automatically once the backend recovers.
- Changed: Workspace recovery now reads the host's lifecycle state and its provider's reachability from the same passive discovery data the rest of minds already uses, instead of firing its own separate `mngr list` each time the recovery page loads.
- Changed: Minds now waits until discovery has actually re-observed the workspace *after* a connection drops before taking you to the recovery page, so a stopped container is recognized as stopped and its restart is dispatched automatically with nothing to confirm.
- Changed: Bumped bundled Latchkey to 2.17.2 (was 2.17.1 earlier in this window).
- Changed: Reworked workspace accent colors. Non-workspace screens (Home, Create, accounts, inbox, auth) and the startup/quitting/error screens now paint a pure-white neutral background instead of the previous light-gray/dark chrome (light mode only for now). The titlebar adopts a workspace's accent only on workspace-scoped screens (the workspace itself plus its settings, sharing, destroying, and recovery screens), instead of keeping the last-opened workspace's accent after navigating away. Pure black and pure white were removed from the swatch picker (still typable in the hex input), and cancelling the Create form now clears the previewed color from the titlebar.
- Changed: `minds pool destroy` (and the `just destroy-pool-host` recipe that wraps it) now fully tears down bare-metal `slice` pool hosts, not just OVH VPSes. Previously a slice teardown either failed trying to cancel a non-existent OVH VPS, or — with `--skip-vps-cancel` — dropped the DB row while leaving the slice's lima VM running on the box. The wrapper now injects both teardown secrets (OVH AK/AS/CK and `POOL_SSH_PRIVATE_KEY`) from the activated tier's Vault, and `just destroy-pool-host <id>` works for a slice with no extra flags.
- Changed: Predefined-permission dialog labels the catch-all permission `all` instead of `any` (the stored value is still `any`); while `all` is checked the specific-permission checkboxes are disabled, and unchecking it re-enables them.
- Changed: Approving or denying a permission request no longer blocks on the agent nudge — `MngrMessageSender.send` dispatches the `mngr message` onto a background thread and returns immediately so the dialog responds without waiting for the network-bound delivery.
- Changed: Cut minds `0.3.1` and standardized the release tag convention on the `minds-v<version>` prefix across both `imbue-ai/mngr` and `imbue-ai/forever-claude-template` (FCT had been the odd one out on bare `v<version>`). `apps/minds/package.json` bumped to `0.3.1`; `templates.py` `FALLBACK_BRANCH` repointed to `"minds-v0.3.1"` (the FCT tag the shipped binary clones at runtime). `apps/minds/docs/release.md` rewritten for the `main`-based, two-PR flow.
- Changed: Trimmed potentially confusing parts of the messages sent to agents after approving or denying permission requests.
- Changed: Workspace recovery self-heals across chrome/sidebar reloads (Electron replays the latest non-healthy workspace status on view (re)load; backend re-asserts statuses periodically) and routes a known-stopped mind from the landing page straight to recovery instead of through the stuck-detection delay.
- Changed: `minds pool {create,list,destroy}` now resolves the staging/production host_pool DSN from Vault (`secrets/minds/<tier>/neon.DATABASE_URL`) automatically (alongside OVH creds and the management SSH key), so the commands work on those tiers without a hand-passed `--database-url`. An explicit `--database-url` still wins; dev/ci continue to auto-resolve from per-env `secrets.toml`.
- Changed: Pool-host docs point at the canonical `minds pool create` flow (via new `just bake-pool-host` / `just list-pool-hosts` / `just destroy-pool-host` recipes) instead of the low-level `mngr imbue_cloud admin pool create` recipe with hand-exported OVH creds and `--management-public-key-file`.
- Changed: Refreshed minds docs after the Vultr→OVH pool migration (environments / vault-setup / host-pool-setup / design / overview / desktop-app / user_story / latchkey-permissions; README install snippet replaced with the real from-source dev flow; corrected Modal app names, `minds run` config-resolution description, `minds` → `minds run`, `mngr events` → `mngr event`, dropped the spurious `kv/` Vault path prefix and the stale `--id <id>` flag from `mngr create` examples).
- Changed: Desktop client's latchkey services catalog (`ServicesCatalog` / `ServicePermissionInfo`) moved to `imbue.mngr_latchkey.services_catalog` and now reads bundled `services.json` directly instead of fetching from the running gateway's `GET /permissions/available`. Removes a dependency on gateway liveness for static package data, and lets the same catalog drive both the desktop dialog and server-side credential sync. The gateway client's now-unused `get_available_services` and `AvailableServiceEntry` are removed.
- Changed: Bumped Latchkey to 2.15.1 to include the Playwright compatibility fix.
- Changed: A workspace no longer flickers out of the minds desktop list when its provider has a transient discovery error — minds retains agents/hosts whose provider errored on a poll and marks the affected workspaces stale (an amber dot in the sidebar) while keeping them fully clickable; they are only removed on an explicit destroy or a later clean poll. On the same provider-set changes that already bounce minds' `mngr forward` observe (provider enable/disable, imbue_cloud account add on signin, account removal), minds now also bounces the detached `mngr latchkey forward` supervisor so latchkey's discovery stays in lockstep without a full minds restart.
- Changed: Overhauled the desktop client's templates — migrated from Jinja2 macros to JinjaX components and extracted recurring inline Tailwind patterns into reusable primitives, with standardization sweeps unifying text colors, corner radii, borders, and shadows. HTML output is semantically unchanged; no user-visible change.
- Changed: Minds no longer runs a second discovery observer; its `mngr forward` subprocess tails the shared discovery events file instead of spawning its own. No user-visible change.
- Changed: Startup env-setup failures are now logged to the console in addition to being shown in the error window.
- Changed: Permission requests are no longer collapsed by service/scope/path — every distinct request the agent makes shows as its own card. Redeliveries of the same request id are still collapsed.
- Changed: Permission-request dialogs now open in a transparent full-content-area modal overlay (`modalView`) stacked above the workspace with a dim backdrop, instead of replacing the main content window.

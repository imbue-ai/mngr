# Changelog - minds

A concise, human-friendly summary of changes for the `minds` app. Entries are categorized using the [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) categories: Added, Changed, Deprecated, Removed, Fixed, Security.

For the full, unedited changelog entries, see [UNABRIDGED_CHANGELOG.md](UNABRIDGED_CHANGELOG.md).

## [Unreleased]

### Added

- Added: Bumped bundled Latchkey to 2.21.0.
- Added: Workspace/account state now syncs across a user's devices, end-to-end encrypted. Each associated workspace has a record on the Imbue Cloud connector — plaintext metadata (name, color, provider, which device hosts it) plus a secrets blob (the workspace's SSH key material and its backup `restic.env`) encrypted under a per-account data key that only the user's master password can unwrap. The master password's only role is wrapping that per-account key; new backup repositories are initialized with the workspace's own random password, the per-workspace "backup encryption method" / master-password inputs are gone from the create form and workspace settings, and the Settings page's password change rewraps the key (and pushes/clears synced secrets) instead of rekeying every repository. Clearing the password scrubs all synced secrets server-side; while empty, only metadata syncs. The landing page and sidebar list workspaces on the user's other devices as greyed rows with an "on <device>" badge and a remove-from-list action; a new-device unlock banner asks for the master password once and installs the account key. Reconcile heals a missing server-side key bundle (converted legacy installs push their bundle up so a reinstall after a machine loss can still unlock). Legacy local state (`workspace_associations.json` — or the older `sessions.json` — plus `backup_password_hash` and `backup_password`) is converted once at startup and renamed aside with a `.pre-sync` suffix. Destroying a workspace tombstones its record so its backups remain reachable from any device.
- Added: Cloud (imbue_cloud) workspaces are fully accessible from any installation after unlocking with the master password. Unlocking (and every sync pass for unlocked accounts) materializes each cloud record's synced SSH key material into the mngr profile's per-host layout (compare-and-write, self-healing, never touching hosts this install leased itself), eagerly materializes backup envs, sweeps key dirs for destroyed/removed workspaces, and bounces discovery. Remote cloud tiles show derived access states ("connecting" while discovery catches up, "unreachable" when a healthy snapshot lacks the host, "sync error" with detail when materialization fails), and the workspace list updates live when a tile flips to accessible.
- Added: Post-signin sync-status banner on the workspace list, create, and accounts pages. When an account signs in on a device with no locally synced records yet, the banner shows "Fetching synced workspaces…" while the first pull is in flight, a retrying notice if it fails, and a dismissible "Found N synced workspaces – View" link once it lands (brand-new accounts never see it). Backed by per-account initial-sync tracking on the sync scheduler and a new `/_chrome/sync-initial-status` endpoint.
- Added: Bug reports attach the detached `mngr latchkey forward` daemon's logs (`events.jsonl` and `latchkey_forward.log`) and the shared mngr discovery event stream (`events.jsonl`), alongside the existing flat minds logs — the missing files where provider discovery actually logs and where the startup replay reads from, without which discovery bugs (e.g. destroyed workspaces reappearing at startup) were undiagnosable from a bug report.
- Added: Each install is assigned a stable, randomly-generated anonymous user id (no PII), persisted at `<data_dir>/anonymous_user_id`, and attached to every Sentry error report across all of minds' reporting surfaces (Python backend, `mngr latchkey forward` daemon, browser web UI, Electron main process). Lets Sentry report the number of distinct installs affected by each issue.
- Added: Released **minds v0.3.8** — pairs the current mngr `main` with a default-workspace-template snapshot that adds a safe `update-self` workspace-update flow and honest rendering of workspace-permission requests. On the mngr/minds side it carries cloud workspaces accessible from any unlocked installation, the OAuth sign-in hang fix, the post-signin synced-workspace fetch banner, and auto-verification of paid users.
- Added: Released **minds v0.3.7** — pairs the current mngr `main` with a default-workspace-template snapshot that includes the earlyoom OOM-prioritization changes.
- Added: Documented a fast-forward release path in `apps/minds/docs/release.md`. For low-risk bumps (no functional mngr/minds code, no `system_interface`-facing vendor change), skip the CI-surface PRs and pre-merge launch-to-msg, fast-forward the pair onto `main` to freeze the SHA pair, tag, and run launch-to-msg once on the tags.
- Added: Electron main-process logs now persist to `~/.minds/logs/electron.log` (uncaught exceptions and unhandled rejections captured) alongside the backend's `minds.log`, so main-process failures are durably diagnosable in packaged builds. Both logs rotate at 10MB (matching the Python backend's jsonl sink) keeping the 10 newest gzipped rotations. Bug reports upload both current logs plus the most recent gzipped rotation of each under accurate names (`backend_logs` / `electron_logs`); a report filed while the backend is down still attaches both.
- Added: Chrome-style crash recovery across all three per-window views. A workspace content view whose renderer crashes (e.g. killed over a long sleep) now shows an "Aw, Snap!"-style crash page with the crash reason and a manual Reload button instead of a blank white screen. The titlebar renderer's death shows a compact in-titlebar error strip with a Reload button (leaving the workspace content running); the overlay/menu renderer is silently reloaded so the next sidebar/inbox/help open works. Out-of-memory renderer deaths (`oom`, in any of the three views) report to Sentry automatically (subject to the error-reporting opt-in) labeled by which view died; native renderer crashes already flow to Sentry as minidumps, and sleep/external kills (`killed`) are deliberately left unreported as unactionable noise.
- Added: Helpful guidance when creating a workspace from a private/nonexistent GitHub repo. Any failed clone of a github.com URL now shows a creating-page explanation ("the repository looks private"), recommends signing in with the GitHub CLI (`gh auth login`, linking to the official quickstart), and offers the alternative of cloning the repository locally and entering its path instead. Non-GitHub git remotes get the same guidance minus the `gh auth login` recommendation. The create-operation status API (`GET /api/v1/workspaces/operations/create/<id>`) gained an optional `error_kind` field (`GITHUB_AUTH_REQUIRED` / `GIT_AUTH_REQUIRED`) alongside the human-readable `error` message.
- Added: Docker compute provider picks its container runtime per platform. The create form gained an advanced "Container runtime" setting (runc vs runsc) that defaults to runc on macOS (where gVisor is unavailable) and runsc on Linux, and can be overridden per workspace — macOS users no longer need the `MNGR__PROVIDERS__DOCKER__DOCKER_RUNTIME=runc` env-var workaround. Selecting runsc stacks a new `docker_runsc` create-template overlay on top of the shared `docker` template, so the gVisor choice is the only difference between the two runtimes.
- Added: Settings page gains a left nav with a Permissions section (Connectors, File sharing, Workspace delegation) that inspects and revokes agent access across all workspaces. Revoking removes only the permission rule; saved sign-ins stay so agents can reconnect later.
- Added: Released minds v0.3.6 — rolls up all mngr/minds changes that landed on `main` since v0.3.5; pins `FALLBACK_BRANCH` at the `minds-v0.3.6` forever-claude-template tag.
- Added: `minds server list` / `minds server prep` — env-aware wrappers around `mngr imbue_cloud admin server {list,prep}` that resolve the host_pool DSN from the activated env and inject the tier's `POOL_SSH_PRIVATE_KEY` from Vault, so operators no longer need to hand-export those values to inspect or (re-)prep bare-metal boxes.
- Added: Backup service verification + one-click idempotent "Update backup service" action on running workspaces. The backup status check gains a per-workspace verification (one exec compares the installed `libs/host_backup` code against the `minds-v*` tag matching the app version, checks the `host-backup` supervisord program is RUNNING, and compares the workspace `restic.env` against the minds-side canonical copy); any problem shows a single warning badge on the workspace tile. The Update button checks out the code at the target tag, commits it, runs `uv sync`, and restarts + verifies the service (auto-rollback on failure). Backups can now be enabled on a configure-later workspace post-creation, and a workspace's destination can be changed (fresh provisioning against the new repo, old canonical env archived, existing snapshots retained). A per-workspace "backup verification" toggle disables the checks + badge for workspaces that deliberately run without backups.
- Added: Modal as a selectable compute provider in the create form ("Modal (1-day ephemeral)", DIRECT mode — the local machine authenticates to Modal with its own token). Selecting it shows a short note on how to authenticate (`uv tool install modal`, `modal token new`, or `MODAL_TOKEN_ID` / `MODAL_TOKEN_SECRET`). The create wires up like the other remote providers (a `modal` host address plus the `main` + `modal` provisioning templates), and `bootstrap` writes a default `[providers.modal]` block at startup. Modal deliberately gets no Stop/Resume/state-label lifecycle UI — it behaves like the other cloud providers (aws/vultr/imbue_cloud); a Modal sandbox that has timed out may still appear in the list (there is no liveness polling to mark it dead), and resuming a dead sandbox from a snapshot is intentionally not offered.
- Added: Minds-tailored TMR mapper prompt at `apps/minds/tmr/mapper.j2`, used by the `tmr-minds` variant. Keeps the shared docstring-as-contract philosophy but drops the mngr-only tutorial-block guidance and calls out minds-specific setup blockers (a Docker daemon, a snapshot image, a deployed env, or secrets), which the agent flags with `FIXME(tmr)` and a BLOCKED status rather than hacking around.
- Added: Workspace rename — new `POST /api/v1/workspaces/<agent_id>/rename` endpoint updates the workspace's normalized host name (slug) and its human-readable display name together, so the two never drift. When the new name normalizes to the same slug as the current host name, only the display name changes (works on every provider, online or offline). Collisions with another active workspace on the same provider are rejected (409). Workspace names are now decoupled into two concepts with a single canonical home each: the human-readable display name lives in a `workspace_display_name` label on the workspace's `system-services` agent (arbitrary characters, mixed case), and the host name is a normalized lowercase slug derived from it. The `workspace` label is removed entirely (discovery now keys off `is_primary`, per-provider name collisions check against the actual host name, and the display name comes from the new label). Legacy workspaces fall back to showing their host name; this fallback is temporary and can be removed around September 2026.
- Added: Titlebar buttons (Main Menu, Home, Back, Forward, Get help, Requests, and window controls) show custom styled tooltips on hover and keyboard focus, replacing the OS `title=` tooltips that were unreliable in the desktop window and could not extend past the titlebar. Tooltips appear after a short hover delay, float above both the chrome and workspace content, and reposition to stay on-screen; the buttons keep an accessible name (`aria-label`) so screen readers still announce them. Landing-page per-workspace action buttons (Start, Stop, Restart, Open in new window, Settings) and dialog close buttons show the same styled tooltips.
- Added: `apps/minds/docs/persistent-terminals.md` — a short doc explaining the lifecycle of the dockview terminals (in-memory tmux sessions that survive tab close, page reload, and terminal-service restarts, but not a container restart) and how to opt in to on-disk persistence. The in-workspace terminal banner links here. The terminal feature itself lives in the forever-claude-template repo.
- Added: "Have an agent help fix the problem" option in the get-help modal (default choice in a loaded workspace) — describing the problem spawns a new chat in that workspace that diagnoses and fixes the issue, and the modal closes itself once the chat is ready.
- Added: Pre-baked Lima VM image cache for local workspace creation (issue #2306) — the current release's image is downloaded via `desync` content-defined-chunking deltas, verified against an Ed25519/minisign signature, and booted by Lima instead of rebuilding the toolchain in-VM. New `imbue.minds.lima_image` package with background prefetch on `minds run` startup, fallback to in-VM build when no image is published or verification fails, per-env opt-in via `client.toml` (`lima_image_base_url`, `lima_image_minisign_public_key`), and `MINDS_DISABLE_LIMA_IMAGE_CACHE=1` to disable. Bundles the `desync` binary alongside `uv`/`git`/`lima`.
- Added: Live validation of the create-form's advanced "Name" field — format errors and availability conflicts surface inline as you type. The availability check is case-insensitive and scoped to the selected compute provider (and account, for Imbue Cloud); pressing Create with an invalid or taken name surfaces the inline error instead of starting creation.
- Added: Versioned `/api/v1/workspaces` cross-workspace management API reachable by agents through the latchkey gateway — list, get detail, version, list/export backups, create, destroy, start/stop (with operation-status polling and SSE operation logs), restart, SSH, and `PATCH` workspace fields. All routes accept either the central bearer key (agents) or the desktop session cookie.
- Added: `GET /api/schema` — self-describing OpenAPI 3.1 document of every agent-reachable Minds API route, generated at request time from the live Flask routes plus pydantic models. Default-granted to every workspace.
- Added: `GET /api/v1/accounts` — lists signed-in accounts so an agent can discover the id/email to pass to workspace association. Gated by a new must-ask `minds-accounts-read` permission (not in the agent baseline).
- Added: Cross-workspace SSH tunnel broker for local Docker/Lima targets — `POST /api/v1/workspaces/<id>/ssh` against a local target brokers a reverse SSH tunnel into the calling workspace and returns a loopback endpoint, in addition to the existing remote-target path.
- Added: User-controlled error reporting via a "Help improve Minds" consent screen (shown on first launch and after upgrading) and a Settings page reached from the sidebar. Two toggles ("Report unexpected errors" and "Include logs") take effect live, gating what Sentry sends across the backend, the browser web UI, and the Electron main process. The `MINDS_SENTRY_ENABLED` and `MINDS_SENTRY_S3_UPLOADS` env vars are retired.
- Added: Minds-provided Google OAuth consent screen is now tried first when granting an OAuth Google permission (Gmail, Calendar, Drive, Docs, Sheets, People, Analytics), falling back to the old "create your own Google project" self-setup flow only if that fails — most users no longer see the self-setup step. A user's own Google OAuth client is never cleared. `google-directions` (API key, not OAuth) and non-Google services are unchanged.

- Added: "Get help" button (question mark) in the top bar opens a modal for reporting a bug to Imbue. Reports include optional app-diagnostics / workspace-details / remote-access checkboxes (remembered across reports), attach logs when the include-logs setting is on, and show a copyable Sentry reference ID on success. Workspace-recovery and full-app error screens carry report buttons too; a backend-down report still files from the app shell with host info and (opt-in) logs.
- Added: In-workspace agent bug reporting via a new authenticated `POST /api/v1/agents/<id>/report` endpoint.
- Added: Welcome / sign-in screen shown whenever the desktop app is functionally empty (signed out of every account, no workspaces) instead of silently restoring a stale window from a prior session.

- Added: Discovery-health watchdog — minds now watches its own workspace-discovery pipeline and tries to recover it automatically. The watchdog detects a stall (via snapshot freshness) and self-heals with a cheap producer nudge that leaves your open workspace untouched, then, if that doesn't help, a heavier restart. If the pipeline can't be recovered (healing attempts exhausted, the discovery consumer died, or a fresh launch never managed a first update), minds takes over the whole app with a clear "disconnected from your workspaces and can't automatically reconnect" screen and a "Restart Minds" button instead of leaving you stuck on an indefinite "Loading workspace" page. The old "Forwarding subprocess died — restart minds" notification is removed in favor of this screen.
- Added: Backend Sentry error reporting — `minds run` initializes Sentry during startup (Flask integration carries request context: transaction name, URL, method, query string, headers), reports Python backend errors with attached log files and traceback-with-locals uploaded to S3. Selects DSN and S3 bucket from the activated minds env (production / staging / dev with no S3 uploads). Gated behind `MINDS_SENTRY_ENABLED` (default off); S3 attachment uploads behind `MINDS_SENTRY_S3_UPLOADS` (default off, even in production / staging). Performance tracing is disabled (`traces_sample_rate=0.0`). Reports the desktop app version as the Sentry release and the git SHA as the `git_sha` tag.
- Added: Hardened Sentry error reporting against failures inside Sentry event processing (the `before_send` hook, Sentry callbacks, and the HTTP transport) — failures now always land in the local app log and report via a minimal Sentry event. Made `log_error_inside_sentry` non-reentrant so a failure inside `before_send` can no longer recurse until the stack is exhausted.
- Added: `minds pool backfill-host-keys` — env-aware wrapper around `mngr imbue_cloud admin pool backfill-host-keys`. Run it once per tier after deploying the host-key-pinning connector to keyscan + record SSH host keys for pool rows and bare-metal boxes baked before the host-key columns existed, restoring leasing / prep for those hosts.
- Added: Compiled Tailwind v4 build step replaces the runtime Tailwind Play CDN — a single minified, tree-shaken `app.min.css` built ahead of time from `static/app.css` (`just minds-css`, replaces `just minds-tailwind`). Packaged builds compile it before packaging via `scripts/build.js` and `pnpm start` compiles it via its `prestart` hook. Fully offline and smaller. `just minds-start` runs the compiler in `--watch` mode alongside Electron, so class changes rebuild live.
- Added: Design-token system with light / dark mode — text, border, surface / fill, status / feedback, and accent / interactive token families, plus a 16x16 icon set (replacing lucide), six-role semantic type ramp, constrained spacing scale, four-step corner-radius scale, and a two-step elevation scale. Switching the whole app between modes is a single `.dark` class on `<html>`. A new dev styleguide at `/_dev/styleguide` (visible in every tier) catalogues the tokens and patterns, with a floating light/dark toggle, sticky TOC, and scrollspy.

- Added: `apps/minds/docs/vendor-mngr-sync.md` as the single source of truth for how FCT's `vendor/mngr` is synced (the two mechanisms: `git archive` for reproducible committed release snapshots, `rsync` for working-tree dev iteration and pool bakes).
- Added: Released minds v0.3.2 (2026-06-22) and v0.3.3 (2026-06-24). Both pin `FALLBACK_BRANCH` at the matching `minds-v<version>` forever-claude-template tag, so a shipped binary clones the FCT snapshot it was verified against. v0.3.3 also carries the Tailwind v4 ToDesktop build fix.
- Added: `minds pool create --backend slice` bakes pool hosts onto bare-metal "slices" (lima VMs carved on a pre-registered, prepped bare-metal box) instead of OVH VPSes. Default `--backend ovh_vps` is unchanged. The slice path resolves the tier's `pool-ssh` private key from Vault and injects it as `POOL_SSH_PRIVATE_KEY` (mirroring the OVH path's secret injection); accepts `--dry-run` and `--max-concurrency`; requires `--server-id` (the explicitly-chosen bare-metal box); and rejects OVH-only flags with a clear error.
- Added: `minds paid {add,remove,list}` command group (plus a `just add-paid-email <email>` one-liner) for managing the connector's paid-user allowlist from an activated env. Like `minds pool`, it resolves the connector URL and the paid-list admin key from `client.toml` and Vault automatically.
- Added: `MngrCaller` keeps a single pre-warmed, single-use `mngr` process running ahead of time and hands each call's argv over an anonymous, connected `socketpair`, so in-app `mngr` invocations (e.g. the `mngr message` after a permission decision) skip the multi-second interpreter-and-plugin import cost. `MngrMessageSender` routes through it by default.
- Added: imbue_cloud workspace creation now passes the form's repository through to the lease as `-b repo_url=<repository>` (the imbue_cloud provider canonicalizes it via the new `repo_identity.canonicalize_repo_source`), so the fast path can only adopt a pre-baked host that genuinely matches the requested repo. Previously the repo was dropped and only an operator-chosen branch label was matched, so a request for one repo could silently adopt a host running another.

Each row hovers in a settings gear and an "Open in new window" arrow; two rows at the bottom go to Create and Manage account(s); the menu dismisses on outside-click / Escape, sharing the inbox's modal overlay so the workspace stays in place.

A failed restart polls in the background and returns the user to the workspace once it recovers. Adds a sidebar "Restart workspace…" context menu entry and a per-workspace restart button on the home page.

In the desktop app the content view gains a minimal, allowlist-only relay preload (no `window.minds` bridge); in browser mode the shell navigates the content iframe to the request page instead.

### Fixed

- Fixed: Tooltips clipped on their right side and modal overlays (get help, inbox, workspace menu, sign-in) leaving an un-dimmed strip at the window's right edge on Macs with always-visible ("classic") scrollbars. The global `scrollbar-gutter: stable` rule (added so content pages don't shift when a classic scrollbar appears) also applied to edge-to-edge chrome and overlay surfaces, reserving a 15px gutter that nothing painted. Those surfaces now opt out via a new `OverlaySurface` template wrapper and an `html_class` argument on the Base template.
- Fixed: Synced workspaces silently de-associated by a startup race. The absent-host tombstone pass could tombstone a workspace record while its provider was merely slow to produce its first discovery listing (e.g. a stopped Lima VM right after app start), pushing a spurious "destroyed" state to the connector. The pass now requires the record's specific provider to have produced at least one snapshot before treating absence as evidence. Adds the inverse safety net: a tombstoned record hosted by this install whose workspace is live in local discovery is automatically re-activated and re-pushed (self-healing rows tombstoned by the old race), guarded so an in-flight destroy or a genuinely-destroyed host lingering in discovery is never resurrected.
- Fixed: ~60s hang on app launch and on quit when the local Docker daemon is unresponsive. Minds starts/stops an mngr Docker "state container" around its lifecycle; if the Docker daemon was wedged (its socket accepts connections but never replies), each `docker start`/`docker stop` blocked until its 60-second command timeout, delaying the backend coming up (launch) and the app closing (quit). Launch and quit now probe daemon reachability first (`docker version`, bounded to 5 seconds) and skip the state-container operation when the daemon is unreachable, wedged, or absent. A healthy daemon answers in well under a second.
- Fixed: Google/GitHub OAuth signin flow hanging on "Waiting for you to finish signing in with the browser…" after a successful signin. The OAuth background thread called `_kick_sync_scheduler()`, which resolves the desktop-client state through Flask's application-context proxy; a plain thread has no app context, so the thread crashed with `RuntimeError: Working outside of application context` before recording the flow as done — and before registering the account's imbue_cloud provider entry, so remote workspace discovery for the just-signed-in account silently stayed broken too. The sync scheduler handle is now passed into the thread explicitly, and any unexpected crash while mirroring the signin result now records an "error" flow status so the signin page can never hang on a dead thread.
- Fixed: Workspace whose host was offline since before the app started could be stranded on "Loading workspace" forever, and offline-host restarts on docker were delayed until a later STUCK-triggered reprobe. The recovery page's automatic cold-boot dispatch was silently skipped whenever the health tracker had never probed the workspace — the tracker reports HEALTHY by default for agents it has no record of, and the restart endpoint's "already recovered" veto misread that default as a confirmed recovery. The endpoint-side veto (and the `auto_dispatched` request marker driving it) is removed entirely; `mngr start`'s idempotency (targets STOPPED agents only, starts the host idempotently) absorbs the race for the host tier.
- Fixed: Workspace UI hang once enough streaming tabs were open in one workspace. The workspace origin was served over plain HTTP/1.1 which Chromium caps at ~6 held-open connections per origin, so a workspace with that many long-lived streams (one SSE per open chat tab, plus any `/service/` app's own streams) queued every further request indefinitely. Minds now runs `mngr forward` with `--use-http2` so the workspace origin terminates TLS and negotiates HTTP/2 (which multiplexes many streams over one connection); the Electron shell trusts the proxy's ephemeral self-signed cert for its loopback origins (`localhost`, `*.localhost`, `127.0.0.1`) in the workspace-content session only, and every real https origin still gets Chromium's normal verification. The minds bare-origin backend (Home/Create/chrome/sidebar and its `minds_session` cookie) stays plain HTTP.
- Fixed: Recovery page no longer renders a restart verdict for a workspace whose host could not be observed during discovery. Previously an unreachable imbue_cloud host surfaced as CRASHED, which the recovery classifier read as trusted "container is down" and auto-dispatched an unattended host restart doomed by construction (the restart routes over the same outer SSH that is unreachable). With imbue_cloud now surfacing unreachable hosts as UNKNOWN, the dispatch-tier classifier treats a host state that answers neither "running" nor "offline" as non-evidence and shows the live "Reconnecting" state. The consent-gated HOST_UNRESPONSIVE verdict is reserved for containers observed running but unreachable inside (a RUNNING claim whose in-container exec cleanly failed, or the UNAUTHENTICATED state); HOST_OFFLINE (unattended restart) still requires an actual observation that the container is stopped.
- Fixed: Minds backend now tells Sentry to ignore the paramiko/pyinfra stdlib loggers. The backend brokers cross-workspace reverse SSH tunnels through paramiko, so handled SSH connection-failure noise (logged at ERROR by paramiko's transport thread) would otherwise be captured by Sentry's default logging integration and flood the project with un-rate-limited events. Genuine, actionable failures still reach Sentry as typed exceptions logged through loguru.
- Fixed: Discovery-health watchdog's producer-stall threshold raised from 35s to 180s so a merely-slow-but-alive producer no longer trips it. The old value sat only a few seconds above the worst-case healthy cadence between discovery snapshots (~150s with a single slow provider), triggering spurious supervisor bounces/restarts — each of which re-provisions every managed host.
- Fixed: Permission-request dialogs opened from a notification now close the whole inbox after Approve/Deny instead of advancing to the next pending request (which could surface an unrelated stale request from another agent). Opening the full inbox via the Requests button still advances through pending requests as before.
- Fixed: `minds.log` no longer floods with repeated "Discovery error from ..." warnings — provider-level discovery errors log once per process with an info-level recovery when discovery next succeeds. Host- and agent-attributed errors are unaffected.
- Fixed: The `[providers.modal]` config block's panel-toggled `is_enabled = false` is no longer reset back to enabled on every startup.
- Fixed: Warm `mngr` processes spawned by `MngrCaller` now SIGTERM when their parent dies mid-request, so an orphaned warm process is no longer left behind.
- Fixed: Workspace-recovery flow stranding users on a "Workspace unresponsive" page after their computer wakes from sleep, even when the workspace was healthy the whole time. The recovery page now runs a cheap liveness poll from the moment it opens under every state (so a workspace that comes back on its own returns you to it immediately); a timed-out health probe is treated as "no answer yet, keep checking" rather than proof the workspace is down; the recovery page can appear promptly when a workspace stops responding and shows a live "Reconnecting to your workspace" state while data is still pending; a dropped mid-flight health request now retries in the same reconnecting state instead of dead-ending on a static verdict; and the probe re-reads the host's state at classification time so a fresh discovery update no longer opens the freshness gate for a stale reading.
- Fixed: "Destroying..." spinning forever for cloud workspaces — the detached destroy subprocess (`mngr list | mngr destroy`) now inherits `MNGR_HOST_DIR`, like every other minds->mngr shell-out, so it targets the correct host dir instead of the default (where it silently found nothing to tear down).
- Fixed: "Restart workspace" recovery action for providers that cannot stop a host in place (Modal). A host-restart runs `mngr stop --stop-host` then `mngr start`, but `mngr stop --stop-host` raises `HostShutdownNotSupportedError` for such providers, which previously aborted the whole restart. The restart worker now treats that specific error as "skip the stop" and proceeds to `mngr start`, which restarts the host on its own (reconnect-if-alive, else recreate-from-snapshot). Provider-agnostic — keys off the error via mngr's exported `HOST_SHUTDOWN_NOT_SUPPORTED_MESSAGE` constant, not a hardcoded backend name.
- Fixed: "Report a bug" button doing nothing on the full-app error takeover screen (discovery-pipeline takeover). The error-state window layout collapsed the modal to zero size, and the takeover screen's full-window drag region intercepted every click meant for the modal (macOS unions drag regions across stacked views). The error layout now lets an open modal overlay the full window (its own dim backdrop shows the error screen through it), and the takeover screen drops its window-drag region while a modal is open, mirroring the main chrome titlebar.
- Fixed: Bug-report form's text field losing focus once per second whenever it was opened over a "Loading workspace" screen. Both the proxy loader and the recovery page's "restarting" state used to full-reload themselves every second; in Electron a full reload steals OS focus from any sibling view layered above it. Both now poll the workspace in the background and only navigate once its state actually changes, leaving the focused report form untouched.
- Fixed: Slow memory leak in the discovery consumer — cached per-host SSH info is now forgotten when a host is removed, instead of accumulating for the lifetime of the session.
- Fixed: Permission-grant flow treated `UNKNOWN` latchkey credential status as "needs credential setup" — it now treats it as "proceed", so the dialog no longer spuriously prompts for credentials latchkey cannot vouch for (e.g. `rawCurl` credentials or minds-internal API scopes served by a gateway extension).
- Fixed: Sharing now requires at least one email — `PUT .../sharing/<service>` with no `emails` previously created the tunnel and registered the service but applied no Cloudflare Access policy, leaving the service publicly reachable; the empty/missing list is now rejected up front before any tunnel side effects.

Such ids are now rejected up front at the gateway, and as defense-in-depth the consumer catches errors per-request — previously the error killed the thread, stopping every permission request, for any agent or service, from reaching the UI until restart.
- Fixed: ToDesktop cloud build (and the scheduled `minds launch-to-first-message` health check) failing on every commit since the Tailwind v4 migration. ToDesktop's cloud builder installs the app with `pnpm recursive install --prod`, which omits devDependencies but still runs lifecycle scripts; the `postinstall` hook compiled CSS via the dev-only `tailwindcss` CLI, so the install aborted with "command not found" before any platform could be packaged. - Fixed: Creating a mind from a remote git URL (e.g. a GitHub HTTPS URL) when no branch is specified. Cloning without an explicit branch previously left the local clone on detached HEAD, so the downstream `mngr create` mirror push (which only pushes `refs/heads/*` + `refs/tags/*`) failed with "No refs in common". The no-branch clone now uses a plain `git clone`, which resolves the remote's default branch natively and leaves a real named local branch checked out.
- Fixed: Local-Docker Electron e2e test (`test_create_local_docker_workspace_via_electron`) failing once the discovery-health watchdog landed. - Fixed: `just minds-build` failing during `uv lock` with "no version of overlay==0.1.0". `imbue-mngr` now depends on the unpublished workspace package `overlay`, which was missing from the build's hand-maintained bundled-package lists; added `overlay` to all four mirrored lists so it is bundled as a wheel and resolved locally.
- Fixed: Workspace post-create setup hardened against slow `mngr` invocations — onboarding permissions-preference `mngr exec` timeout raised from 30s to 60s, and debug logging added around each `mngr imbue_cloud …` subprocess so a slow or timed-out post-create operation (Cloudflare tunnel create, backup bucket create, etc.) is attributable instead of surfacing only as a bare "exit -15".
- Fixed: Rare bug where approving a latchkey permission request could fail with "Could not apply grant through the latchkey gateway: ... 500 ... ENOENT" when the agent's per-host permissions file had never been created (e.g. agent creation's finalize/link step was skipped or failed). The desktop client now self-heals on a new permission request: it checks whether the host's canonical `latchkey_permissions.json` exists and, if not, recreates it from the agent's opaque permissions handle so the user's approval takes effect without re-creating the agent. It also idempotently re-registers the requesting agent in the host's allowlist, covering the case where discovery-time auto-registration skipped the agent while the host file was missing.
- Fixed: `apps/minds/docs/release.md` now tags the verified mngr SHA (`GREEN_MNGR_SHA` from step 4), not `main` HEAD which can drift past it; the merge step now uses a merge commit (not squash) so the verified SHA stays reachable on `main`, the vendor-match check verifies the FCT commit that actually gets tagged, and a new Session setup section defines `GH_TOKEN`/`MNGR`/`FCT`/`FCT_DIR` once up front so the runbook is copy-paste-correct. Caught while cutting `minds-v0.3.1`, where `main` had drifted +58 unrelated files past the verified SHA.
- Fixed: Pending permission requests whose originating agent's host can no longer be resolved (e.g. after the agent's workspace has been shut down) are now hidden from the desktop client inbox instead of rendering with raw agent ids. The request is left untouched on the gateway, so it reappears in the inbox if the workspace comes back. The inbox badge count and the rendered cards are driven off the same filter, so they stay in agreement.

- Fixed: Startup race where the minds desktop client could cache a stale latchkey gateway port and then fail every call with `[Errno 111] Connection refused`; the gateway client now self-heals on `httpx.ConnectError`/`ConnectTimeout`, and supervisor restart + pre-warm now run sequentially on a single background thread.

- Fixed: Signing-key generation race on a fresh data directory that intermittently logged users out (or returned 500 from `/authenticate`); generation is now serialized so concurrent requests can no longer read an empty key or race to write conflicting keys.
- Fixed: Deny button on the latchkey permission-request dialog now works even when the requested scope is not in the gateway's services catalog (e.g. a typo or stale catalog); the deny flow falls back to the raw scope string so the pending request is always torn down and the agent is always notified.

A failure immediately replaces the progress UI with a prominent red "We couldn't set up your workspace" heading, the underlying error message in a red box, the rotating tips stopped, and "Back to setup" / "Home" buttons to recover. The collapsible "Show details" log remains available.

Requesting a non-existent branch now fails cleanly at clone time.
- Fixed: Re-opening a permission request that had already been approved or denied no longer shows the actionable grant/deny form. The recovery page no longer flashes up for a workspace that is actually healthy: a workspace is only treated as stuck after the background probe loop confirms it unreachable with a sustained run of failed HTTP probes. - Fixed: Requests-panel X button was unclickable when the panel auto-opened at startup; startup state-priming now also replays the current modal-open state.
- Fixed: Startup loading window no longer flashes at the default centered position before jumping to its saved bounds; saved bounds from the previous session are now applied to the initial window before its loading screen renders.
- Fixed: Stopping a local mind from the landing page now closes any other window open to that mind, instead of leaving it stranded redirecting to the recovery page and auto-restarting the host. A window mid-restart is left alone, and if the open window is the only one left it falls back to the home page rather than closing (which would quit the app).

- Fixed: Window restore on app relaunch now reopens windows to the mind they were showing (persisted/restored by agent identity instead of the bare backend root); a slow provider can no longer drop a window from the first discovery snapshot (restore retains windows for minds known from the persisted last-good topology, dropping only on positive evidence of destruction); a reopened "create workspace" window whose creation no longer exists redirects to the landing page instead of getting stuck on "Unknown agent creation"; and after "Shut down all" the next launch restarts the docker state container early so reopening doesn't drop restored windows onto the create-workspace form.

- Fixed: Shutdown race where the background workspace-record sync loop could crash with `MngrCallerNotInitializedError` — the shared mngr caller was being torn down while a sync pass was still running. - Fixed: Creating a workspace from a plain local directory (non-worktree clone) with an explicit branch no longer runs `git checkout -B <branch> FETCH_HEAD` in the user's own template checkout. ### Security

- Security: Redact secrets from persisted logs. The `mngr create` subprocess is now given a log-safe process `name` with the latchkey gateway password and permissions-override JWT masked, so those secrets no longer leak into the JSONL log's `thread_name` field or `ProcessError` messages. The `modal secret create` subprocess used by `minds env` deploy tooling is likewise named with its `KEY=VALUE` secret values masked so raw Vault secrets no longer reach the thread name or error messages. The Electron backend log (`minds.log`, uploaded with bug reports) now masks the `mngr forward` preauth cookie — a reusable, session-lifetime bearer token — before writing backend stdout events to disk. The single-use one-time login code is left as-is (spent immediately at session start).

### Added

- Added: New `minds run` command that spawns `mngr forward` as a subprocess and consumes its JSONL envelope stream instead of running forwarding in-process.

- Added: minds installs a grandparent-death watcher so the Python backend self-terminates ~3 s after Electron crashes, cascading into `mngr observe` / `mngr events` / latchkey children.

### Changed

- Changed: Bumped bundled Latchkey to 2.8.0 and switched minds to a single shared `latchkey gateway` subprocess for all agents, with password-protected `LATCHKEY_GATEWAY_LISTEN_PASSWORD`.
- Changed: Reverse-tunnel health check now retries broken tunnels forever (capped at one attempt per 5 minutes via exponential backoff) instead of giving up after 10 consecutive failures.
- Changed: minds "Create a Project" screen — removed the .env checkbox, added an independent "AI provider" choice, renamed "Launch mode" to "Compute provider", added an optional `GH_TOKEN` Advanced field; revoked imbue_cloud sessions auto-disable the matching provider block.
- Changed: minds no longer persists `imbue_cloud` account identity to disk; only the workspace↔account map lives on disk, identity is sourced on demand.

- Removed: `LaunchMode.DEV` from minds — the web create form, `/create`, and `/api/create-agent` now only offer `LOCAL` / `LIMA` / `CLOUD` / `IMBUE_CLOUD`; the DEV-only latchkey helper and `MINDS_ALLOW_HOST_LOOPBACK` env var are gone.
- Removed: `apps/minds_workspace_server/` from the monorepo — migrated to `forever-claude-template`'s `apps/system_interface/` and consumed at runtime; the release Dockerfile's node/npm install step is dropped.

### Fixed

- Fixed: `minds run` no longer pegs a CPU as agents/hosts come and go — reverse-tunnel bookkeeping is pruned on agent destroy and the 30 s health-check loop applies per-tunnel exponential backoff.
- Fixed: WebSocket broadcaster queue-full flood — stuck WS clients are evicted after 50 consecutive queue-full broadcasts, and the broadcaster cancels wedged handler tasks blocked on `send_text(...)`.
- Fixed: Closing the last tab in a minds workspace no longer leaves a blank screen — the primary agent's chat tab is automatically reopened when the dockview becomes empty.
- Fixed: `apps/minds/scripts/propagate_changes` now protects `.claude/settings.local.json` from `rsync --delete`, preventing per-agent `UserPromptSubmit` hook loss and the resulting 90 s `send_message` hang.

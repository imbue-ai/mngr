Phase 0 of the minds chrome mithril migration (`specs/minds-chrome-mithril-migration/spec.md`): the frontend toolchain, with no user-visible behavior change yet.

New `apps/minds/frontend/` TypeScript package (mithril 2.x, strict tsc, vitest + jsdom) compiled by esbuild into a single IIFE classic script, `static/dist/chrome.bundle.js`, exposing the `window.MindsUI` namespace of mount functions. The swap-engine-compatible mount protocol (boot-state JSON island, synchronous mount, `minds:page-teardown` unmount) is implemented in `frontend/src/mount.ts` and covered by vitest.

`ChromeShell.jinja` and `OverlaySurface.jinja` load the bundle as a shell script (once per document, classic/not deferred, so inline per-page mount calls always find it); the swap engine never re-executes it.

The dev styleguide (`/_dev/styleguide`) gained a "JS components" section mounting a smoke component through the full mount protocol -- the live catalog slot for later converted components.

Build wiring: `build:js` / `watch:js` pnpm scripts run everywhere `build:css` does (`prestart`, `pnpm start` watch, `scripts/build.js`, the visual-diff harness); the bundle is force-included in the minds wheel via `[tool.hatch.build] artifacts`; `static/app.css` declares `frontend/src` as a Tailwind `@source` so component-only utilities are generated. Like the compiled stylesheet, a missing (unbuilt) bundle makes the server log a `just minds-js` hint at startup.

Phase 1 (chrome-state contract + frontend foundations): the SSE `/_chrome/events` payloads (`workspaces`, `providers_state`, `requests`, `system_interface_status`) are now built through pydantic models in `desktop_client/chrome_state.py`, with byte-identity tests pinning the exact legacy wire format (key order, absent-vs-null optionals). A `build_chrome_boot_state()` helper assembles the same connect-time snapshot for page boot islands, and `ChromeShell.jinja` gained a `boot_state` prop that renders the `#minds-boot-state` island inside the swappable page root.

The frontend package gained the Phase 2+ foundations, mirrored from the Python contract and covered by vitest: `chrome_state.ts` (payload types), `host.ts` (one Host interface with Electron-IPC and browser implementations -- shared EventSource with a single reconnect loop, swap-engine-aware navigation via chrome.js's new `window.__mindsNavigateContent` export, modal-kind routing), and `store.ts` (accent/name cache, optimistic mind start/stop transients with interim-payload clobber guards, provider-toggle pending logic, requests/status/auth state, backup-health bridge). No user-visible behavior change.

Phase 2 (shared primitives + workspace switcher menu): the switcher menu's interior is now a mithril component tree (`WorkspaceMenu` / `WorkspaceRow` plus the `Icon` / `Badge` / `Spinner` primitives), mounted into a positioning shell by both hosts -- chrome.js's inline browser menu and Electron's overlay sidebar page (whose `Sidebar.jinja` is now a pure positioning shell with an inline mount; backdrop dismissal moved into the component via `host.closeModal()`). The icon path data moved to `frontend/src/icons.ts` as the canonical source (templates.py keeps a shrinking copy for the remaining Jinja pages, with a byte-drift guard test).

Deleted: `static/sidebar_workspace_row.js` (the shared row builder), `static/sidebar.js` (the Electron menu driver), `templates/SidebarBottom.jinja`, chrome.js's `renderWorkspaces`, and the duplicated menu panel markup in `ChromeShell.jinja`. The styleguide's "Sidebar items" section and a new JS-primitives line render through the live components.

Behavior preserved: grouping/sort (Private first, accounts alphabetical, headers only with multiple groups), current-row highlight tracking the accent-source workspace (settings screens included), open-in-new-window arrows on Electron only (never on current/remote rows), remote rows non-navigable with a location badge, stale + backup-warning dots, native context menu (Electron), backdrop + Escape dismissal in both modes, and the -24px menu anchor alignment.

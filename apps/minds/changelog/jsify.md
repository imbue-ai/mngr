Phase 0 of the minds chrome mithril migration (`specs/minds-chrome-mithril-migration/spec.md`): the frontend toolchain, with no user-visible behavior change yet.

New `apps/minds/frontend/` TypeScript package (mithril 2.x, strict tsc, vitest + jsdom) compiled by esbuild into a single IIFE classic script, `static/dist/chrome.bundle.js`, exposing the `window.MindsUI` namespace of mount functions. The swap-engine-compatible mount protocol (boot-state JSON island, synchronous mount, `minds:page-teardown` unmount) is implemented in `frontend/src/mount.ts` and covered by vitest.

`ChromeShell.jinja` and `OverlaySurface.jinja` load the bundle as a shell script (once per document, classic/not deferred, so inline per-page mount calls always find it); the swap engine never re-executes it.

The dev styleguide (`/_dev/styleguide`) gained a "JS components" section mounting a smoke component through the full mount protocol -- the live catalog slot for later converted components.

Build wiring: `build:js` / `watch:js` pnpm scripts run everywhere `build:css` does (`prestart`, `pnpm start` watch, `scripts/build.js`, the visual-diff harness); the bundle is force-included in the minds wheel via `[tool.hatch.build] artifacts`; `static/app.css` declares `frontend/src` as a Tailwind `@source` so component-only utilities are generated. Like the compiled stylesheet, a missing (unbuilt) bundle makes the server log a `just minds-js` hint at startup.

Phase 1 (chrome-state contract + frontend foundations): the SSE `/_chrome/events` payloads (`workspaces`, `providers_state`, `requests`, `system_interface_status`) are now built through pydantic models in `desktop_client/chrome_state.py`, with byte-identity tests pinning the exact legacy wire format (key order, absent-vs-null optionals). A `build_chrome_boot_state()` helper assembles the same connect-time snapshot for page boot islands, and `ChromeShell.jinja` gained a `boot_state` prop that renders the `#minds-boot-state` island inside the swappable page root.

The frontend package gained the Phase 2+ foundations, mirrored from the Python contract and covered by vitest: `chrome_state.ts` (payload types), `host.ts` (one Host interface with Electron-IPC and browser implementations -- shared EventSource with a single reconnect loop, swap-engine-aware navigation via chrome.js's new `window.__mindsNavigateContent` export, modal-kind routing), and `store.ts` (accent/name cache, optimistic mind start/stop transients with interim-payload clobber guards, provider-toggle pending logic, requests/status/auth state, backup-health bridge). No user-visible behavior change.

Phase 2 (shared primitives + workspace switcher menu): the switcher menu's interior is now a mithril component tree (`WorkspaceMenu` / `WorkspaceRow` plus the `Icon` / `Badge` / `Spinner` primitives), mounted into a positioning shell by both hosts -- chrome.js's inline browser menu and Electron's overlay sidebar page (whose `Sidebar.jinja` is now a pure positioning shell with an inline mount; backdrop dismissal moved into the component via `host.closeModal()`). The icon path data moved to `frontend/src/icons.ts` as the canonical source (templates.py keeps a shrinking copy for the remaining Jinja pages, with a byte-drift guard test).

Deleted: `static/sidebar_workspace_row.js` (the shared row builder), `static/sidebar.js` (the Electron menu driver), `templates/SidebarBottom.jinja`, chrome.js's `renderWorkspaces`, and the duplicated menu panel markup in `ChromeShell.jinja`. The styleguide's "Sidebar items" section and a new JS-primitives line render through the live components.

Behavior preserved: grouping/sort (Private first, accounts alphabetical, headers only with multiple groups), current-row highlight tracking the accent-source workspace (settings screens included), open-in-new-window arrows on Electron only (never on current/remote rows), remote rows non-navigable with a location badge, stale + backup-warning dots, native context menu (Electron), backdrop + Escape dismissal in both modes, and the -24px menu anchor alignment.

Phase 3 (Landing page): the landing page's workspace rows, health / liveness / backup badges, start-stop-restart controls, remote tiles, providers panel, sync-unlock banner and bottom-left launchers are now the mithril `LandingPage` component, mounted synchronously from the boot island (`build_chrome_boot_state()` + landing extras) so the first paint arrives complete. Row-set changes rerender in place from the SSE-fed store -- the refetch-on-drift full-page path and its freshness/teardown timer plumbing are gone (`minds:refresh-local-page` itself stays for sync-unlock and the discovering state).

The chrome workspaces contract gained additive fields so rows stay complete across live pushes: a per-row friendly `provider` label, remote-row `host_id` / `state_detail`, and `destroying_status_by_agent_id` (a failed destroy now updates the row live). The backup fetch fan-out moved into a frontend service (`landing_service.ts`) with the same derivation (backing-up precedence, newest-snapshot time, 75-minute "Created N ago" window) and export flow.

Deleted: ~767 lines of Landing.jinja inline JS plus its Jinja row markup (the page is now a shell + island + mount). The `landing-stopped-mind-restart` Playwright contract spec drives the real compiled bundle instead of the inline handler; the visual-diff harness now waits for the `data-minds-mounted` marker so client-rendered pages screenshot after mounting.

Phase 4 (titlebar): the titlebar interior (breadcrumb + workspace icon-tabs, contextual back, requests badge with its 99+ cap, report-a-bug button with the agent-assist gating, window controls) is now the mithril `TitleBar` component, mounted into the server-rendered `#minds-titlebar` skeleton (which stays as the pre-mount first paint). The URL classification (`classifyContent`), accent application (document CSS variables + `titlebar-surface`, with a guard that keeps the server-seeded accent until the first real accent push), and badge state are store selectors/effects; Electron's content-url/accent/current-workspace IPC pushes subscribe in the mount, and browser mode's chrome.js pushes through `MindsUI.setContentUrl` / `setAccentScopeAgent` / `setDisplayedWorkspaceAgent`.

chrome.js shrank ~390 lines to its intended remainder -- swap engine, mode setup, switcher toggle/anchor math, recovery redirect, host glue. Deleted: `classifyContent` / `applyTitlebarContext` / `updateRequestsBadge` / the accent cache + `applyTitleAccent` / all titlebar button wiring.

A new `scripts/titlebar_first_paint_check.py` harness locks in the no-flash protocol: CPU-throttled screencast captures over real-rendered pages assert the first painted frame's titlebar state (neutral home, accent-seeded workspace pages, at-most-one transition per hub hop, titlebar element survival). Verified identical to the pre-phase baseline.

Phase 5 (inbox list): the inbox's left pane -- the pending-request card list and the auto-open footer -- is now the mithril `InboxList` component, mounted from the page's boot island and fed by the chrome `requests` payload, which gained a `cards` field (id, kind label, workspace name, display name, accent) alongside the existing count/ids. The list re-renders straight from SSE pushes, so the `/inbox/list` fragment endpoint, `render_inbox_list_fragment`, `InboxList.jinja`, and the shell's `fetchListFragment` innerHTML refetch are deleted. The right detail pane stays a server-rendered fragment by design (each request handler composes its own form); the component owns fetching/swapping it on selection and calls back into the page's inline script to re-wire the form logic (Approve gating, share-path pickers, grant/deny submission all stay inline).

Behavior preserved: most-recent-first order with the homepage-tile accent per card, auto-select of the first pending request, `?selected=` deep links, `keep_open` semantics (whole-inbox opens advance to the next request on resolve; single-request opens dismiss), the `(denying...)` faded state until the SSE push prunes the card, immediate removal of granted cards, the canonical "no longer available" fragment when the selection resolves elsewhere, empty-state collapse, and the fire-and-forget auto-open toggle. New in browser mode: the list now live-refreshes from the shared EventSource (previously Electron-only).

Electron's main process now replays the FULL cached `workspaces` / `requests` payloads when priming a (re)loaded chrome view or overlay iframe, instead of reconstructing partial events that dropped fields the store-fed components need (the old partial `requests` replay would have wiped a freshly-seeded card list).

Phase 6 (sharing editor): the sharing editor -- heading takeover, shared-URL section with the Cloudflare readiness polling, three-state access list (existing / added / removed), Update-or-Share / Disable submission with the in-place state refresh, and inline error reporting -- is now the mithril `SharingEditor` component, mounted from a `sharing` boot island by both the full `/sharing` page (browser fallback) and the Electron sharing modal. The status refresh already spoke JSON (`GET /api/v1/workspaces/<id>/sharing/<service>`), so no new endpoint was needed. The modal's `is_modal` island flag keeps the component's heading plain-text (nothing may navigate the overlay iframe) and swaps Cancel to the dismiss callback; the full page keeps its linked heading server-rendered for the pre-mount paint.

Deleted: `static/sharing.js` (397 lines) and the dual-rendered editor markup in `SharingEditor.jinja` (now a mount container + island; the Associate prompt stays server-rendered). New shared frontend modules: `ui.ts` (the Button/TextInput class recipes mirrored from templates.py's catalog globals, now also used by the landing page's ghost icon button) and `api_errors.ts` (the normalizeApiError port; the static twin remains only for the not-yet-migrated create form).

Behavior preserved: emails render as text nodes only (no HTML injection surface), owner-email pre-population when disabled, URL-proposed drafts folded in once, the provisioning spinner until the Access edge answers ready (with the "may take a moment" note after 12 probes), Copy/Copied feedback, gray-out during saves, and inline 502 error display with retry.

Phase 8 (guardrails + docs): a new minds-specific ratchet file (`test_minds_ratchets.py`) locks hand-rolled DOM building (`createElement` / `.innerHTML` / `insertAdjacentHTML`) in `static/*.js` and inline template scripts at the post-migration counts (19 static-JS chunks, 2 template chunks; the vendored sentry bundle and the compiled component bundle are excluded), so dual-rendered UI cannot silently return -- new UI belongs in the component bundle. The docs now describe the component architecture: `templates/README.md`'s JS-styling section became "Components vs remaining Jinja" (the component/Jinja split, the mount protocol pointer, the `ui.ts` / `icons.ts` recipe mirrors, the new ratchet), and `desktop-app.md`'s stale "App shell" section (which still described a `webContents.insertCSS`-injected titlebar) was rewritten around the actual multi-WebContentsView + component-bundle architecture. Stale comments referencing deleted files (`sidebar.js`, `sidebar_workspace_row.js`, `applyTitlebarContext`, chrome.js accent painting) were swept from templates, static JS, `main.js`, and the styleguide.

Phase 7 (browser-mode modal parity): browser mode no longer loses workspace context on modal opens. The chrome shell mounts a mithril `ModalHost` (a persistent `#minds-modal-host` layer) whose openModal shows each modal as a full-viewport iframe of the same server overlay pages Electron's modal view loads (`/settings/modal`, `/accounts/modal`, `/auth/signin-modal`, `/inbox`, `/help`, `/sharing/<id>/<svc>/modal`) -- mirroring Electron's overlay-iframe architecture instead of componentizing modal content, so the modal pages stay JinjaX and work in both hosts unchanged.

The hosted iframe adopts a `window.minds`-compatible bridge from the parent (installed by the bundle before any page script runs), so every modal page's Electron code path drives the in-document layer: close buttons and backdrops dismiss it, `navigateContent` (including the sign-in modal's `MINDS_AUTH_NAV` return-to flow) lands behind the modal and dismisses, modal-to-modal opens (Manage Accounts -> Add account) swap the hosted page in place, and chrome SSE events fan into the frame from the document's shared EventSource. Escape dismisses wherever focus sits -- the shell document (right after an open, before the user clicks into the modal) or each loaded frame -- standing in for Electron main's before-input-event, and the titlebar Requests button keeps its toggle semantics. Inline entry points on trusted local pages (welcome sign-in buttons, the create screen's sign-in prompt, workspace settings' Manage sharing links) gained an in-document-modal branch between the Electron bridge and the standalone-page navigation fallback.

Full-page routes (`/settings`, `/accounts`, `/inbox`, `/help`, full-page sharing) remain as deep links with their previous dismiss-by-navigation behavior in documents without a mounted ModalHost.

Review fixes: disabling sharing mid-provisioning no longer lets a late readiness probe resurface the dead share URL (the poll loop is invalidated on disable); the browser ModalHost reloads its iframe on every open -- a same-URL reopen previously left the frame in place with its chrome-event feed silently dropped -- and the help modal now toggle-closes like Electron's `toggleHelp`. Electron main lost the vestigial inbox-list refresh debounce (the SSE broadcast already delivers every requests event to an open inbox) and gained a shared helper for the cached-requests priming payload; `InboxList` releases its store subscription on page teardown; and the `ui.ts` button/input class recipes got a byte-drift guard test against their `templates.py` twins (like the icons mirror). `SharingEditor` likewise stops its readiness poll loop (and the Copy-label reset timer) when the page mount is torn down -- a pending retry now bails before probing once its loop is invalidated.

Fixed a crash on the persistent chrome route (`GET /_chrome?agent=agent-...`): the titlebar-breadcrumb name resolution still indexed the workspace rows as dicts (`ws["name"]`, `ws.get("id")`), but those rows became `ChromeWorkspaceEntry` models during the migration, so any load carrying a crumb id with at least one authenticated workspace raised `AttributeError: 'ChromeWorkspaceEntry' object has no attribute 'get'`. Resolution now reads the model attributes -- extracted into `_resolve_crumb_workspace_name` and covered by unit tests -- and the now-unused, mistyped `initial_workspaces` passthrough (the template never rendered it) was dropped from `render_chrome_page` and `Chrome.jinja`.

Added an opt-in `MINDS_REMOTE_DEBUGGING_PORT` for local inspection: when set, the Electron main process exposes the Chrome DevTools Protocol on `127.0.0.1:<port>` (with `--remote-allow-origins`, which modern Chromium requires before a CDP client may attach) so `chrome://inspect`, Puppeteer, or the chrome-devtools MCP can read each surface's console, network, and errors. Dev-only; ignored in packaged builds. It complements the existing in-app `MINDS_OPEN_DEVTOOLS` (which only opens DevTools windows on this machine), now documented alongside it.

Fixed the titlebar workspace breadcrumb (name + Workspace/Settings tabs) intermittently vanishing after navigating a workspace -> its settings -> back, and sometimes when opening a workspace from Home. The persistent chrome shell's swap engine stamped the `/_chrome` loading-wrapper path as the titlebar's content URL; `/_chrome` classifies as Home, which hides the workspace crumb. This raced main's authoritative push of the displayed workspace's URL, so the crumb showed or hid nondeterministically ("fragile"). The wrapper swap now stamps neither the content URL nor the accent -- both are owned by main's IPC pushes for the displayed workspace -- mirroring the `/_chrome` accent guard that was already in place. Verified live over CDP: the crumb now stays put across workspace/settings/Home flows, and injecting the old `setContentUrl('/_chrome')` reproduces the drop. A `TitleBar` regression test locks in the contract the fix relies on: `/_chrome` classifies as home, and stamping that wrapper URL as the content URL hides the workspace crumb.

Removed the recovery page's in-place ("surgical") system-services restart tier. The `INTERFACE_UNRESPONSIVE` classification and its auto-dispatched services-scope restart never fixed real user issues, and auto-restarting off a heuristic verdict violates the recovery principle that auto-action needs unambiguous evidence. A container that is exec-reachable but whose interface is not answering `GET /` with 200 now renders the consent-gated "Workspace unresponsive" page -- unless supervisord reports the interface `STARTING`/`BACKOFF` (self-heal in progress), in which case the page keeps checking. The v1 restart API accepts only `scope: "host"`; `"services"` returns 400.

The recovery classifier now branches on the raw host state rather than a collapsed "is it running?" answer: `STOPPED`/`CRASHED` keep today's unattended host restart (the path that revives workspaces after a laptop reboot); `STOPPING` is transitional (keep checking); `FAILED` is consent-gated (an unattended start on a failed-to-create host mostly re-fails); the new `UNREACHABLE` provider state (host rejected this machine's access) renders the terminal "Can't connect" page with retry/report only and a canned access-rejected reason, since a restart routes through the same rejected credential.

Stopping a workspace through the agent-facing `/api/v1/workspaces/<id>/stop` route now closes any Electron window open to it (skipped mid-restart), the same way the landing-page Stop already did -- previously the open window would observe the dead interface, redirect to recovery, and auto-restart the host, silently undoing an agent-requested stop. Both stop paths share one mechanism: a one-shot `workspace_stopped` chrome SSE event; browser mode navigates the content frame home.

Every restart-failure branch (stop step, start step, readiness timeout, services-agent-not-found, worker spawn failure) now logs at error level so it reaches error reporting; the readiness-timeout and services-agent-not-found branches were previously entirely unlogged.

The recovery page no longer dead-ends on "Reconnecting" when the discovery stream itself has stalled (e.g. the `mngr latchkey forward` supervisor died, so no snapshot taken after the outage onset can ever land). An in-container exec probe that *completes* without reaching the container (dead inner sshd, container not actually running) is now direct evidence and resolves to the consent-gated "Workspace unresponsive" verdict without waiting on the snapshot-freshness gate; a probe *timeout* still keeps checking (it observed nothing -- the macOS-sleep protection). The exec probe is also attempted when the workspace's provider has produced no discovery snapshot for over 90 seconds, regardless of the last-known host state, so a stopped container under a dead discovery producer reaches a restart affordance instead of spinning forever. With discovery flowing normally, a trusted `STOPPED`/`CRASHED` observation still auto-restarts unattended and skips the exec round-trip.

The recovery diagnostics list gains an "Is the system-services agent running?" row: the in-container probe scans `/proc/*/environ` for the agent's `MNGR_AGENT_ID` marker (the same marker mngr's own stop path uses to find agent processes), so a workspace whose system-services agent was stopped (`mngr stop system-services`) names that cause directly instead of showing only downstream supervisorctl connection errors.

The recovery page now names what it is doing while a restart is in flight instead of reusing the generic "Loading workspace" spinner. The auto-dispatched `host_offline` cold-boot renders "Bringing your workspace back online" with a message that the workspace was offline and is being started back up; a manual "Restart workspace" click (and a page load that lands mid-restart) renders a generic "Restarting your workspace" state. Both still return the user to the workspace automatically the moment it comes back.

The sidebar right-click menu no longer offers "Restart system interface": the services-scope restart tier it dispatched was removed in this PR (the API now rejects `scope: "services"` with 400), so the item had become a silent no-op. "Restart workspace" remains.

The health-probe log line now includes the `unreachable_reason` when the verdict is `backend_unreachable`, so a provider-error verdict (e.g. a transient docker discovery failure at app startup) is diagnosable from the log instead of appearing only as an unexplained tier.

The "Can't connect to ..." (backend-unreachable) recovery page no longer dead-ends on a transient provider error: it keeps re-probing on the same slow cadence as the "Reconnecting" state, so when the provider's next clean discovery snapshot clears the error, the page re-classifies to the real verdict and the recovery flow continues (including the unattended offline restart when the workspace's container is in fact stopped). The re-probe timer is also guarded so the Retry button cannot stack parallel probe chains.

The consent-gated "Workspace unresponsive" verdict also stays live now: a workspace whose container is stopped can land there first (a completed exec failure under absent discovery cannot distinguish "stopped" from "broken ssh"), and the page previously never re-probed, stranding the user on a consent page whose restart should have been unattended. Both unresponsive renders schedule the slow re-probe, so the first trusted STOPPED snapshot re-classifies to the offline tier and dispatches the automatic cold boot. The verdict renders also reset each other's elements -- previously moving from the backend-unreachable state to the unresponsive one left the Retry button and the provider-error paragraph visible alongside the Restart button.

Discovery snapshots replayed from before the minds backend started no longer surface their provider errors as current state. The detached discovery producer keeps polling while the app is closed, and the quit flow deliberately stops the docker state container, so the pre-start backlog's last docker snapshot reliably carries a manufactured "Docker state container is stopped" error -- which the app then displayed at every startup even though it restarts that container before discovery runs. The replayed snapshot's topology still merges (last-good retention); a genuinely-broken provider re-asserts its error on the first fresh discovery cycle.

A failed top-level load of a workspace URL in the desktop app (e.g. `ERR_CONNECTION_REFUSED` against a stale forwarded port restored from a previous session, or `ERR_NETWORK_CHANGED` across a network flap) no longer leaves the window permanently blank: the content view now routes to that workspace's recovery page, whose liveness poll returns the user to the workspace -- via a freshly-built URL on the current session's forward port -- the moment it answers, and whose restart tiers handle a genuinely-down workspace.

A window sitting on a workspace's recovery page now persists (and restores) as the workspace itself in the saved window state. The recovery URL's `return_to` query embeds an absolute URL on that session's `mngr_forward` origin, whose port changes every run -- previously a window quit on the recovery page would, on the next launch, restore the recovery page, go healthy, and 302 to the dead old port (the source of the blank-white-window-on-startup reports). Restoring into the workspace re-enters recovery through the normal unhealthy-redirect with fresh parameters if it is still down.

Removed the recovery page's in-place ("surgical") system-services restart tier. The `INTERFACE_UNRESPONSIVE` classification and its auto-dispatched services-scope restart never fixed real user issues, and auto-restarting off a heuristic verdict violates the recovery principle that auto-action needs unambiguous evidence. A container that is exec-reachable but whose interface is not answering `GET /` with 200 now renders the consent-gated "Workspace unresponsive" page -- unless supervisord reports the interface `STARTING`/`BACKOFF` (self-heal in progress), in which case the page keeps checking. The v1 restart API accepts only `scope: "host"`; `"services"` returns 400.

The recovery classifier now branches on the raw host state rather than a collapsed "is it running?" answer: `STOPPED`/`CRASHED` render as offline; `STOPPING` is transitional (keep checking); `FAILED` renders the consent page (a plain start on a failed-to-create host mostly re-fails); the `UNAUTHENTICATED` provider state (host rejected this machine's access at its access boundary) renders the terminal "Can't connect" page with retry/report only and a canned access-rejected reason, since a restart routes through the same rejected credential. The tiers are display-only: the recovery restart itself is dispatched unconditionally on page entry (see below), never off a classified verdict.

Stopping a workspace through the agent-facing `/api/v1/workspaces/<id>/stop` route now closes any Electron window open to it (skipped mid-restart), the same way the landing-page Stop already did -- previously the open window would observe the dead interface, redirect to recovery, and auto-restart the host, silently undoing an agent-requested stop. Both stop paths share one mechanism: a one-shot `workspace_stopped` chrome SSE event; browser mode navigates the content frame home. This one-shot chrome-events broadcaster now also carries the agent bug-report `open_help` event, which previously had its own near-identical broker -- both edge-triggered events fan out through a single per-connection queue on the SSE stream.

Every restart-failure branch (stop step, start step, readiness timeout, services-agent-not-found, worker spawn failure) now logs at error level so it reaches error reporting; the readiness-timeout and services-agent-not-found branches were previously entirely unlogged.

The recovery page no longer dead-ends on "Reconnecting" when the discovery stream itself has stalled (e.g. the `mngr latchkey forward` supervisor died, so no snapshot taken after the outage onset can ever land). An in-container exec probe that *completes* without reaching the container (dead inner sshd, container not actually running) is now direct evidence and resolves to the consent-gated "Workspace unresponsive" verdict without waiting on the snapshot-freshness gate; a probe *timeout* still keeps checking (it observed nothing -- the macOS-sleep protection). The exec probe is also attempted whenever the resolver's host state is not yet trustworthy -- a stale pre-onset snapshot, or a dead discovery producer whose freshness gate can never open -- regardless of the last-known host state. So a workspace under a dead discovery producer gathers direct evidence and reaches a resolved page immediately, with no fixed staleness threshold to wait out, instead of spinning forever. With discovery flowing normally, a fresh trusted state is left to the classifier and skips the exec round-trip (a transitional `STOPPING` keeps checking rather than being prematurely consent-gated).

The recovery diagnostics list gains an "Is the system-services agent running?" row: the in-container probe scans `/proc/*/environ` for the agent's `MNGR_AGENT_ID` marker (the same marker mngr's own stop path uses to find agent processes), so a workspace whose system-services agent was stopped (`mngr stop system-services`) names that cause directly instead of showing only downstream supervisorctl connection errors.

The recovery page's in-flight copy claims only what it knows. A manual "Restart workspace" click renders "Restarting your workspace" (the restart is known -- the user asked for it). The unconditional entry dispatch cannot know whether its start-only restart will do anything, so it stays on the neutral "Loading workspace" spinner -- except when the workspace's host currently reads offline (STOPPED/CRASHED), where a real cold boot is coming and the page renders "Bringing your workspace back online". The offline reading is a display-only hint from the resolver (`data-host-offline` at render time plus an `X-Workspace-Offline` header on every convergence-poll tick): a stale reading at a cold launch self-corrects one-way to the offline copy once discovery lands the STOPPED observation, and only for the entry dispatch -- a manual bounce's transient mid-restart STOPPED never relabels the page as an offline revival. Nothing dispatches or branches off the hint.

The sidebar right-click menu no longer offers "Restart system interface": the services-scope restart tier it dispatched was removed in this PR (the API now rejects `scope: "services"` with 400), so the item had become a silent no-op. "Restart workspace" remains.

The health-probe log line now includes the `unreachable_reason` when the verdict is `backend_unreachable`, so a provider-error verdict (e.g. a transient docker discovery failure at app startup) is diagnosable from the log instead of appearing only as an unexplained tier.

The "Can't connect to ..." (backend-unreachable) recovery page no longer dead-ends on a transient provider error: it keeps re-probing on the same slow cadence as the "Reconnecting" state, so when the provider's next clean discovery snapshot clears the error, the page re-classifies to the real verdict and the recovery flow continues (including the unattended offline restart when the workspace's container is in fact stopped). The re-probe timer is also guarded so the Retry button cannot stack parallel probe chains.

The consent-gated "Workspace unresponsive" verdict also stays live now: the page previously never re-probed, so its verdict and diagnostics went stale the moment evidence changed. Both unresponsive renders schedule the slow re-probe, keeping the failure page's verdict current. The verdict renders also reset each other's elements -- previously moving from the backend-unreachable state to the unresponsive one left the Retry button and the provider-error paragraph visible alongside the Restart button. Dispatching a restart silences the re-probe chain, so a stale probe result can no longer overwrite the "Restarting your workspace" state after the dispatch.

The recovery page's restart is now dispatched unconditionally on page entry, as a start-only restart (`start_only` in the v1 restart API, renamed from `host_already_stopped`) -- with no knowledge of the host's state, and no probe or classified verdict gating it. The dispatch is safe regardless of that state: `mngr start` checks ground truth at commit time, no-ops on a live host, targets only STOPPED agents, and cold-boots a stopped one; and in-app stops close their windows first, so a window that reaches recovery over a stopped host implies an out-of-app stop whose revival is intended. A stopped workspace therefore revives on first entry -- at app startup, after a laptop reboot, after an out-of-app stop -- with no discovery-freshness crawl and no dependence on the resolver's (possibly stale, replayed, or flickering) lifecycle mirror. The manual "Restart workspace" button keeps the full stop+start bounce, which is never dispatched without a click. The probe log records the classifier's inputs (host state, trust, exec outcome) alongside each verdict.

Discovery snapshots replayed from before the minds backend started no longer surface their provider errors as current state. The detached discovery producer keeps polling while the app is closed, and the quit flow deliberately stops the docker state container, so the pre-start backlog's last docker snapshot reliably carries a manufactured "Docker state container is stopped" error -- which the app then displayed at every startup even though it restarts that container before discovery runs. The replayed snapshot's topology still merges (last-good retention); a genuinely-broken provider re-asserts its error on the first fresh discovery cycle.

minds-side `mngr exec` calls that exist for diagnostics or cleanup no longer auto-start a stopped host (`mngr exec` boots the host by default): the backup-stack execs (the periodic verification check -- observed live cold-booting a stopped container it believed online off a stale RUNNING snapshot -- plus env injection and update scripts), the workspace version read, the assist support probe and chat create, and the tunnel-token cleanup all pass `--no-start` now. Two execs deliberately keep the auto-start behavior pending a design call: the tunnel-token inject behind share-enable and the agent-facing SSH-grant read/write. Every diagnostic and cleanup exec now fast-fails against a stopped host instead of booting it, so the recovery page's verdicts can no longer be invalidated mid-probe by a background exec silently booting the host.

The workspace version read's exec was also repaired to actually reach the workspace: it passed the git command as multiple tokens after a `--` separator, which `mngr exec` (whose CLI takes a single trailing COMMAND argument) parsed as extra agent names -- the call always errored out and the read silently fell back to the creation-time version label. It now sends the git command as one shell string and extracts its output through the `--format json` envelope (raw human-format stdout carries mngr's trailing "Command succeeded" status line).

A failed top-level load of a workspace URL in the desktop app (e.g. `ERR_CONNECTION_REFUSED` against a stale forwarded port restored from a previous session, or `ERR_NETWORK_CHANGED` across a network flap) no longer leaves the window permanently blank: the content view now routes to that workspace's recovery page, whose liveness poll returns the user to the workspace -- via a freshly-built URL on the current session's forward port -- the moment it answers, and whose restart tiers handle a genuinely-down workspace.

A window sitting on a workspace's recovery page now persists (and restores) as the workspace itself in the saved window state. The recovery URL's `return_to` query embeds an absolute URL on that session's `mngr_forward` origin, whose port changes every run -- previously a window quit on the recovery page would, on the next launch, restore the recovery page, go healthy, and 302 to the dead old port (the source of the blank-white-window-on-startup reports). Restoring into the workspace re-enters recovery through the normal unhealthy-redirect with fresh parameters if it is still down.

Host lifecycle state is deliberately not persisted across a quit. An earlier revision of this branch persisted the offline (`STOPPED`/`CRASHED`) host states in the last-good topology and seeded them as overrides on launch, so the recovery page could *know* a host was stopped early enough to auto-dispatch its cold boot -- but live quit/relaunch traces showed that mirror being erased by stale discovery observations the pipeline structurally produces (in-flight docker polls that started before the stop and landed after it, and the replayed pre-start backlog's pre-stop `RUNNING` snapshots), stranding the launch on a multi-minute "unresponsive" crawl anyway. With the restart now dispatched unconditionally on recovery entry, that knowledge is unnecessary and the whole subsystem (persisted offline map, launch-time seeding, discovery reconcile, the classifier's offline freshness-gate exemption, and the probe's offline exec-skip) is removed. Host-state overrides remain as in-memory-only UI optimism for lifecycle clicks, and the quit-time and per-workspace stop paths now log their targets and outcomes for diagnosability. The persisted topology reader also tolerates unknown fields now, so a topology file written by a different app version (e.g. one carrying the removed offline-state field) no longer voids the whole topology -- and with it the system-services fallback -- on the first launch after an upgrade.

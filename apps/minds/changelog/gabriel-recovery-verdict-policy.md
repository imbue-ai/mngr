Removed the recovery page's in-place ("surgical") system-services restart tier. The `INTERFACE_UNRESPONSIVE` classification and its auto-dispatched services-scope restart never fixed real user issues, and auto-restarting off a heuristic verdict violates the recovery principle that auto-action needs unambiguous evidence. A container that is exec-reachable but whose interface is not answering `GET /` with 200 now renders the consent-gated "Workspace unresponsive" page -- unless supervisord reports the interface `STARTING`/`BACKOFF` (self-heal in progress), in which case the page keeps checking. The v1 restart API accepts only `scope: "host"`; `"services"` returns 400.

The recovery classifier now branches on the raw host state rather than a collapsed "is it running?" answer: `STOPPED`/`CRASHED` keep today's unattended host restart (the path that revives workspaces after a laptop reboot); `STOPPING` is transitional (keep checking); `FAILED` is consent-gated (an unattended start on a failed-to-create host mostly re-fails); the new `UNREACHABLE` provider state (host rejected this machine's access) renders the terminal "Can't connect" page with retry/report only and a canned access-rejected reason, since a restart routes through the same rejected credential.

Stopping a workspace through the agent-facing `/api/v1/workspaces/<id>/stop` route now closes any Electron window open to it (skipped mid-restart), the same way the landing-page Stop already did -- previously the open window would observe the dead interface, redirect to recovery, and auto-restart the host, silently undoing an agent-requested stop. Both stop paths share one mechanism: a one-shot `workspace_stopped` chrome SSE event; browser mode navigates the content frame home.

Every restart-failure branch (stop step, start step, readiness timeout, services-agent-not-found, worker spawn failure) now logs at error level so it reaches error reporting; the readiness-timeout and services-agent-not-found branches were previously entirely unlogged.

The recovery page no longer dead-ends on "Reconnecting" when the discovery stream itself has stalled (e.g. the `mngr latchkey forward` supervisor died, so no snapshot taken after the outage onset can ever land). An in-container exec probe that *completes* without reaching the container (dead inner sshd, container not actually running) is now direct evidence and resolves to the consent-gated "Workspace unresponsive" verdict without waiting on the snapshot-freshness gate; a probe *timeout* still keeps checking (it observed nothing -- the macOS-sleep protection). The exec probe is also attempted when the workspace's provider has produced no discovery snapshot for over 90 seconds, regardless of the last-known host state, so a stopped container under a dead discovery producer reaches a restart affordance instead of spinning forever. With discovery flowing normally, a trusted `STOPPED`/`CRASHED` observation still auto-restarts unattended and skips the exec round-trip.

The recovery diagnostics list gains an "Is the system-services agent running?" row: the in-container probe scans `/proc/*/environ` for the agent's `MNGR_AGENT_ID` marker (the same marker mngr's own stop path uses to find agent processes), so a workspace whose system-services agent was stopped (`mngr stop system-services`) names that cause directly instead of showing only downstream supervisorctl connection errors.

- Removed the sidebar surface. The home page (`/`) was already showing the same workspace list and is now the single entry point.
- Each home-page workspace row gains an always-visible "open in new window" icon button (next to the existing restart and settings icons), matching the Electron-only `openWorkspaceInNewWindow` IPC the sidebar previously called from its hover icon and right-click context menu. The row-level right-click context menu is preserved.
- The right-side requests panel becomes an inbox **modal** instead. Triggering it (chrome titlebar bell, auto-open on new request, or a workspace-content `is_user_requested` event) opens a transparent overlay over the whole window with a two-pane layout (list of pending requests on the left, selected request's permission UI on the right). Click-out / Escape closes; the workspace behind stays exactly where it was.
- The permission-request dialog and the requests inbox are merged: granting / denying a request from the inbox does not close the modal -- the detail pane auto-advances to the next pending request, and the modal only closes when the inbox empties.
- `is_user_requested` events no longer navigate the workspace content view; they auto-open the inbox modal with that request selected via a new `force_open_event_id` field on the chrome SSE.
- New backend endpoints: `GET /_chrome/requests-inbox` (full inbox page), `GET /_chrome/requests-inbox/list` (list-pane fragment), `GET /_chrome/requests-inbox/detail/{event_id}` (detail-pane fragment, body-only). The old `/_chrome/requests-panel` and `/_chrome/sidebar` routes are gone.
- Browser-mode parity: the chrome page hosts the inbox in a hidden full-window iframe shown by the requests-toggle, so the same UX works without Electron.

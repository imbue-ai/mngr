The desktop app now bundles `qemu-img`, so the pre-baked Lima image fast-create path (issue #2306) works out of the box on a clean machine. `qemu-img` converts the downloaded image `raw`->`qcow2` on every platform (including Apple Silicon, where Lima runs the VM via `vz`); previously it was resolved only from `PATH`, so the fast path silently required a Homebrew qemu install and fell back to the slow in-VM build without one.

There is no upstream `qemu-img` release for macOS, so the bundled binary is built from SHA256-pinned source tarballs by `scripts/build-qemu-payload.sh` (modeled on `containers/podman-machine-qemu`): a tools-only QEMU with every optional feature disabled and glib plus its deps (libffi, libintl, pcre2) linked statically, producing a single binary that links only always-present system libraries. The payload is fetched at build time from a GitHub release, SHA256-pinned like the other bundled binaries (`uv`, `git`, `restic`, `desync`), and signed for notarization under the hardened runtime.

Both pre-baked-image tools now resolve their binary from an explicit env var set by the Electron shell in dev and packaged mode -- `MINDS_QEMU_IMG_BINARY` for the `raw`->`qcow2` conversion (`QemuImageFormatConverter`) and `MINDS_DESYNC_BINARY` for the image download (`DesyncImageChunkStore`) -- falling back to a `PATH` lookup when the var is unset. Dev mode inherits the developer's `PATH` untouched, so without this the bundled binaries were unreachable there and the fast-create path silently needed a system-wide `desync` and a Homebrew `qemu`.

Linux payloads are produced by `scripts/build-qemu-payload-linux.sh` (Docker + Alpine): Linux has no static-libc limitation, so QEMU's `--static` against musl yields a fully static ELF with zero runtime dependencies. Both linux arches are published on the same GitHub release; ToDesktop's Linux builder runs the same `beforeInstall` hook, so these assets are required for `pnpm dist` even though no Linux app ships yet.

Also fixed: `desync` was never staged by `scripts/build.js`, only by the ToDesktop `beforeInstall` hook. That hook's output is written where the packager folds it into `app.asar`, which the app never reads -- it resolves bundled binaries under `process.resourcesPath`. Every packaged app since desync was introduced has therefore carried an unreachable copy of it (an x86_64 one, since ToDesktop's mac agent is Intel) and no usable one, so the pre-baked image download could not work in any packaged build. `build.js` now stages desync alongside the other binaries, `ensure-binaries.js` requires it in dev, and -- since this is the first build in which desync actually ships inside the `.app` -- it is signed for notarization under the hardened runtime like `limactl`, `restic`, and `qemu-img`.

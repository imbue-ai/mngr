The desktop app now bundles `qemu-img`, so the pre-baked Lima image fast-create path (issue #2306) works out of the box on a clean machine. `qemu-img` converts the downloaded image `raw`->`qcow2` on every platform (including Apple Silicon, where Lima runs the VM via `vz`); previously it was resolved only from `PATH`, so the fast path silently required a Homebrew qemu install and fell back to the slow in-VM build without one.

Because there is no upstream static `qemu-img` for macOS (it links glib, and macOS has no static libc), the bundled binary is a relocated, self-contained payload -- `bin/qemu-img` plus its dylib closure under `lib/`, with load paths rewritten to `@executable_path/../lib` -- produced by `scripts/build-qemu-payload.sh` and fetched at build time from a GitHub release, SHA256-pinned like the other bundled binaries (`uv`, `git`, `restic`, `desync`). The packaged app resolves it via an explicit `MINDS_QEMU_IMG_BINARY` env var (so resolution never depends on `PATH` ordering), and every shipped Mach-O in the payload is code-signed for notarization under the hardened runtime.

The `raw`->`qcow2` conversion in the pre-baked-image prefetch (`QemuImageFormatConverter`) now sources its binary from `MINDS_QEMU_IMG_BINARY`, falling back to a `PATH` `qemu-img` when the var is unset (dev machines with Homebrew qemu).

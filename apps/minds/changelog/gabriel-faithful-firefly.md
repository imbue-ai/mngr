Fixed the app getting permanently stuck on "Discovering agents..." (never falling through to the create form) after a workspace vanished without minds ever observing its host destroyed -- e.g. an imbue_cloud pool host returned to the pool, which disappears from discovery without a DESTROYED transition.

- The last-good agent topology (the fallback that keeps a workspace's window restorable across a transient discovery loss) previously only dropped a remembered host on a positive DESTROYED observation, so a workspace that simply stopped appearing lingered forever and kept the landing page on the discovering spinner. A remembered host absent from discovery is now also dropped once its owning provider produces a clean, non-errored snapshot that does not list it -- proof the workspace is genuinely gone rather than the provider being slow. Absence with no authoritative provider signal (the provider errored, or has not reported yet) is still retained, preserving the slow-cold-start protection.

- Last-good records now carry their owning provider so the prune can be scoped per host: an unrelated provider's outage (common when several cloud providers are configured but unused) no longer blocks clearing a genuinely-gone workspace. Records persisted before this change carry no provider and instead clear once discovery is demonstrably functioning and the host is absent; a still-live workspace is re-recorded (now with its provider) the moment discovery re-enumerates it.

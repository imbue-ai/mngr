Fixed a startup race that could strand a restored workspace window on the `mngr forward` "Sign in" page ("Use the login URL printed in the terminal where `mngr forward` is running"). On launch the app pre-sets the proxy's session cookie asynchronously, but restored workspace windows navigated straight to the proxy without waiting for that write to land; if a window reached the proxy first (most likely on a heavily loaded machine, or when a second minds instance had already taken the default forward port), it was redirected to the plugin's terminal-oriented login page -- a dead end for the app, which had no way to know the cookie was moments away. Startup restore now waits for the preauth cookie to be written before navigating any restored workspace view (bounded so a missing signal can never hang startup), so the app no longer lands on that page. The plugin login page is unchanged and still correct for real command-line `mngr forward` users.

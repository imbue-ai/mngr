Workspace Claude auth moved out of the create flow: the AI-provider choice (and API-key field) is gone from the create form and `POST /api/v1/workspaces` (a stale `ai_provider` field from an old client is silently ignored), and the desktop client no longer mints LiteLLM keys or injects `ANTHROPIC_API_KEY`/`ANTHROPIC_BASE_URL` at create time. Workspaces boot unauthenticated and sign in through their own in-workspace Claude sign-in modal, which writes credentials into the env block of the shared `CLAUDE_CONFIG_DIR/settings.json`.

- New mint-only AI-key page at `/settings/ai-keys?workspace=<host_id>` (linked from the workspace modal's "Sign in with Imbue" option): resolves the workspace's owning account from the workspace-record store (errors with guidance when no account is associated), mints a LiteLLM key with the workspace host id fixed into the alias/metadata (100 USD / 1d rolling budget), and copies an env-var-style credential blob to the clipboard for pasting back into the modal.

- The Electron e2e chat test now exercises the new first-boot flow: create an unauthenticated workspace, sign in through the real modal UI (API-key path), then chat. The `test_litellm_via_workspace` deployment test is implemented for real: connector key mint, local Docker workspace creation, credential submission through the workspace's modal endpoint, a live chat through the litellm proxy, and a spend assertion against the env's litellm database.

Creating a Docker mind no longer intermittently fails with `rsync: [Receiver] getcwd(): No such file or directory` when two minds are created for the same repo at roughly the same time. Each creation clones the template repo into a stable, repo-name-keyed temp dir (`minds-clone-<repo>`, kept stable so Docker layer caching hits), which means two concurrent same-repo creations target the *same* directory: the second creation's clean-and-re-clone could delete that directory out from under the first creation's still-running `mngr create` (whose working directory and git-mirror source it is), and Homebrew rsync 3.4.2 turns a deleted working directory into a hard `getcwd()` abort. Minds now serializes on the clone path, so a second creation of the same repo waits until the first has finished its clone-and-create step before it touches the shared directory. Creations of different repos still run concurrently, and a plain local-directory source (used in place, never cloned) is unaffected. The trade-off is intentional: two concurrent same-repo creations now run one after the other rather than racing.

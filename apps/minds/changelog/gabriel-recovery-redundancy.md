Workspace recovery now reads the host's lifecycle state and its provider's reachability from the same passive discovery data the rest of minds already uses (the workspace list, the providers panel), instead of firing its own separate `mngr list` each time the recovery page loads. Having two independent samplers of the same state could let the recovery page disagree with the sidebar; collapsing to one source removes that inconsistency. As a side benefit the recovery page now responds effectively instantly (it reads in-memory state rather than waiting on a network round-trip), and a dropped connection can no longer strand the diagnostic on a slow probe.

Minds now waits until it has up-to-date information about a workspace before taking you to the recovery page. In the brief window right after a connection drops -- before discovery has caught up -- it keeps you on the "Loading workspace" screen rather than bouncing you to a recovery page that might offer a destructive "Restart workspace" button based on stale information. Once discovery refreshes (typically within a few seconds) it takes you straight to the right place: the recovery options if the workspace genuinely needs one, or the "Can't connect to ..." page if the provider is down (which still reconnects you automatically once it recovers).

The "Can't connect to ..." page no longer tells you to "check your internet connection" -- that was misleading for local providers like Docker, which run independently of the network. Instead it shows a provider-agnostic message and surfaces the actual error reported by the provider (e.g. Docker's "Docker Desktop is manually paused. Unpause it through the Whale menu or Dashboard.") verbatim, so the real cause is visible without minds having to hand-author a message for each provider.

The recovery page previously had two near-identical "backend is unreachable" screens that differed only by the kind of error: a transient outage (e.g. a stopped Docker daemon) got a "Can't connect to ..." page that auto-reconnects you, while anything else (e.g. a paused Docker daemon, which Docker reports differently, or an expired login) got a separate "Can't reach your workspace" page that did not. The split depended on the exact exception the provider raised, so the same situation -- Docker simply isn't available right now -- could land on either page, and the paused-Docker case lost the auto-reconnect. These are now a single page: it always shows the provider's own error, offers a Retry, and always reconnects you automatically once the backend recovers. Diagnostics are hidden on this page, since the cause is the external backend (shown verbatim), not anything inside your workspace.

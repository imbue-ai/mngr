The Workspace Settings "Backups" section was reorganized for clarity. A new "Recent backups" table shows the newest restic snapshots as rows -- each with its time (relative, with the exact local time on hover) and a per-snapshot Download action that saves that point-in-time snapshot as a zip. The newest row is tinted and marked with a green "Latest" badge (the same green pill as the landing page's "Backed up" badge). Rows are ordered newest to oldest. A "View all N backups" footer appears only when there are more snapshots than the table shows and opens a new full-history page. Not-configured, no-backups-yet, load-error, and backing-up states each get their own clear message. The landing page's per-workspace "download" link was removed for local workspaces in favor of the settings-page table; remote-device rows keep it, since they have no settings page and it is the only way to retrieve their backups.

The backup controls below the table were rewritten in plain language, grouped under two subheadings with one-line descriptions: "Where your backups are stored" (formerly "Configure backups...", now "Change storage location", with friendlier provider options and a "Save" button) and "Fix backup problems" (formerly "Update backup service", now "Update backup software", explaining that it reinstalls/restarts the backup software and never touches files or existing backups). Health-check problem messages now say what is wrong and which button fixes it, and the technical installed/minimum/target version triple was replaced by a single sentence that mentions an available update only when there is one. "Backup service verification" is now a third subheading in the same style, with a one-line description of what the checks do, placed between the storage setting and "Fix backup problems" (check first, then fix).

The backup tables do not show a snapshot-size column. Snapshot sizes are only recorded by recent restic versions, so a size column would show "--" for most existing snapshots; it was left out rather than displaying a mostly-empty column.

Each row in the settings page's "Recent backups" table now has a working "Restore" action: after a confirmation dialog (which explains that everything changed since that backup will be replaced and that a backup of the current state is taken first), the whole workspace is rewound to that snapshot, in place. The restore runs as a tracked operation with the same progress line, Cancel-while-waiting, and "Stop chats and try again" flow as "Update backup software": running chats block it, an in-flight hourly backup is waited out, and only one operation can run per workspace at a time (a double-pressed Restore is rejected rather than stacked). Inside the workspace, a deterministic script pauses the backup service, takes a tagged `pre-restore` safety snapshot (so any restore is itself undoable), downloads the snapshot into a staging directory before deleting anything, swaps it into place while preserving the current backup credentials, reinstalls dependencies, and restarts all workspace services. Restore is disabled (with an explanatory tooltip) while the workspace is offline, since it must run inside the workspace. It is backed by a new `POST /api/v1/workspaces/<id>/backups/<snapshot_id>/restore` route (202 + the shared backup-operation polling/logs routes, which now also report and cancel restore operations). The restore script understands both hourly-snapshot layouts: on btrfs-backed hosts the hourly backup captures the whole unified host volume (the workspace lives in a nested `host_dir/` inside the snapshot), while on plain-docker hosts (and for the `pre-restore` safety snapshots) the snapshot root is the workspace itself; the script identifies the workspace level by its `code/` checkout and refuses to swap in a snapshot that has none.

A new backup-history page at `/workspace/<id>/backups` shows every snapshot of a workspace, newest first, 20 per page with Newer/Older pagination and a "Showing X-Y of N backups" range indicator, so a long history is never loaded all at once. Rows look exactly like the settings page's "Recent backups" table -- relative time (exact local time on hover) and a per-snapshot Download action; the repository's newest snapshot carries the green "Latest" badge. (The history page's Restore buttons are disabled and point at the settings page, which hosts the restore flow.) Both tables render their rows through one shared script (`backup_table.js`, exposed as `window.mindsBackupTable`), so they cannot drift apart visually. A "Back to settings" link sits below the table, styled like the standard "Back to workspaces" footer link. It is backed by a new paginated API route, `GET /api/v1/workspaces/<id>/backups/snapshots?offset=&limit=` (limit defaults to 20, capped at 100), which returns one page of snapshots plus the total count and -- unlike the full `/backups` route -- never execs into the workspace, so paging is a single restic listing per request and works for offline workspaces.

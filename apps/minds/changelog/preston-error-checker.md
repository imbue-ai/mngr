Made the permission-request handling robust to a malformed `agent_id`. Previously a permission request whose `agent_id` was not a valid `agent-...` id (e.g. an agent that hand-crafted the request body and supplied a placeholder like `ENV_AGENT`) raised `InvalidRandomIdError`, which killed the whole permission-requests consumer thread in `minds run` -- silently stopping every subsequent request from ever reaching the UI -- and could take down the desktop request panel. Such requests are now logged-and-skipped for host-permission recovery (`minds run`) and filtered out of the displayable set (desktop client) instead of raising, so one bad request no longer breaks permission handling for everyone else.

Hardened two flaky creation unit tests (`test_start_creation_api_key_ai_does_not_mint_litellm_key` and `test_start_creation_subscription_ai_does_not_mint_litellm_key`) against heavy offload contention. Their real failure point was the internal `_wait_until_finished` completion assertion (a 10s deadline), not the pytest timeout, so they now use the same 20s internal budget as the local-compute sibling test (under the existing 30s pytest timeout) and carry `@pytest.mark.flaky` as a retry backstop. No product behavior change.

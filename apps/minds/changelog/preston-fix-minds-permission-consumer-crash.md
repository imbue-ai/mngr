Fixed: a permission request whose `agent_id` is not a valid `agent-…` id (e.g. an agent that hand-crafts the `POST /permission-requests` body and supplies a placeholder like `ENV_AGENT`) no longer crashes the `latchkey-permission-requests-consumer` thread. Previously that uncaught `InvalidRandomIdError` killed the consumer, after which no permission request — for any agent or service — was surfaced to the UI until restart (and it re-crashed on the same record each restart). Malformed requests are now skipped with a warning in both the consumer's host-permission recovery and the displayable-requests filter.

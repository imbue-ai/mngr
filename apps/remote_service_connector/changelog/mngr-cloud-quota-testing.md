Merged the speed-up-sharing work into the cloud-quota branch stack: the new `GET /tunnels/by-agent/{agent_id}` endpoint resolves the caller's tunnel for one agent in 2 Cloudflare calls (name filter + one config fetch) instead of enumerating every tunnel. The merged endpoint test was updated to set a tunnel auth policy first, as required by the cloud-quotas pass's always-on Access-Application hardening.

New `POST /sharing/enable` endpoint enables (or updates) sharing for one service in a single call: it ensures the tunnel (idempotent), adds the service, and applies the caller's Access policy directly to the service's Access Application -- replacing a pre-existing app's policies on re-enable -- and returns the tunnel (with token) plus service info. This collapses the client's previous create-tunnel + add-service + set-service-auth sequence (three round trips) into one, and closes the partial-state window where a service could briefly run under the fallback policy between add and auth-set. `ForwardingCtx.add_service` gained a `service_policy` parameter backing this.

Per-bucket live-usage reads are now made concurrently (bounded at 8 simultaneous Cloudflare REST calls) instead of sequentially, and `GET /account` additionally runs its three network-backed usage sources (Cloudflare tunnel count, bucket usage, LiteLLM spend) concurrently with each other. This applies to every live storage measurement -- the `GET /account` usage display, the bucket-creation storage gate, the sweep's downgrade confirmation, and the cleanup-grant baseline/recheck -- and takes a 42-bucket account's `GET /account` from roughly 19 seconds to a couple of seconds. Error semantics are unchanged: the display path still warns and counts a failed bucket read as zero, and enforcement paths still raise.
